# 任务系统模板

## 问题场景

你的智能体系统需要管理后台工作——子智能体在后台运行、Shell 命令在执行、API 请求在等待响应。你面临以下问题：

- 后台任务完成了，但主对话不知道——结果丢了
- 子智能体崩溃了，它领取的任务卡在"进行中"状态，没有人处理——**僵尸任务**（Zombie Task）
- 5 个后台任务并发运行，每个需要独立的状态（当前目录、权限上下文、文件缓存），但它们在同一个 Node.js 进程中——状态互相污染
- 用户按了 Ctrl+C，主智能体退出，但后台的 Shell 命令还在跑——**孤立进程**（Orphan Process）

Claude Code 在 `src/tasks/` 中实现了完整的任务生命周期管理，解决了以上所有问题。

## 分步实现指南

### 第 1 步：定义任务状态机

💡 **状态机**（State Machine）是一种建模方式，每个任务在任何时刻都处于一个确定的状态，状态之间的转换遵循预定义的规则。状态机的核心价值是**防止非法状态转换**——比如一个 `completed` 的任务不能再变回 `running`。

```
                      ┌──────────┐
                      │ created  │
                      └────┬─────┘
                           │ register() — 分配 ID，持久化
                           ▼
                      ┌──────────┐
              ┌───────│ pending  │←──────────────┐
              │       └────┬─────┘               │
              │            │ pickup() — 工作者    │ retry()
              │            │ 接手，开始心跳       │ retryCount++
              │            ▼                     │
              │       ┌──────────┐               │
              │       │ running  │───────────────┘
              │       └──┬───┬───┘    失败且可重试
              │          │   │
              │  成功     │   │ 失败（不可重试 / 超过重试上限）
              │          ▼   ▼
              │  ┌──────────┐ ┌──────────┐
              │  │completed │ │  failed  │
              │  └──────────┘ └──────────┘
              │
              │  用户取消 / 系统取消
              ▼
         ┌──────────┐
         │cancelled │
         └──────────┘
```

**转换规则是单向的**——除了 `failed → pending`（重试），任务不能后退。这是设计约束，不是实现限制。允许任意转换会让系统行为不可预测。

**每个转换的守卫（Guard）：**

```typescript
function transition(taskId: string, newStatus: TaskStatus) {
  const task = registry.get(taskId)
  
  // 守卫：验证转换合法性
  const allowed = TRANSITIONS[task.status]
  if (!allowed.includes(newStatus)) {
    throw new Error(`非法转换: ${task.status} → ${newStatus}`)
  }
  
  // 执行转换
  task.status = newStatus
  task.updatedAt = Date.now()
  
  // 转换后的副作用
  if (newStatus === 'running') task.startedAt = Date.now()
  if (newStatus === 'completed') notifyParent(task)
  if (newStatus === 'failed') maybeRetry(task)
  if (newStatus === 'cancelled') task.abortController.abort()
}
```

### 第 2 步：构建全局任务注册表

所有任务必须注册在一个全局注册表中。这是任务追踪、UI 显示、清理和僵尸检测的**唯一真相来源**（Single Source of Truth）。

**为什么必须是单例？** 如果有两个注册表，一个任务可能在注册表 A 中显示为 `running`，在注册表 B 中根本不存在。清理时注册表 A 认为任务还在跑而不清理，注册表 B 不知道有这个任务——结果是孤立进程。

```typescript
// 进程级单例
class TaskRegistry {
  private static instance: TaskRegistry
  private tasks = new Map<string, Task>()
  
  static getInstance(): TaskRegistry {
    if (!this.instance) this.instance = new TaskRegistry()
    return this.instance
  }
  
  register(task: Task): void {
    if (this.tasks.has(task.id)) {
      // 重注册支持（重连场景）
      this.tasks.set(task.id, { ...this.tasks.get(task.id), ...task })
      return
    }
    this.tasks.set(task.id, task)
  }
  
  getActive(): Task[] {
    return [...this.tasks.values()].filter(t => 
      t.status === 'pending' || t.status === 'running'
    )
  }
  
  getByAgent(agentId: string): Task[] {
    return [...this.tasks.values()].filter(t => t.agentId === agentId)
  }
  
  prune(): void {
    // 删除已通知父级的终态任务，防止注册表无限增长
    for (const [id, task] of this.tasks) {
      if (['completed', 'failed', 'cancelled'].includes(task.status) && task.notified) {
        this.tasks.delete(id)
      }
    }
  }
}
```

### 第 3 步：实现后台任务执行

后台任务独立于主对话流运行。关键设计决策：

**立即返回 task_id。** 不要让主对话等待后台任务完成。返回 `{ taskId: "abc123" }`，主对话继续工作。任务完成时通过通知传递结果。

**独立的 AbortController。** 每个后台任务有自己的 `AbortController`。用户取消主对话不应该取消所有后台任务——也许用户只想取消主对话但保留测试运行。

**磁盘输出流（Disk Output Stream）。** 任务输出不放在内存中——流式写入磁盘文件。原因：
- 后台任务可能产生 MB 级别的输出（测试报告、构建日志）
- 多个后台任务同时运行，内存压力加倍
- 父级智能体用 ReadFile 工具按需读取，可以用 offset 做增量读取

```typescript
async function runBackgroundTask(task: Task) {
  const outputFile = path.join(tmpDir, `task-${task.id}-output.log`)
  const outputStream = fs.createWriteStream(outputFile)
  
  registry.transition(task.id, 'running')
  
  try {
    const result = await executeTask(task, {
      stdout: outputStream,
      abortSignal: task.abortController.signal,
    })
    registry.transition(task.id, 'completed')
    task.result = result
    injectNotification(task)  // XML 通知注入父级对话
  } catch (err) {
    if (err.name === 'AbortError') {
      registry.transition(task.id, 'cancelled')
    } else {
      registry.transition(task.id, 'failed')
      task.error = err.message
      maybeRetry(task)
    }
  } finally {
    outputStream.close()
  }
}
```

### 第 4 步：心跳与僵尸保护

💡 **僵尸任务**（Zombie Task）是卡在 `running` 状态但没有活跃工作者的任务。产生原因：工作者进程被 kill、OOM、网络断开、或者任务完成逻辑中的 Bug。

**心跳机制：**

长时间运行的任务每 30 秒写入一个心跳时间戳。

```typescript
function startHeartbeat(taskId: string): NodeJS.Timeout {
  return setInterval(() => {
    const task = registry.get(taskId)
    if (task && task.status === 'running') {
      task.lastHeartbeat = Date.now()
    }
  }, 30_000)
}
```

**僵尸收割者（Zombie Reaper）：**

后台进程每 60 秒扫描一次注册表。

```typescript
function startZombieReaper(): NodeJS.Timeout {
  return setInterval(() => {
    const now = Date.now()
    for (const task of registry.getActive()) {
      if (task.status !== 'running') continue
      if (now - task.lastHeartbeat > 90_000) {  // 90 秒没有心跳
        logger.warn(`Zombie detected: task ${task.id}`)
        
        if (task.retryCount < 3) {
          registry.transition(task.id, 'pending')  // 重试
          task.retryCount++
        } else {
          registry.transition(task.id, 'failed')  // 永久失败
          task.error = 'Zombie: exceeded max retries'
        }
      }
    }
  }, 60_000)
}
```

**为什么是 30/60/90 秒？** 心跳间隔（30s）要足够频繁以便及时检测死亡。扫描间隔（60s）不需要太频繁——每分钟一次就够了。超时阈值（90s）= 3 个心跳周期，允许偶尔的心跳延迟（GC 暂停、I/O 阻塞）不被误判为死亡。

### 第 5 步：智能体退出清理

每个智能体在退出时必须清理它生成的后台任务。不清理 = 孤立进程。

**何时发生：** 父级智能体完成工作或被终止，但它启动的后台 Shell 命令还在运行。如果没有清理逻辑，这个 Shell 命令会永远运行——直到用户手动 `kill` 或系统重启。

```typescript
async function runAgent(definition: AgentDefinition) {
  const heartbeatTimer = startHeartbeat(taskId)
  const childTasks: string[] = []
  
  try {
    // ... 智能体运行，可能生成后台子任务 ...
    childTasks.push(subTaskId)
  } finally {
    // ① 停止自己的心跳
    clearInterval(heartbeatTimer)
    
    // ② 取消所有子任务
    for (const childId of childTasks) {
      const child = registry.get(childId)
      if (child && child.status === 'running') {
        child.abortController.abort()
        registry.transition(childId, 'cancelled')
      }
    }
    
    // ③ 清理其他资源
    // MCP 连接断开、Shell 进程终止、文件缓存清除、
    // 消息数组释放、UI 状态条目移除
  }
}
```

**关键：** 清理逻辑必须在 `finally` 块中。如果放在 `try` 末尾，智能体抛出异常时清理不会执行。

### 第 6 步：AsyncLocalStorage 上下文隔离

同一 Node.js 进程中并发运行的多个任务需要隔离状态。`AsyncLocalStorage` 提供了这种隔离，不需要通过每个函数调用传递上下文对象。

```typescript
import { AsyncLocalStorage } from 'node:async_hooks'

interface TaskContext {
  taskId: string
  agentId: string
  cwd: string
  permissionContext: PermissionContext
  fileCache: Map<string, string>
}

const taskStorage = new AsyncLocalStorage<TaskContext>()

// 每个任务在自己的隔离上下文中运行
async function executeInContext(task: Task, work: () => Promise<void>) {
  const context: TaskContext = {
    taskId: task.id,
    agentId: task.agentId,
    cwd: task.workingDirectory,
    permissionContext: createPermissionContext(task),
    fileCache: new Map(),
  }
  
  await taskStorage.run(context, work)
  // work 内部及其所有嵌套异步调用都能通过
  // taskStorage.getStore() 访问到 context
}
```

**嵌套上下文：** 如果任务 A 的执行过程中启动了子任务 B，子任务 B 在自己的 `run()` 中运行，它的 `getStore()` 返回子任务上下文——不会看到父任务的状态。这是正确的隔离行为。

**不要跨边界共享可变对象。** 如果两个任务的 `context.fileCache` 引用同一个 `Map`，一个任务修改缓存会影响另一个任务。每个任务的上下文对象必须是独立创建的。

### 第 7 步：通知系统

任务完成时通过 XML 通知注入父级对话，让模型知道发生了什么。

```typescript
function injectNotification(task: Task) {
  const notification = `<task-notification task_id="${task.id}" status="${task.status}">
  Task "${task.description}" ${task.status}.
  ${task.result ? `Result: ${task.result.summary}` : ''}
  ${task.error ? `Error: ${task.error}` : ''}
</task-notification>`
  
  parentConversation.injectMessage({
    role: 'user',
    content: notification,
  })
  
  task.notified = true
}
```

**注入时机：** 在父级对话的下一轮开始前，不是中间。中间注入会打断模型当前的推理链。

## 决策矩阵

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 子智能体执行研究任务 | 后台任务 + XML 通知 | 不阻塞主对话，结果异步传递 |
| Shell 命令 `npm test` | 后台任务 + 磁盘输出流 | 输出可能很大，不应占内存 |
| 串行的文件编辑 | 前台任务（不需要任务系统） | 阻塞式执行，直接返回结果 |
| Swarm 工作者 | 后台任务 + 心跳 + 僵尸检测 | 工作者可能崩溃，需要恢复 |
| 一次性计算 | 不需要任务系统 | 开销不值得 |

## 常见陷阱

**1. 多个注册表实例。** 模块被不同路径导入（相对路径 vs 绝对路径、CommonJS vs ESM 双重实例）导致创建了两个"单例"。用 `globalThis` 或进程级变量确保真正的唯一性。

**2. 清理不在 finally 中。** 把 `clearInterval(heartbeat)` 放在 `try` 末尾。智能体抛异常时心跳计时器继续运行——它每 30 秒更新一个已经不存在的任务的心跳，占用 CPU 且阻止僵尸检测。

**3. 注册表无限增长。** 不调用 `prune()`，已完成的任务永远留在注册表中。运行几千个任务后，`getActive()` 需要遍历几千个条目，其中只有 5 个是活跃的。定期修剪已通知的终态任务。

**4. 输出全放内存。** 10 个后台任务各产生 5MB 输出 = 50MB 内存。如果任务输出用字符串累积而不是流式写磁盘，长时间运行后 OOM。

**5. 心跳太频繁或太稀疏。** 1 秒一次心跳：10 个任务 = 每秒 10 次文件 I/O，对文件系统造成不必要的压力。5 分钟一次心跳：工作者崩溃后 15 分钟才能被检测——太慢。30 秒是经过实践验证的平衡点。

**6. 忘记处理 AbortError。** `AbortController.abort()` 会导致正在 `await` 的操作抛出 `AbortError`。如果没有专门的 catch 分支处理它，`AbortError` 会被当作普通错误走重试逻辑——用户明确取消了但任务又重新开始了。

**7. AsyncLocalStorage 在 Worker Threads 中不共享。** `AsyncLocalStorage` 是线程局部的（Thread-Local）。如果你用 `worker_threads` 做并发，每个 Worker Thread 需要自己的 `AsyncLocalStorage` 实例。这通常不是 Claude Code 的场景（它用异步并发而非线程并发），但如果你在其他环境中复用这个模式需要注意。
