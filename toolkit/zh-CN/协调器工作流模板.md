# 协调器工作流模板

## 问题场景

用户说："把我们的支付模块从 Stripe v2 API 迁移到 v3。" 这个任务涉及后端路由、前端组件、数据库 Schema、Webhook 处理器、测试——一个智能体按顺序做所有事情要么太慢（串行读取 50 个文件），要么质量差（在处理后端时已经忘了前端的细节）。

协调器工作流（Coordinator Workflow）是解决这类复杂任务的标准模式：一个主导智能体（协调者）通过结构化的多阶段流程编排多个子智能体，确保先理解再行动、先验证再交付。

Claude Code 在 `src/coordinator/coordinatorMode.ts` 中实现了这个模式。

## 架构概览

```
┌──────────────────────────────────────────────────────────────┐
│                      协调者 (Coordinator)                     │
│                                                              │
│  阶段 1: 研究 (Fan-Out)                                      │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐                     │
│  │研究员 A   │ │研究员 B   │ │研究员 C   │  ← 后台并行        │
│  │后端 API   │ │前端组件   │ │测试覆盖   │                    │
│  └─────┬─────┘ └─────┬─────┘ └─────┬─────┘                  │
│        │             │             │                         │
│        └──────┬──────┴──────┬──────┘                         │
│               ▼             ▼                                │
│       scratchpad/       scratchpad/                           │
│       research-*.md     research-*.md                        │
│                                                              │
│  阶段 2: 综合 (Synthesis) ← 协调者自己做，不委托             │
│  ┌──────────────────────────────────────┐                    │
│  │ 读取所有研究结果 → 写出逐文件计划      │                    │
│  │ → scratchpad/plan.md                 │                    │
│  └──────────────────────────────────────┘                    │
│                                                              │
│  阶段 3: 实现 (Implementation)                               │
│  ┌──────────┐ ┌──────────┐                                   │
│  │实现者 A   │ │实现者 B   │  ← 前台串行（避免文件冲突）      │
│  │后端修改   │ │前端修改   │                                  │
│  └──────────┘ └──────────┘                                   │
│                                                              │
│  阶段 4: 验证 (Verification) ← 全新智能体，无实现上下文      │
│  ┌──────────────────────────────────────┐                    │
│  │ 验证者：运行测试、审查 diff、检查回归   │                   │
│  └──────────────────────────────────────┘                    │
└──────────────────────────────────────────────────────────────┘
```

## 分步实现指南

### 阶段 1：研究（扇出 / Fan-Out）

协调者在**一个轮次**中并行生成多个研究智能体。每个研究智能体获得一个**具体的、聚焦的问题**，不是模糊的指令。

**好的研究指令 vs 坏的研究指令：**

| 坏（模糊） | 好（具体） |
|-----------|-----------|
| "调查支付模块" | "列出 `src/payments/` 下所有调用 `stripe.charges.create()` 的文件路径和行号" |
| "了解前端的支付流程" | "在 `src/components/` 中找到所有引用 `PaymentForm` 或 `CheckoutFlow` 的组件，列出它们的 props 接口" |
| "看看测试情况" | "运行 `npm test -- --testPathPattern=payment` 并报告：通过数、失败数、覆盖的关键路径" |

**为什么研究员只收集事实，不提建议？** 因为研究员只看到系统的一个切面。研究员 A 看到后端有 15 个文件用了旧 API，可能建议"全部替换"——但它不知道研究员 C 发现其中 3 个文件的测试依赖旧 API 的特定行为。综合是协调者的工作，需要全局视角。

**关键实现细节：**
- 使用**后台模式**生成研究智能体——它们之间是独立的
- 在**一条消息中生成所有研究智能体**（一个协调者轮次，多个 AgentTool 调用）。不要一个一个生成——那是不必要的串行化
- 研究员将发现写入**草稿板文件**（`scratchpad/research-{topic}.md`），不要只通过通知传递——文件可以被后续阶段的智能体独立读取
- 研究员不应该有写文件的权限（除了草稿板）——给它们工具白名单：`ReadFile`、`Grep`、`Bash`（只读命令）

### 阶段 2：综合

所有研究通知到达后，协调者**自己**分析结果。

这是协调器模式的核心价值所在——协调者是唯一能看到所有研究结果的智能体。它必须做以下事情：

1. **读取所有研究报告**（从草稿板文件）
2. **识别冲突和依赖**。研究员 A 说"文件 X 需要修改"，研究员 B 说"文件 X 的类型定义不能变"——协调者需要解决这个矛盾
3. **写出逐文件的实现计划**。不是"迁移支付模块"这种抽象计划，而是：
   - `src/payments/charge.ts`：将 `stripe.charges.create()` 替换为 `stripe.paymentIntents.create()`，参数映射：`source` → `payment_method`，`amount` → `amount`（不变）
   - `src/payments/webhook.ts`：添加 `payment_intent.succeeded` 事件处理，保留 `charge.succeeded` 向后兼容
4. **将计划写入草稿板**（`scratchpad/plan.md`）

**为什么不能把综合委托给子智能体？** 因为综合需要判断力——当两个研究结果矛盾时，需要决定信哪个。这个判断依赖对用户意图的理解，而只有协调者（参与了用户对话）有这个上下文。

### 阶段 3：实现

根据计划执行修改。

**小改动（< 5 个文件）**：协调者直接实现。为 3 行改动生成一个子智能体的开销（上下文创建、模型调用、清理）远超直接编辑。

**大改动（> 5 个文件）**：委托给实现智能体，但给出**精确的指令**：

```
修改 src/payments/charge.ts：
1. 第 23 行：将 stripe.charges.create({ source, amount, currency })
   替换为 stripe.paymentIntents.create({ payment_method: source, amount, currency })
2. 第 45 行：将返回值从 charge.id 改为 paymentIntent.id
3. 遵循现有的错误处理模式（try/catch + logger.error）
```

**文件冲突管理：**
- 如果两个实现智能体需要修改同一个文件 → **必须串行**（使用前台模式）
- 如果两个实现智能体修改不同的文件 → 可以并行（使用后台模式）
- 实现前检查：文件自研究阶段以来有没有被修改过？如果有，需要重新评估计划

### 阶段 4：验证

一个**全新的智能体**——没有参与过研究或实现——验证工作。

**为什么必须是全新的？** 这是协调器模式中最重要的洞察。做了实现的智能体会倾向于相信自己的工作是正确的。它"知道"自己的意图，所以会忽略实际的问题。心理学上叫**确认偏差**（Confirmation Bias）。全新的智能体没有这个包袱。

验证智能体的工作清单：
1. 运行完整测试套件，比对修改前后的结果
2. 运行类型检查（`tsc --noEmit`），确认没有类型错误
3. 用 `git diff` 审查所有变更，检查：
   - 有没有非计划内的修改？
   - 有没有调试代码或注释遗留？
   - 修改是否一致（同一种 API 变更在所有文件中用了相同的模式）？
4. 将验证结果写入草稿板（`scratchpad/verification.md`）

## 决策矩阵：什么时候用什么变体

| 场景 | 推荐变体 | 理由 |
|------|---------|------|
| 横切变更（涉及多子系统） | 完整四阶段 | 需要先理解所有子系统再行动 |
| 改动已明确（"把 X 改成 Y"） | 跳过研究，直接实现 + 验证 | 不需要研究——计划已知 |
| 纯调研（"我们应该用什么方案？"） | 研究 + 综合 + 报告（跳过实现和验证） | 只需要分析，不需要修改代码 |
| 快速修复（改 1-2 个文件） | 不用协调器——直接在当前智能体中做 | 四阶段的开销远超收益 |
| 迭代开发（改了 → 测试失败 → 再改） | 循环变体：实现 → 验证 → 如果失败则回到实现 | 限制最大迭代 2-3 次，防止死循环 |

## 草稿板（Scratchpad）使用规范

草稿板是临时文件系统目录，用于阶段间共享数据。

**为什么用文件而不是消息？**

| 维度 | 文件（草稿板） | 消息（对话上下文） |
|------|--------------|-------------------|
| 持久性 | 跨智能体生命周期 | 智能体终止时丢失 |
| 访问范围 | 任何智能体都能读 | 仅父子关系 |
| 上下文成本 | 按需读取部分内容 | 始终占用上下文窗口 |
| 审计 | 事后可检查 | 需要日志才能保留 |
| 大小限制 | 无限制 | 受上下文窗口限制 |

**文件命名约定：**
```
scratchpad/
  research-backend-api.md       # 研究员 A 的发现
  research-frontend-components.md  # 研究员 B 的发现
  research-test-coverage.md     # 研究员 C 的发现
  plan-v1.md                    # 协调者的实现计划（版本 1）
  plan-v2.md                    # 修订计划（如果验证发现问题）
  verification-report.md        # 验证智能体的结果
```

**清理时机：** 任务完成后在 `finally` 块中删除临时目录。不清理 = 长时间运行后磁盘上积累几百个临时目录。

## 检查清单

### 任务评估（决定是否使用协调器）

- [ ] 任务涉及 3 个以上文件或 2 个以上子系统？（是 → 考虑协调器）
- [ ] 需要先理解系统才能修改？（是 → 需要研究阶段）
- [ ] 对正确性要求高？（是 → 需要验证阶段）
- [ ] 改动已经明确？（是 → 跳过研究，直接实现 + 验证）

### 研究阶段

- [ ] 每个研究员都收到了具体的、可执行的问题？
- [ ] 研究员只收集事实，不做建议？
- [ ] 所有研究员在一个协调者轮次中并行生成？
- [ ] 研究结果写入草稿板文件，不只在通知中？
- [ ] 研究员的工具白名单限制为只读工具？

### 综合阶段

- [ ] 协调者自己做综合，没有委托给子智能体？
- [ ] 计划是逐文件的，包含具体的修改内容？
- [ ] 矛盾和依赖被识别和解决？
- [ ] 计划写入草稿板文件？

### 实现阶段

- [ ] 小改动由协调者直接实现？
- [ ] 大改动委托时给了精确的、逐文件的指令？
- [ ] 涉及同一文件的修改用前台模式串行执行？
- [ ] 不同文件的修改可以后台并行？

### 验证阶段

- [ ] 验证者是全新的智能体（没有实现阶段的上下文）？
- [ ] 验证者运行了测试和类型检查？
- [ ] 验证者审查了 `git diff`？
- [ ] 验证结果写入草稿板？

## 常见陷阱

**1. 综合委托。** "根据你的研究结果修复它"——研究员只看到系统的一个切面，它的"修复"会是局部最优。综合需要全局视角，只有协调者有。

**2. 验证者继承上下文。** 让实现智能体"顺便"验证自己的工作。这等于没有验证——实现者不会发现自己的 Bug。全新上下文是不可协商的。

**3. 在同一文件上并行实现。** 两个智能体同时修改 `auth.ts`——后写的覆盖先写的变更，导致数据丢失。涉及重叠文件时必须串行。

**4. 模糊的研究指令。** "调查认证系统"太模糊——研究员不知道该看什么，返回一堆不聚焦的信息。"列出 `src/auth/` 下所有调用 `req.session` 的文件路径"才是好指令。

**5. 跳过验证。** "测试通过了就行"——但你怎么知道测试覆盖了修改的场景？验证智能体的价值不只是跑测试，还包括审查 diff 和检查非测试覆盖的问题。

**6. 过多阶段用于简单任务。** 改一个 typo 不需要四阶段流程。协调器的基础开销（4+ 次模型调用、草稿板 I/O、智能体创建/清理）对于简单任务来说是纯浪费。

**7. 无限迭代。** "验证失败 → 重新实现 → 再验证 → 再失败 → ..."。设置最大迭代次数（建议 2-3 次），超过后停下来报告问题让用户决策。
