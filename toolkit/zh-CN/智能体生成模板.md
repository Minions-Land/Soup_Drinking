# 智能体生成模板

## 问题场景

你的系统需要动态创建子智能体（Sub-Agent）来处理工作。但你立刻面临一系列设计决策：

- 子智能体应该和父级共享对话历史，还是从空白开始？
- 子智能体应该阻塞父级直到完成（同步），还是在后台运行（异步）？
- 子智能体崩溃了怎么办？它占用的资源（MCP 连接、文件句柄、Shell 进程）谁负责清理？
- 多个子智能体并发运行时，它们的状态如何隔离？

Claude Code 在 `src/tools/AgentTool/` 中实现了完整的智能体生成系统，支持三种生成模式，解决了上述所有问题。

## 智能体定义格式

每个智能体需要一个定义来描述其身份、能力和约束。Claude Code 在 `src/tools/AgentTool/built-in/` 下为每种内置智能体类型维护一个定义文件。

```typescript
type AgentDefinition = {
  agentType: string              // 唯一名称："code-reviewer", "researcher"
  whenToUse: string              // 注入父级 prompt 的描述——模型据此选择
  source: 'built-in' | 'user' | 'project' | 'plugin'  // 来源分类
  
  // 能力约束
  tools?: string[]               // 工具白名单（只能用这些工具）
  disallowedTools?: string[]     // 工具黑名单（不能用这些工具）
  model?: string                 // 模型覆盖，或 'inherit'（继承父级）
  maxTurns?: number              // 最大对话轮数（防止失控）
  
  // 权限和隔离
  permissionMode?: 'default' | 'plan' | 'bubble'
  isolation?: 'worktree'         // Git worktree 隔离（独立工作目录）
  
  getSystemPrompt(): string      // 智能体的系统提示词
}
```

**`whenToUse`** 是最重要的字段之一——它直接注入父级智能体的系统提示词中，父级模型根据这个描述决定是否生成这个子智能体。写得不好，模型就不会用或者在错误的时候用。

**`maxTurns`** 是安全阀。没有它，一个卡住的子智能体会无限循环消耗 Token。Claude Code 的内置智能体通常设置 10-30 轮。

## 三种生成模式详解

### 模式 1：前台（同步阻塞）

父级智能体调用子智能体后阻塞，直到子智能体完成才继续。

**调用流程：**
1. 父级调用 `AgentTool`，指定前台模式
2. 子智能体获得**共享的中断控制器**（AbortController）——用户在父级终端按 Ctrl+C 会同时取消父级和子级
3. 子智能体的权限提示（"允许写入 src/auth.ts 吗？"）显示在父级的终端
4. 子智能体完成后，结果直接作为 `AgentTool` 的返回值传回父级

**什么时候用：**
- 父级的下一步决策依赖子级的结果
- 修改文件时需要串行执行避免冲突
- 需要用户审批子级的操作（权限提示在父级终端可见）

**什么时候不用：**
- 多个子任务可以并行——前台模式会串行执行，浪费时间
- 子任务与父任务独立——阻塞父级没有必要

### 模式 2：后台（异步独立）

父级智能体发起子智能体后立即继续自己的工作。子智能体在后台独立运行。

**调用流程：**
1. 父级调用 `AgentTool`，指定后台模式
2. 系统立即返回一个 `task_id`，父级继续工作
3. 子智能体获得**独立的中断控制器**——父级取消不影响子级（除非显式取消）
4. 子智能体的权限提示**自动拒绝**——因为它没有终端来询问用户
5. 子智能体完成时，通过 XML 通知注入父级对话：`<task-notification task_id="xxx" status="completed">`

**什么时候用：**
- 多个真正独立的并行任务：运行测试、构建、格式检查
- 长时间运行的研究任务，父级不需要等结果
- 子任务不涉及需要用户审批的操作

**什么时候不用：**
- 子任务需要写入和其他任务相同的文件——并发写入冲突
- 子任务需要用户交互审批——后台模式会自动拒绝所有权限请求

### 模式 3：分叉（上下文继承）

💡 **分叉（Fork）** 指子智能体继承父级的完整对话历史和系统提示词的精确字节副本，复用 LLM 的 Prompt Cache。

**调用流程：**
1. 父级调用 `forkSubagent()`（`src/tools/AgentTool/forkSubagent.ts`）
2. 系统复制父级的 `renderedSystemPrompt` 和对话前缀的精确字节
3. 在副本末尾追加新任务指令
4. 子智能体获得独立的中断控制器
5. 子智能体始终在后台运行
6. **必须使用与父级相同的模型**——不同模型的系统提示格式不同，无法命中 Prompt Cache

**什么时候用：**
- 子智能体的任务建立在父级已有的研究上下文之上
- 父级已经读过大量文件、理解了项目架构，子级需要这些知识
- 需要节省 Token（Prompt Cache 命中 = 零重新分词成本）

**什么时候不用：**
- 子智能体的任务与父级上下文无关——复制是纯浪费
- 需要用不同的模型（更便宜/更快）
- 父级上下文很小（< 2K Token）——缓存收益可忽略

## 上下文继承矩阵

| 组件 | 前台 | 后台 | 分叉 |
|------|------|------|------|
| 对话消息 | 全新 | 全新 | 父级的完整副本 |
| 系统提示 | 智能体自己的 | 智能体自己的 | 父级的精确字节 |
| 中断控制器 | 共享（父子联动） | 独立 | 独立 |
| 文件读取缓存 | 全新 | 全新 | 从父级克隆 |
| 权限上下文 | 继承规则 | 继承规则 + 自动拒绝 | 继承规则 + 冒泡 |
| Prompt Cache | 不共享 | 不共享 | 共享（核心优势） |
| 模型选择 | 可不同 | 可不同 | 必须相同 |

## 决策矩阵：选择哪种模式

```
你的下一步是否依赖子级的结果？
├── 是 → 前台模式（阻塞父级，结果直接可用）
└── 否 → 子级是否需要父级的对话上下文？
         ├── 是 → 分叉模式（继承一切，共享缓存，必须同模型）
         └── 否 → 后台模式（全新上下文，独立生命周期）
```

| 场景 | 推荐模式 | 理由 |
|------|---------|------|
| "修改 auth.ts 然后更新相关测试" | 前台 | 第二步依赖第一步的修改结果 |
| "搜索所有用了废弃 API 的文件" | 后台 | 不影响主对话，结果通过通知传递 |
| "基于我们刚才讨论的架构，深入研究数据库层" | 分叉 | 需要父级的讨论上下文 |
| "并行运行测试、lint、类型检查" | 后台 × 3 | 三个任务完全独立 |
| "先研究后端 API，再研究前端调用" | 后台（如果独立）或前台（如果有依赖） | 取决于任务间是否有依赖 |

## 通信机制

### 1. 通知（Notification）
后台和分叉智能体完成时注入 `<task-notification>` XML 消息到父级对话。通知包含 `task_id`、`status`、摘要内容。父级模型在下一轮看到通知并据此行动。

### 2. 进度回调（Progress Callback）
子智能体在执行期间通过 `onProgress` 报告进度。父级可以实时看到"子智能体正在搜索 src/..."。

### 3. SendMessage
💡 **SendMessage** 允许在子智能体完成后，从外部向其发送新指令继续工作。适用于追加任务或修正方向。

### 4. 邮箱（Mailbox）
在 Swarm/Team 场景中，智能体之间的点对点通信。基于文件系统实现——每个智能体有一个收件箱文件。详见 Swarm 团队模板。

## 生命周期清理

这是最容易出 Bug 的地方。智能体启动时分配的**每个资源**都必须在 `finally` 块中释放：

```typescript
async function runAgent(definition, context) {
  const mcpConnections = []
  const shellProcesses = []
  
  try {
    // ... 智能体运行 ...
  } finally {
    // 必须清理的资源（按顺序）：
    for (const conn of mcpConnections) await conn.disconnect()  // 1. MCP 连接
    for (const proc of shellProcesses) proc.kill('SIGTERM')     // 2. Shell 进程
    context.fileCache.clear()                                    // 3. 文件缓存
    context.messages.length = 0                                  // 4. 消息数组（帮助 GC）
    taskRegistry.remove(taskId)                                  // 5. 任务注册
    statusBar.removeEntry(agentId)                               // 6. UI 状态
  }
}
```

忘记清理的后果：
- MCP 连接不断开 → 连接泄漏，最终耗尽连接池
- Shell 进程不终止 → 孤立进程（Orphan Process），持续占用 CPU/内存
- 消息数组不释放 → 内存泄漏，长时间运行后 OOM

## 检查清单

### 设计阶段

- [ ] 确认了子智能体的 `agentType` 和 `whenToUse`？
- [ ] 选择了正确的生成模式（前台/后台/分叉）？
- [ ] 确认了 `maxTurns` 上限？
- [ ] 确认了工具白名单/黑名单——子智能体不应该有比父级更多的权限？
- [ ] 确认了权限模式——后台智能体自动拒绝是否可接受？

### 实现阶段

- [ ] `finally` 块清理了所有资源？（MCP、Shell、缓存、消息、注册、UI）
- [ ] 中断控制器（AbortController）的隔离正确吗？
- [ ] 分叉模式复制的是精确字节，不是重新渲染的系统提示？
- [ ] 后台智能体的结果通过通知传递，不是轮询？
- [ ] 子智能体崩溃时，任务状态被正确更新为 `failed`？

### 测试阶段

- [ ] 子智能体正常完成——结果传递正确？
- [ ] 子智能体超时——`maxTurns` 限制生效？
- [ ] 子智能体崩溃——资源被清理、任务标记失败？
- [ ] 父级取消——前台模式子级也被取消？后台/分叉模式子级不受影响？
- [ ] 并发子智能体——状态隔离（AsyncLocalStorage）正确？

## 常见陷阱

**1. 后台智能体需要写文件但权限被自动拒绝。** 后台模式没有终端来询问用户。如果子智能体需要写文件，要么用前台模式，要么在父级的权限配置中预先添加 Allow 规则。

**2. 分叉后修改系统提示词。** 任何修改都破坏 Prompt Cache 的字节匹配。分叉的系统提示必须和父级完全一致——这是"免费午餐"的代价。

**3. 无限生成。** 子智能体有权限再生成子智能体。A 生成 B，B 生成 C，C 生成 D... 没有深度限制就可能耗尽系统资源。设置最大嵌套深度（Claude Code 限制为 3-4 层）。

**4. 在 finally 块之外做清理。** 把清理代码放在 `try` 块末尾而不是 `finally` 块——智能体抛出异常时清理不会执行。`finally` 是唯一保证执行的位置。

**5. 生成太多智能体做简单任务。** 改一行代码不需要生成研究智能体 + 实现智能体 + 验证智能体。直接在当前智能体中做。生成的开销（上下文创建、模型调用、清理）对于小任务来说不划算。
