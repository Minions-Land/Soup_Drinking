# Swarm 团队模板

## 问题场景

你有 200 个测试文件需要更新 import 路径，或者 150 个 React 组件需要从 Class Component 迁移到 Function Component。这些任务有两个特点：

1. **量大**——一个智能体按顺序做需要几个小时
2. **独立**——Task A 和 Task B 互不影响，可以并行

如果用协调器模式（一个领导者分配工作），领导者本身成了瓶颈：它要分析每个任务、分配给工作者、等待结果、分配下一个。200 个任务就意味着 200 次领导者的决策轮次。

Swarm（群体）模式的核心思路是去中心化：领导者只定义任务列表，工作者自主从共享队列中领取任务、执行、标记完成。领导者不参与过程管理。

Claude Code 在 `src/utils/swarm/` 中实现了完整的 Swarm 基础设施。

## 队列协议详解

共享工作队列是 Swarm 模式的核心数据结构。Claude Code 在 `src/utils/swarm/` 中提供了基于文件系统的实现。

### 任务状态机

```
                  ┌─────────┐
                  │ pending │ ← 初始状态
                  └────┬────┘
                       │ 工作者领取
                       ▼
                  ┌─────────┐
              ┌───│ claimed │
              │   └────┬────┘
              │        │ 执行完毕
              │        ▼
              │   ┌─────────┐
              │   │completed│ ← 终态
              │   └─────────┘
              │
              │ 执行失败 / 工作者崩溃
              ▼
         ┌─────────┐    重试（retryCount < max）
         │ failed  │──────────────────────────→ pending
         └─────────┘
```

### 每步做什么

**1. 工作者轮询队列。** 读取任务列表，找到 `status: 'pending'` 且依赖已满足的任务。如果没有可用任务，等待后重试（建议间隔 2-5 秒）。

**2. 工作者原子性领取任务。** 两个工作者同时看到同一个 `pending` 任务时，只有一个能成功领取。

文件系统上的原子领取有两种实现方式：

**方式 A：原子重命名**（推荐，更简单）
```
task-pending-001.json → task-claimed-worker2-001.json
```
文件系统的 `rename()` 是原子操作。第一个执行 `rename` 的工作者成功，第二个因为源文件已不存在而失败。

**方式 B：单文件 + 版本号**（适合任务数少的场景）
```json
{
  "id": "001",
  "status": "pending",
  "version": 3,
  "claimedBy": null
}
```
工作者读取 version=3，尝试写入 `{ status: "claimed", version: 4, claimedBy: "worker-2" }`。写入前检查 version 是否仍然为 3——如果不是，说明被其他工作者抢先了。

**3. 工作者执行任务。** 执行过程中定期发出心跳（建议每 30 秒一次）。心跳是一个时间戳文件或字段，表明"我还活着"。

**4. 工作者标记结果。** 成功：`status: 'completed'`，附带结果摘要。失败：`status: 'failed'`，附带错误信息，增加 `retryCount`。

**5. 工作者发现新任务（可选）。** 执行过程中可能发现额外工作——比如修复一个文件时发现关联文件也需要修改。工作者可以创建新任务添加到队列。

**6. 重复直到队列为空。** 所有任务都到达终态（`completed` 或永久 `failed`）时，领导者综合结果。

## 自主性光谱：三种团队模式

| 模式 | 领导者角色 | 工作者自主性 | 适用场景 |
|------|-----------|-------------|---------|
| **Swarm（高自主性）** | 创建任务列表后旁观 | 完全自主：自选任务、自决方法 | 大量独立任务（lint 修复、测试更新） |
| **Team（平衡）** | 创建任务 + 监控 + 卡住时介入 | 自主执行但可互相通信 | 有松散依赖的并行工作 |
| **Managed Team（低自主性）** | 分配具体任务给具体工作者 | 按指令执行、报告完毕 | 需要精确控制的关键路径任务 |

### Swarm 模式

- 工作者从队列自主拉取，不需要领导者分配
- 工作者能力和权限相同（同构 / Homogeneous）
- 工作者之间通过邮箱通信（通知发现、请求协助）
- 领导者在所有任务完成前不做任何事

### Team 模式

- 领导者创建初始任务列表和高级目标
- 工作者可以创建子任务和互相通信
- 领导者定期检查进度，在以下情况介入：
  - 某个工作者连续失败 3 个任务
  - 两个工作者通过邮箱报告了冲突
  - 总进度停滞超过预期时间
- 适合任务间有松散耦合的场景

### Managed Team 模式

- 领导者明确分配：工作者 A 做 Task 1-5，工作者 B 做 Task 6-10
- 工作者完成后向领导者报告，领导者决定所有下一步
- 接近协调器模式，但工作者仍有一定执行自主性
- 适合任务间有强依赖、需要精确排序的场景

## 后端选项

| 后端 | 实现 | 隔离级别 | 适用场景 |
|------|------|---------|---------|
| **进程内** | 所有工作者在同一 Node.js 进程 | AsyncLocalStorage | 2-5 个轻量工作者，开发调试 |
| **tmux 会话** | 每个工作者独立 CLI 进程 | 进程级 | 需要可视化监控，长时间任务 |
| **iTerm2** | macOS iTerm2 原生分割面板 | 进程级 | macOS 开发，更丰富的 UI |
| **Docker 容器** | 每个工作者运行在容器中 | 完全隔离 | 不信任的代码执行，生产部署 |

**进程内**是 Claude Code 的默认选择，也是最简单的方案。所有工作者共享一个 Node.js 进程，通过 `AsyncLocalStorage` 隔离状态。权限提示冒泡到领导者的终端。限制是单进程的 CPU 和内存上限。

**tmux 会话**（`src/utils/swarm/` 支持）让每个工作者在独立终端中运行。开发者可以 `tmux attach` 看到每个工作者的实时输出，甚至手动干预。会话在终端关闭后存活——适合长时间运行的任务。

## 崩溃恢复

工作者可能在任务执行中崩溃（网络断开、OOM、进程被 kill）。Swarm 系统必须检测并恢复。

### 心跳机制

```
工作者每 30 秒写入心跳时间戳
领导者/其他工作者每 60 秒扫描一次
如果某工作者的心跳 > 90 秒前 → 判定为崩溃
```

### 崩溃恢复流程

1. **检测**：心跳超时
2. **标记**：工作者状态设为 `crashed`
3. **释放任务**：工作者领取的 `claimed` 任务释放回 `pending`，`retryCount++`
4. **重试决策**：
   - `retryCount < 3` → 任务回到 `pending`，其他工作者可以领取
   - `retryCount >= 3` → 任务标记为永久 `failed`，记录到失败报告
5. **通知**：广播崩溃事件，其他工作者知道队列中有释放的任务

### 重连协议

工作者重启后的行为：

1. 检查自己的状态文件——是否有上次的身份信息？
2. 如果有：检查上次领取的任务是否仍由自己持有
   - 是 → 从检查点恢复（如果任务支持检查点）
   - 否（已被释放）→ 作为新工作者加入
3. 如果没有：作为新成员加入团队
4. 注册心跳计时器（每 30 秒）

## 邮箱系统

工作者之间的点对点通信。基于文件系统实现，每个工作者一个收件箱文件。

```
mailbox/
  worker-1-inbox.jsonl    # 工作者 1 的收件箱
  worker-2-inbox.jsonl    # 工作者 2 的收件箱
  leader-inbox.jsonl      # 领导者的收件箱
```

每条消息一行 JSON（JSONL 格式），追加写入。

```json
{"from": "worker-1", "to": "worker-2", "type": "info", "content": "file X imports from file Y, FYI", "timestamp": 1712345678}
```

**使用场景：**
- 工作者发现了和另一个工作者任务相关的信息
- 工作者请求协助（"我不确定这个类型断言是否安全"）
- 工作者通知领导者发现了计划外的问题

**不应该用邮箱的场景：**
- 任务分配——用工作队列
- 状态同步——用权限状态文件
- 大数据传递——用草稿板文件

## 权限同步

Swarm 中多个工作者可能需要相同的权限（比如编辑 `src/` 下的文件）。如果每个工作者都弹出权限提示，用户要批准 N 次同样的操作。

解决方案：**共享权限状态文件**。

```json
// permissions-state.json
{
  "rules": [
    { "tool": "WriteFile", "pattern": "src/**/*.ts", "behavior": "allow", "grantedAt": 1712345678 }
  ]
}
```

当工作者 A 获得"允许编辑 `src/`"的批准后，写入权限状态文件。工作者 B 在检查权限时先读取状态文件——如果有匹配规则，直接使用，不再询问用户。

**冲突解决：** 最后写入优先（Last-Write-Wins）。这是简单且对于权限场景足够好的策略——权限只会变得更宽松（grant）或被重置（会话结束），不存在两个工作者同时缩小权限的场景。

## 决策矩阵

| 条件 | Swarm | Team | Managed Team | 协调器 |
|------|-------|------|-------------|--------|
| 任务数量 | > 10 | 5-20 | 3-10 | 1-5 |
| 任务独立性 | 完全独立 | 松散耦合 | 有依赖 | 有强依赖 |
| 工作者同构性 | 完全相同 | 大致相同 | 可能不同 | 角色分化 |
| 领导者参与度 | 最小（发布+等待） | 监控+介入 | 分配+协调 | 全程编排 |
| 基础设施复杂度 | 中（队列+心跳） | 高（队列+邮箱+监控） | 低（直接分配） | 低 |
| 扩展性 | 线性 | 线性 | 受限于领导者 | 受限于领导者 |

## 检查清单

### 设计阶段

- [ ] 任务是独立的吗？（有依赖 → 用协调器，不用 Swarm）
- [ ] 任务是同构的吗？（需要不同能力 → 用 Team 或 Managed Team）
- [ ] 任务数量 > 5 吗？（少于 5 个 → Swarm 开销不值得）
- [ ] 确认了工作者数量上限？（受 API 速率限制约束）
- [ ] 选择了合适的后端？（进程内 / tmux / Docker）

### 实现阶段

- [ ] 任务领取是原子性的？（原子重命名 / 版本号检查）
- [ ] 心跳间隔设置了？（建议 30 秒）
- [ ] 心跳超时设置了？（建议 90 秒）
- [ ] 最大重试次数设置了？（建议 3 次）
- [ ] 崩溃恢复流程：释放任务 + retryCount++ + 通知？
- [ ] 权限同步：共享权限状态文件？
- [ ] 邮箱系统：基于文件的收件箱 + 并发安全？

### 运行阶段

- [ ] 监控：活跃工作者数、队列深度、完成率、失败率？
- [ ] 异常检测：某工作者连续失败 → 通知领导者？
- [ ] 清理：任务完成后删除临时文件（队列、邮箱、心跳）？
- [ ] 超时：总任务时间上限？（防止无限运行）

## 常见陷阱

**1. 单文件队列。** 所有任务和状态放在一个 JSON 文件里。工作者数量 > 3 时，文件读写锁竞争成为瓶颈。用每任务一文件的方案——200 个任务 = 200 个小文件，原子重命名替代文件锁。

**2. 领导者轮询。** 领导者每秒检查"有没有任务完成了"。应该用通知机制——工作者完成任务时通过邮箱或通知告知领导者，领导者不需要主动轮询。

**3. 无工作者上限。** 生成 50 个并发工作者，每个都在调用 LLM API。API 的速率限制（Rate Limit）会导致大量 429 错误。根据 API 配额设置工作者上限——通常 3-8 个工作者是合理范围。

**4. 任务粒度太细。** 200 个 1 行修改各自是一个任务。工作者领取、初始化上下文、调用模型的开销可能比实际修改还大。把相关的小修改合并成一个任务——比如"修复 `src/utils/` 目录下所有 10 个 lint 告警"作为一个任务，而不是 10 个任务。

**5. 任务粒度太粗。** 把"迁移整个支付模块"作为一个 Swarm 任务——这不是 Swarm 该干的事。Swarm 处理的是大量细粒度独立任务，不是少量粗粒度复杂任务。

**6. 忽略任务间的隐式依赖。** 表面上两个文件的修改是独立的，但它们 import 了同一个模块，修改后可能导致类型冲突。Swarm 的前提是任务真正独立——如果有隐式依赖，在验证阶段（由协调器负责）会发现，但此时修复成本更高。

**7. 不清理死工作者。** 工作者崩溃后没有被检测和清理，它领取的任务永远卡在 `claimed` 状态。心跳 + 僵尸检测不是可选的——它是 Swarm 系统的必要组件。
