# 群体/团队模板

设计群体和团队模式的实用模板 -- 多个 Agent 作为对等体协作，共享工作队列并通过消息传递进行协调。与协调者模式（一个领导者，多个工作者）不同，群体模式将权力和决策分配给各 Agent。

## 共享工作队列模式

工作队列是实现群体协调的核心数据结构。每个 Agent 提取任务，处理它们，并将结果和新任务推回。

**实现选项：**
- 基于文件的队列（最简单，适用于 2-5 个 Agent，单文件竞争问题）
- 每任务一文件的队列（更好的扩展性，通过原子重命名声明任务）

### 队列协议

1. Agent 读取队列，找到依赖已满足的未声明任务
2. Agent 原子性地声明任务
3. Agent 处理任务
4. Agent 标记任务完成或失败
5. Agent 可能创建在工作中发现的新任务
6. 重复直到没有任务剩余

## Agent 自主性 vs 领导者控制

- **群体（高自主性）：** Agent 自己从队列中拉取任务决定做什么。无领导者。通过队列和邮箱协调。每个 Agent 有相同的能力和权限。适用于并行搜索、独立文件处理、分布式测试。
- **团队（平衡）：** 领导者创建初始工作队列和高级任务。成员自主执行，可以创建子任务和互相通信。领导者监控进度并在卡住时介入。
- **受管团队（低自主性）：** 领导者分配特定任务给特定 Agent。Agent 完成后向领导者报告。领导者根据结果决定所有下一步。

## 权限同步

群体系统中，权限状态必须跨 Agent 一致。如果一个 Agent 获批编辑 `src/auth/` 中的文件，所有 Agent 都应继承该批准。使用共享权限状态文件，带时间戳的最后写入优先冲突解决。

## 后端选项

- **进程内（默认）：** 所有 Agent 共享一个进程。最简单。权限提示冒泡到领导者终端。适用于 2-5 个轻量级 Agent。
- **tmux 会话：** 每个 Agent 作为单独的 CLI 进程。可单独监控、重启或终止。会话在终端关闭后存活。适用于开发者可见的团队和长时间运行的任务。
- **iTerm2（macOS）：** 类似 tmux 但使用 iTerm2 原生分割面板 API。更丰富的 UI。终端关闭时会话丢失。适用于 macOS 开发和视觉监控。
- **Docker/容器：** 完全隔离。适用于不受信任的 Agent 和生产部署。

## 重连

Agent 可能因网络问题、崩溃或用户中断而断开。重连协议：检查现有状态文件 -> 如有当前任务且仍由自己声明则从检查点恢复 -> 如无状态则作为新成员加入 -> 注册心跳计时器（每 30 秒）。

## 崩溃恢复

当 Agent 在任务中途崩溃时：检测心跳超时 -> 标记为已崩溃 -> 将声明的任务释放回队列（增加重试计数）-> 广播通知团队。恢复策略：重试任务 / 重新分配给不同 Agent / 升级到领导者 / 跳过任务 / 中止团队。

## 邮箱系统

Agent 之间的点对点通信，基于文件的邮箱。每个 Agent 有一个收件箱文件。使用文件锁保证并发安全。定期轮询或每个任务完成后检查收件箱。
