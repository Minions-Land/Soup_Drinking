# 权限系统模板

## 问题场景

你的智能体能读文件、写文件、执行 Shell 命令、调用外部 API。它在一个真实的文件系统上运行，用户把自己的代码库交给它。你需要回答这些问题：

- 智能体想写入 `~/.ssh/config`——应该允许还是拒绝？
- 智能体想执行 `rm -rf node_modules`——这是合法的清理还是危险操作？
- 用户说过"允许编辑 src/ 下的文件"——这个规则在什么范围内有效？当前会话？永久？
- 后台子智能体需要权限批准但没有终端——怎么办？

权限系统是智能体从"有趣的 Demo"到"可以在生产环境跑"的关键门槛。Claude Code 的权限系统在 `src/Tool.ts` 的工具生命周期和 `src/permissions/` 中实现，下面是其设计的完整拆解。

## 分步实现指南

### 第 1 步：定义两阶段验证

这是权限系统的核心结构——分离"合不合法"和"允不允许"。

💡 **两阶段验证**（Two-Stage Validation）：第一阶段检查输入的逻辑有效性（文件存在吗？路径合法吗？），第二阶段检查授权策略（用户允许这个操作吗？规则怎么说？）。两件事分开做，因为它们的性质不同。

**阶段 1：validateInput(input, context)**

纯逻辑检查。不涉及用户交互，不涉及策略判断。

```typescript
// 示例：WriteFile 的输入验证
function validateInput(input) {
  if (!path.isAbsolute(input.filePath)) {
    return { result: false, message: "路径必须是绝对路径" }
  }
  if (!fs.existsSync(path.dirname(input.filePath))) {
    return { result: false, message: `父目录不存在: ${path.dirname(input.filePath)}` }
  }
  return { result: true }
}
```

验证失败时，错误消息直接返回给模型——模型根据消息自我修正（"路径必须是绝对路径"→ 模型下一轮会用绝对路径重试）。所以错误消息必须是**可操作的**（Actionable）：告诉模型怎么修正，不只是说"错了"。

**阶段 2：checkPermissions(input, context)**

纯授权检查。返回三种行为之一：

```typescript
type PermissionResult = 
  | { behavior: 'allow' }                    // 预先批准
  | { behavior: 'ask', message: string }      // 询问用户
  | { behavior: 'deny', message: string }     // 静默拒绝
```

为什么不是布尔值？因为 `true/false` 只有两种状态。智能体系统需要第三种状态——"我不确定，问用户"（`ask`）。大量操作既不应该被无条件允许，也不应该被无条件拒绝。

### 第 2 步：设计权限模式

💡 **权限模式**（Permission Mode）是一组预设的默认行为，决定工具调用在没有匹配规则时怎么处理。

| 模式 | 读文件 | 写文件 | Shell 命令 | 适用场景 |
|------|--------|--------|-----------|---------|
| `default` | allow | ask | ask | 交互式开发（最常用） |
| `plan` | allow | allow | ask | 半自主：信任文件编辑，Shell 谨慎 |
| `acceptEdits` | allow | allow | ask | 受信任的编辑会话 |
| `bypassPermissions` | allow | allow | allow | CI/CD 流水线，完全受信任 |
| `auto` | allow | allow | allow | SDK/无头模式，非交互式 |

**关键设计决策：** `default` 模式对写操作返回 `ask`，不是 `deny`。这意味着用户可以逐个批准写操作，系统不会阻止你做事。`deny` 用于硬安全边界（写入项目目录外、访问 `~/.ssh/`），不用于一般的写操作。

### 第 3 步：实现规则引擎

规则是基于模式匹配（Pattern Matching）的条目，预先回答权限问题。

```typescript
type PermissionRule = {
  tool: string           // 工具名称："WriteFile", "Bash"
  pattern: string        // glob 或正则匹配输入
  behavior: 'allow' | 'deny'
  scope: 'session' | 'project' | 'global'
}

// 示例规则
{ tool: "WriteFile", pattern: "src/**/*.ts", behavior: "allow", scope: "project" }
{ tool: "Bash", pattern: "npm test*", behavior: "allow", scope: "session" }
{ tool: "WriteFile", pattern: "/etc/**", behavior: "deny", scope: "global" }
```

**评估顺序（优先级递减）：**

1. **会话规则（Session Rules）**——用户在当前对话中说"允许编辑 src/ 下的文件"时创建。会话结束即失效。优先级最高，因为用户的实时意图最权威。
2. **项目规则（Project Rules）**——存储在 `.claude/settings.json` 中。项目级配置，团队共享。比如"这个项目中 `npm test` 始终允许"。
3. **全局规则（Global Rules）**——存储在 `~/.claude/settings.json` 中。用户级配置，跨项目生效。比如"永远不要写入 `/etc/`"。
4. **权限模式默认值**——根据当前模式的预设行为。
5. **回退到 `ask`**——没有任何规则匹配时，询问用户。这是 Fail-Closed 原则的体现。

**会话规则为什么是短暂的？** 防止意外的永久授权。用户在一次调试中说"允许执行 `rm -rf dist/`"，这个规则不应该永远生效——下次可能 `dist/` 里有不同的内容。短暂规则 = 安全。

### 第 4 步：破坏性操作检测

某些操作需要**更高的审查级别**，无论权限模式是什么。

```
自动检测的破坏性模式：
  Shell 命令: rm -rf, git push --force, git reset --hard
  SQL:        DROP TABLE, TRUNCATE, DELETE without WHERE
  文件:       覆盖系统文件 (/etc/, /usr/, ~/.ssh/)
```

处理方式：

1. **无论权限模式如何，始终 `ask`**。即使 `bypassPermissions` 模式，`rm -rf /` 也要问。
2. **显示不可逆性警告**。不是简单的"允许？"，而是"此操作将永久删除 dist/ 目录下的所有文件，不可恢复。确认？"
3. **后台智能体自动拒绝**。后台智能体没有终端来询问用户。破坏性操作在后台模式下无条件拒绝。
4. **审计日志**。记录所有破坏性操作的尝试，无论是否被允许。

### 第 5 步：沙箱集成

💡 **沙箱**（Sandbox）是权限系统的最外层硬边界。沙箱拒绝的操作无法被任何规则、用户批准或权限模式覆盖。

```
沙箱检查 → 权限规则检查 → 权限模式检查
```

沙箱在权限规则**之前**运行。这意味着：

- 即使全局规则说 `allow`，沙箱说 `deny` 就是 `deny`
- 即使用户在权限提示中点击"允许"，沙箱不可以被绕过
- 沙箱是"最后防线"，保护的是系统完整性而非用户意图

典型沙箱规则：不允许写入项目根目录之外的文件系统、不允许访问 `/etc/`, `/usr/`, `~/.ssh/` 等系统目录、不允许执行修改全局系统状态的命令。

### 第 6 步：多智能体权限冒泡

当子智能体（在后台运行）需要权限批准但自己没有终端时，有两种处理方式：

**`bubble` 模式**：将权限请求转发到父级智能体的终端。用户看到"子智能体 [researcher] 请求写入 src/api.ts，允许？"。用户可以：
- 允许（仅此次）
- 拒绝
- 允许所有（为这个子智能体的同类操作创建会话规则）

**无 `bubble` 模式**（默认）：自动拒绝。子智能体必须在没有写权限的情况下完成工作——或者报告失败，由父级决定下一步。

## 决策矩阵

| 场景 | 推荐模式 | 规则配置 |
|------|---------|---------|
| 交互式开发，用户全程在终端前 | `default` | 按需批准，频繁操作加会话规则 |
| 半自主编码，用户间歇性检查 | `plan` | 项目级规则 allow 常见编辑路径 |
| CI/CD 流水线中的自动化任务 | `bypassPermissions` | 沙箱限制项目目录范围 |
| SDK 集成，无终端 | `auto` | 沙箱 + 全局规则做硬限制 |
| 多智能体，子级需要审批 | 子级 `bubble` | 父级权限提示中继 |
| 多智能体，子级完全只读 | 子级 `default` + 工具白名单 | 只给只读工具 |

## 常见陷阱

**1. 默认 allow 而非 ask。** 最常见也最危险的错误。意味着任何忘记配置权限的新工具自动获得完全操作权限。Claude Code 的 `buildTool()` 默认 `ask`——忘记配置只会多一次用户提示，不会造成安全漏洞。

**2. 验证和授权混在一起。** 用户看到"权限被拒绝"但实际上文件路径就是错的。分离后用户看到的是"文件不存在: /tmp/xxx"——更准确，更可操作。

**3. 会话规则永久化。** 用户说"允许 `rm dist/`"——如果这个规则变成永久规则，下次 `dist/` 里可能有不该删的文件。会话规则应该随会话结束而失效。

**4. 沙箱可被覆盖。** 如果用户批准可以覆盖沙箱，那沙箱就不是真正的安全边界。沙箱必须是不可覆盖的（Non-Overridable）。

**5. 忽略命令的变体形式。** 检测了 `rm -rf` 但没检测 `find . -delete`、`perl -e 'unlink ...'`、`python -c 'import shutil; shutil.rmtree(...)'`。破坏性操作的检测需要覆盖常见的变体。

**6. 子智能体权限和父级完全相同。** 子智能体不应该有比父级更多的权限。但也不应该简单继承——后台子智能体需要自动拒绝写操作（除非配置了 bubble），即使父级允许写操作。
