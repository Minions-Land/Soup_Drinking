# 工具设计模板

## 问题场景

你在构建一个智能体系统，模型需要通过"工具"（Tool）与外部世界交互——读文件、查数据库、调 API。你面临的工程问题是：

- 工具太多（20+），每个独立实现权限检查和错误处理，行为不一致
- 模型经常选错工具或传错参数——它根据 Schema 和描述做决策，但你的 Schema 和描述质量参差不齐
- 工具返回的结果太大，撑爆上下文窗口；或者太小，模型无法从中推理
- 新开发者不知道实现一个工具需要关心哪些方面

本模板基于 Claude Code 的 `src/Tool.ts` 中 `buildTool()` 模式，帮你系统化地设计每一个工具。

## 分步实现指南

### 第 1 步：定义能力边界

在写任何代码之前，先用两句话回答：
- 这个工具**做什么**？（一句话）
- 这个工具**不做什么**？（一句话）

例如 `ReadFileTool`：做——读取指定路径的文件内容并返回文本。不做——不写文件、不列目录、不跟随符号链接到项目外部。

这两句话决定了后续所有设计决策。如果你发现"做什么"需要三句话才能说清楚，那你需要拆分工具。Claude Code 的做法：`ReadFile` 和 `WriteFile` 分开，`ListDirectory` 独立。不是 `FileOperation(action: 'read' | 'write' | 'list')`。

**为什么拆分？** 因为 `isReadOnly`、`checkPermissions`、`prompt()` 对于读和写是完全不同的。一个"超级工具"的权限逻辑会变成 `if (action === 'read') { ... } else if (action === 'write') { ... }` 的条件地狱。

### 第 2 步：设计 inputSchema

```typescript
inputSchema: z.object({
  filePath: z.string()
    .describe("要读取的文件的绝对路径"),
  offset: z.number().int().min(0).optional()
    .describe("从第几行开始读取，0-based。不提供则从头开始"),
  limit: z.number().int().min(1).max(10000).optional()
    .describe("最多读取多少行。不提供则读取全部"),
})
```

关键原则：

**Schema 越严格，模型幻觉参数越少。** 如果一个参数只有 3 种合法值，用 `z.enum(['a', 'b', 'c'])`，不要用 `z.string()`。模型看到 `string` 会自由发挥；看到 `enum` 会从中选择。

**每个字段必须有 `describe()`。** 模型根据 description 决定传什么值。没有 description 的字段就像没有路标的十字路口——模型只能猜。

**必填字段（Required Fields）用 `z.object({})` 的默认行为。** 可选字段用 `.optional()`。不要把所有字段都设为可选——模型会跳过它认为"不重要"的必填参数。

### 第 3 步：实现 call()

```typescript
async call(input, context, canUseTool, parentMessage, onProgress) {
  // 1. 执行核心操作
  const content = await fs.readFile(input.filePath, 'utf-8')
  
  // 2. 结构化返回（不是原始字符串）
  return {
    data: {
      content: lines.join('\n'),
      totalLines: allLines.length,
      returnedLines: lines.length,
      truncated: lines.length < allLines.length,
    }
  }
}
```

原则：

**返回结构化数据，不是原始字符串。** 模型从 `{ totalLines: 500, returnedLines: 100, truncated: true }` 能推理出"文件很长，我只看到了一部分"。从一段纯文本它推理不出这些。

**大声失败（Fail Loudly）。** 静默返回空结果的工具比抛出错误的工具更糟糕。模型无法调试沉默——它不知道"没有结果"是因为文件为空还是文件不存在。抛出带可操作信息的错误：`throw new ToolError("文件不存在: " + path)`。

**结果大小有上限。** Claude Code 的 `buildTool()` 要求填写 `maxResultSizeChars`（默认 100,000 字符）。超过时自动将结果持久化到文件，返回文件路径和摘要。不设上限的工具迟早会返回一个 500KB 的日志文件内容，撑爆上下文。

### 第 4 步：编写 prompt()

```typescript
prompt(): string {
  return `Use ReadFile to read the contents of a file at a given path.
When to use: When you need to see the content of a file.
When NOT to use: To check if a file exists (use Bash with test -f).
Output: Returns file content with line numbers. Large files are truncated.
Tip: Use offset and limit for large files instead of reading everything.`
}
```

**prompt() 是工具设计中最重要的部分。** 模型根据 `prompt()` 而非 `call()` 的实现来决定是否使用工具。你的实现可能完美，但如果 `prompt()` 描述不清楚，模型就不会用或者误用。

实操建议：写完 prompt() 后，用 10 个不同的自然语言请求测试模型是否正确选择此工具。准确率低于 95% 就继续迭代 prompt()。大多数工具质量问题出在这里。

控制在 200 词以内。系统提示词中有 20+ 个工具的 prompt()，每个 200 词已经是 4000 词了。精简。

### 第 5 步：实现 render()

```typescript
render(result): string {
  // 为模型消费格式化——不是为人类
  return `File: ${result.filePath}
Lines ${result.offset}-${result.offset + result.returnedLines} of ${result.totalLines}
${result.truncated ? '(truncated)' : '(complete)'}
---
${result.content}`
}
```

模型将对 `render()` 输出进行推理。据此目的来结构化：

- **表格优于段落**：20 条搜索结果用表格排列，模型能快速比较。散文形式模型要逐句解析。
- **计数优于列表**：如果结果有 500 条匹配，返回"Found 500 matches"加前 20 条，而不是 500 条的完整列表。
- **去除噪音**：日志文件中的时间戳、进程 ID 等通常对模型的推理无帮助。在渲染时去掉。

## 决策矩阵：可选方法

| 方法 | 何时实现 | 何时跳过 | Claude Code 的做法 |
|---|---|---|---|
| `validateInput()` | 输入可能不合法（文件路径、正则、URL） | 输入是 Schema 强约束的枚举值 | 所有文件操作工具都实现 |
| `checkPermissions()` | 工具有副作用（写、删、发送） | 只读且不涉及敏感数据 | `buildTool()` 默认行为：`ask` |
| `isReadOnly()` | 工具纯只读 | 工具有任何写入可能 | `ReadFile` → true, `WriteFile` → false |
| `isConcurrencySafe()` | 工具可以和自身并行运行 | 工具修改共享状态 | `GrepTool` → true, `WriteFile` → false |
| `isDestructive()` | 操作不可逆（`rm -rf`, `DROP TABLE`） | 可以撤销或只读 | `BashTool` 检测危险命令模式 |
| `maxResultSizeChars` | 结果大小不可预测 | 结果是固定格式的小数据 | 所有工具都设置（默认 100K） |
| `progressCallback()` | 工具运行 > 2 秒 | 亚秒级完成 | `BashTool`, `GrepTool` 都实现 |
| `shouldDefer` | 使用频率 < 10% 的对话 | 几乎每次对话都用 | `NotebookEditTool` → true |
| `searchHint` | 工具被延迟加载 | 工具始终在系统提示中 | 多关键词覆盖用户描述方式 |
| `timeout()` | 工具可能挂起（网络、Shell） | 纯内存计算 | `BashTool` 默认 120 秒 |
| `rollback()` | 需要支持"撤销"操作 | 幂等或只读 | 文件编辑保留备份 |

## 检查清单

### 设计阶段（写代码之前）

- [ ] 能用两句话说清"做什么"和"不做什么"吗？
- [ ] 需要拆分成多个工具吗？（一个工具做两件事 = 需要拆分）
- [ ] inputSchema 中每个字段都有 `describe()` 吗？
- [ ] 值有限的字段用了 `enum` 吗？
- [ ] 确认了 `maxResultSizeChars` 的合理上限吗？

### 实现阶段

- [ ] `call()` 返回结构化数据（不是纯字符串）？
- [ ] 错误情况抛出 `ToolError` 并带有可操作信息？
- [ ] `prompt()` 控制在 200 词以内？
- [ ] `prompt()` 包含"何时用"和"何时不用"？
- [ ] `render()` 的输出适合模型推理（表格/计数 > 散文/完整列表）？
- [ ] 设置了 `isReadOnly`、`isConcurrencySafe`、`isDestructive` 标志？

### 测试阶段

- [ ] 用 10 个不同的自然语言请求测试工具选择准确率 > 95%？
- [ ] 模型传的参数正确吗？有没有幻觉参数？
- [ ] 错误情况的消息让模型能自我修正吗？
- [ ] 大结果的截断和摘要行为符合预期吗？
- [ ] 并发调用（如果 `isConcurrencySafe: true`）不会损坏状态？

### 上线后

- [ ] 监控工具被调用的频率——长期不用的工具应该标记 `shouldDefer`
- [ ] 监控 `prompt()` 的迭代——模型行为变化时可能需要更新
- [ ] 收集工具调用失败的日志，分析是输入问题还是执行问题

## 常见陷阱

**1. prompt() 写给人看而不是给模型看。** 你觉得"读取文件内容"很清楚，但模型可能把"检查文件是否存在"也理解为"读取文件"。加明确的"When NOT to use"。

**2. 默认 Fail-Open。** `checkPermissions` 默认返回 `allow`。这意味着任何忘记配置权限的工具自动获得写权限。Claude Code 的 `buildTool()` 默认返回 `ask`——忘记配置的工具会询问用户，不会静默允许。

**3. 结果不截断。** 工具返回了整个数据库查询结果（10,000 行），模型尝试推理所有数据，上下文窗口爆了。设置 `maxResultSizeChars`。

**4. 工具名称太泛。** `Process()`, `Handle()`, `Execute()` 这些名称告诉模型什么？什么都没说。用动词+名词：`QueryDatabase`, `ReadFile`, `SendEmail`。

**5. `call()` 不幂等。** 模型可能重试工具调用（网络超时、上下文被截断后重新请求）。如果第一次调用创建了一个资源，第二次调用不应该再创建一个。默认幂等（Idempotent）是安全的选择。
