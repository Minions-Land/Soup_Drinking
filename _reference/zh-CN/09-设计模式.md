# 可复用设计模式

> 从 Claude Code 中提取的 12 个关键架构模式，可直接复制/改编到你自己的 Agent 系统中。

---

## 模式 1：带安全默认值的工具接口

**问题**：许多工具共享样板代码（权限、验证、渲染）。
**方案**：`buildTool()` 提供安全默认值；工具只覆盖需要的部分。

```typescript
const MyTool = buildTool({
  name: 'MyTool',
  inputSchema: z.object({ ... }),
  maxResultSizeChars: 100_000,
  call: async (args, ctx) => ({ data: result }),
  prompt: async () => "给模型的指令...",
  // 其他一切获得安全默认值
})
```

**适用场景**：任何需要一致性但不想写样板代码的工具/插件系统。

## 模式 2：上下文 Fork 用于 Prompt 缓存共享

**问题**：子智能体需要父级上下文，但重新分词很昂贵。
**方案**：Fork 子级，获得父级系统提示和对话前缀的字节精确副本。

```
父级上下文：
  [系统提示词][消息1][消息2][消息3]
                ↓ Fork 点
子级上下文：
  [系统提示词][消息1][消息2][新任务]
  ↑ 共享 Prompt 缓存 — 零重新分词成本
```

**关键**：复制 `renderedSystemPrompt`。克隆 `contentReplacementState`。创建独立 `AbortController`。

**适用场景**：子智能体需要父级研究上下文的多智能体系统。

## 模式 3：扇出研究 → 综合 → 实现 → 验证

**问题**：复杂任务在实现前需要多个视角。
**方案**：四阶段协调器工作流。

```
阶段 1：研究（并行）
  智能体 A → 搜索后端代码
  智能体 B → 搜索前端代码
  智能体 C → 阅读文档

阶段 2：综合（协调者）
  协调者 → 分析所有结果 → 创建详细实现规范

阶段 3：实现（定向）
  智能体 D → 执行综合后的规范

阶段 4：验证（全新智能体）
  智能体 E（无先前上下文）→ 审查变更 → 运行测试
```

**关键**：验证智能体必须是**全新的**（无实现上下文），防止确认偏差。

## 模式 4：两阶段权限验证

**问题**：需要同时进行逻辑验证和授权检查。
**方案**：分离 `validateInput`（逻辑）和 `checkPermissions`（授权）。

```
阶段 1：validateInput(input, context)
  → "这个文件存在吗？"
  → "这是有效的正则吗？"

阶段 2：checkPermissions(input, context)
  → "用户允许文件写入吗？"
  → "这匹配 always-allow 规则吗？"
```

**适用场景**：任何工具需要有效性检查和授权的系统。

---

## 模式 5：AsyncLocalStorage 实现智能体隔离

**问题**：同一 Node.js 进程中的多个智能体需要隔离的上下文。
**方案**：使用 `AsyncLocalStorage` 维护每个智能体的状态，无需手动传递。

```typescript
const agentContext = new AsyncLocalStorage<AgentState>()

agentContext.run(agentState, async () => {
  // 所有嵌套调用都能看到这个智能体的状态
  const state = agentContext.getStore()
  await doWork()  // 无需显式传递状态
})
```

**适用场景**：进程内多智能体系统、Node.js 中的请求作用域状态。

## 模式 6：渐进式结果交付

**问题**：长时间运行的工具阻塞 UI。
**方案**：带类型化进度事件的 Progress 回调系统。

```typescript
async call(args, ctx, canUse, msg, onProgress) {
  onProgress?.({ toolUseID: id, data: { type: 'bash', status: 'running' } })
  // ... 执行工作 ...
  onProgress?.({ toolUseID: id, data: { type: 'bash', output: '50% 完成' } })
  return { data: result }
}
```

**适用场景**：任何耗时超过几秒的工具。

---

## 模式 7：延迟工具加载

**问题**：系统提示中的工具太多浪费 Token。
**方案**：标记不常用工具为 `shouldDefer: true`，模型通过 `ToolSearchTool` 发现。

```
回合 1：模型有 15 个核心工具 + ToolSearchTool
回合 2：模型调用 ToolSearchTool("jupyter notebook")
  → 找到 NotebookEditTool（延迟加载的）
  → 工具 Schema 加载到下一回合
回合 3：模型可以使用 NotebookEditTool
```

**适用场景**：20+ 工具的系统，Prompt 空间受限。

---

## 模式 8：基于 XML 的异步通知

**问题**：后台智能体需要通知主对话。
**方案**：将 XML 标签通知注入消息流。

```xml
<task-notification task_id="abc123" status="completed">
  智能体任务"研究 API 端点"已完成。
</task-notification>
```

**适用场景**：结果在不可预测时间到达的异步多智能体通信。

---

## 模式 9：共享工作队列（Swarm）

**问题**：多个智能体需要协调，但不想有中心瓶颈。
**方案**：共享任务列表，智能体自主领取工作。

```
领导者：发布任务到 TaskList
工作者 1：轮询 → 领取"任务 A" → 执行 → 完成
工作者 2：轮询 → 领取"任务 B" → 执行 → 完成
工作者 3：轮询 → 领取"任务 C" → 执行 → 完成
```

无需领导者微管理。工作者自主执行。

**适用场景**：可并行化工作负载（跨文件测试、搜索多个仓库）。

---

## 模式 10：多来源命令注册中心

**问题**：命令来自不同来源（内置、插件、技能）。
**方案**：中央注册表发现和过滤命令。

```
Registry.getCommands() →
  内置（硬编码） +
  技能（Prompt 文件） +
  插件（动态） +
  工作流（脚本化）
  → 按认证级别、模式、安全性过滤
  → 返回统一命令列表
```

**适用场景**：具有多个扩展点的可扩展 CLI/聊天系统。

---

## 模式 11：无 GC 的打包缓冲区渲染

**问题**：终端 UI 渲染导致 GC 暂停。
**方案**：使用类型化数组（Int32Array）替代对象存储 Cell 缓冲区。

```
每个 Cell = 2 × Int32：
  字 0：驻留字符 ID
  字 1：打包位字段（前景色、背景色、属性、宽度）
```

字符、样式、URI 使用驻留池。每帧零对象分配。

**适用场景**：GC 暂停明显的高频终端渲染。

---

## 模式 12：协调者-工作者 + 草稿板

**问题**：工作者需要共享发现，但不想通过对话传递。
**方案**：所有智能体可访问的共享文件系统目录（草稿板）。

```
协调者 → 创建草稿板目录
工作者 A → 将发现写入 scratchpad/research-a.md
工作者 B → 读取 scratchpad/research-a.md，写入 scratchpad/research-b.md
协调者 → 读取所有草稿板文件 → 综合
```

路径通过协调者上下文注入。持久跨智能体生命周期。

**适用场景**：有大型中间产物的多智能体系统。
