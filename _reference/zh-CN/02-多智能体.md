## 1. 架构概览

Claude Code 实现了一个分层的多智能体系统，支持三种执行模式：

```
┌─────────────────────────────────────────┐
│         协调者（Coordinator/Leader）      │
│  coordinatorMode.ts — 编排所有工作        │
│  阶段：研究 → 综合 → 实现 → 验证          │
├──────────┬──────────┬───────────────────┤
│ 工作者 1  │ 工作者 2  │    工作者 N        │
│ (Agent)  │ (Agent)  │    (Agent)        │
│ AgentTool│ AgentTool│    AgentTool      │
│ runAgent │ runAgent │    runAgent       │
└──────────┴──────────┴───────────────────┘
```

### 三种执行模式

| 模式 | 描述 | 关键文件 |
|------|------|---------|
| **子智能体** | 独立上下文的后台智能体 | `runAgent.ts` |
| **Fork** | 子级继承父级的完整上下文（Prompt 缓存） | `forkSubagent.ts` |
| **进程内队友** | 通过 AsyncLocalStorage 共享进程的 Swarm 成员 | `inProcessRunner.ts` |

## 2. 智能体生成

### 2.1 AgentTool 调用流程

```
模型调用 AgentTool →
  1. loadAgentsDir() → 查找匹配的智能体定义
  2. createSubagentContext() → 构建子上下文
  3. 选择模式：
     a. fork（isolation: "worktree"）→ forkSubagent()
     b. 后台运行 → runAgent() 作为 LocalAgentTask
     c. 前台运行 → runAgent() 内联
  4. 在 AppState 中注册任务
  5. 通过 onProgress 回调流式传递进度
  6. 返回最终结果给父级
```

### 2.2 内置智能体

| 智能体 | 文件 | 用途 |
|-------|------|------|
| `general-purpose` | `generalPurposeAgent.ts` | 通用研究、代码搜索、多步骤任务 |
| `Explore` | `exploreAgent.ts` | 快速代码库探索 |
| `Plan` | `planAgent.ts` | 架构规划 |
| `verification` | `verificationAgent.ts` | 检查/验证实现 |
| `claude-code-guide` | `claudeCodeGuideAgent.ts` | 回答 Claude Code 相关问题 |
| `statusline-setup` | `statuslineSetup.ts` | 配置状态栏 |

自定义智能体可从 `.claude/agents/` 目录加载。

## 3. 上下文 Fork（Prompt 缓存优化）

核心创新：**forkSubagent** 创建的子智能体继承父级系统提示和对话前缀的**字节精确副本**。

```
父智能体上下文：
  [系统提示词] [用户消息1] [助手回复1] [用户消息2] [助手回复2]
                ↓ Fork 点
子智能体（Fork）：
  [系统提示词] [用户消息1] [助手回复1] [新任务提示]
  ↑ 共享 Prompt 缓存 — 零重新分词成本！
```

**为什么重要**：节省大量 token 成本。子智能体从父级研究的完整上下文开始，无需重新处理。

关键实现（`forkSubagent.ts`）：
- 从父上下文复制 `renderedSystemPrompt`
- 克隆 `contentReplacementState` 用于工具结果预算共享
- 创建独立的 `AbortController` 和 `FileStateCache`

## 4. 智能体间通信

### 4.1 SendMessageTool（消息工具）

智能体间通信的主要机制：

```
智能体 A → SendMessageTool.call({ to: "agent-id", message: "..." })
  → 找到目标智能体的任务
  → 将消息注入目标的"收件箱"
  → 目标智能体在下次轮询时拾取
```

### 4.2 任务通知（XML 块）

异步后台结果通过 XML 标签传递：

```xml
<task-notification task_id="abc123" status="completed">
  智能体任务"研究 API"已完成。
</task-notification>
```

### 4.3 权限冒泡

当子智能体需要用户权限时：
1. 子智能体的 `checkPermissions` 返回 `{ behavior: 'ask' }`
2. 请求冒泡到领导者的 UI
3. 用户批准/拒绝
4. 响应路由回子智能体

## 5. 协调者模式（四阶段工作流）

```
阶段 1：研究（RESEARCH）
  → 并行生成多个研究智能体
  → 每个探索不同的代码区域
  → 收集结果

阶段 2：综合（SYNTHESIS）
  → 协调者分析所有研究结果
  → 创建详细的实现规范
  → 识别文件、依赖、风险

阶段 3：实现（IMPLEMENTATION）
  → 生成实现智能体
  → 执行综合后的规范

阶段 4：验证（VERIFICATION）
  → 生成全新的验证智能体
  → 独立代码审查（防止"橡皮图章"）
  → 运行测试，检查回归
```

**关键**：验证者是一个**全新的智能体**，未见过实现过程，防止确认偏差。

## 8. 可复用的核心模式

### 模式 1：扇出研究
```
领导者 → 并行生成 N 个智能体 → 每个搜索不同区域 → 收集结果 → 综合
```

### 模式 2：Fork-执行-返回
```
父级 → Fork 子级（继承上下文）→ 子级执行任务 → 返回结果 → 父级继续
```

### 模式 3：共享工作队列
```
领导者 → 发布任务到 TaskList → 工作者轮询并领取任务 → 独立执行 → 结果聚合
```

### 模式 4：双层验证
```
智能体 A → 实现变更
智能体 B（全新，无上下文）→ 验证变更 → 防止确认偏差
```

### 模式 5：AsyncLocalStorage 隔离
```
多个智能体共享 Node.js 进程
每个通过 AsyncLocalStorage 拥有隔离的上下文
无需手动通过函数签名传递上下文
```
