# 权限系统模板

设计 Agent 架构中权限系统的实用模板。权限是 Agent 能做什么与被允许做什么之间的安全边界。

## 两阶段验证

每次工具调用在执行前经过两个阶段：

**阶段 1：输入验证（validateInput）** -- 在权限检查之前运行。捕获格式错误的输入。返回清晰的错误消息，不提示用户。

**阶段 2：权限检查（checkPermissions）** -- 在验证成功后运行。返回三种行为之一：
- `allow`：操作已预先批准，立即执行。用于只读操作或匹配用户配置规则的操作。
- `ask`：用户必须明确批准。用于写操作、破坏性操作或已知安全模式之外的操作。
- `deny`：操作被静默阻止。用于违反硬安全边界的操作（如写入项目目录外）。

## 权限模式

| 模式 | 行为 | 使用场景 |
|------|------|----------|
| `default` | 写入时询问，允许读取 | 交互式开发 |
| `plan` | 允许读写，shell 命令时询问 | 半自主工作 |
| `acceptEdits` | 允许所有文件编辑，shell 时询问 | 受信任的编辑会话 |
| `bypassPermissions` | 允许一切 | CI/CD，完全受信任的自动化 |
| `auto` | 允许一切（非交互式） | SDK/无头模式 |

### 多 Agent 系统中的模式继承

规则：宽松的父级模式（bypassPermissions、acceptEdits、auto）始终覆盖子级。Agent 的模式仅在父级处于 `default` 模式时生效。

## 权限规则

规则是基于模式的条目，预先回答权限问题。评估顺序：
1. 检查会话规则（最具体，最高优先级）
2. 检查项目规则（.claude/settings.json）
3. 检查全局规则（~/.claude/settings.json）
4. 检查权限模式默认值
5. 回退到 `ask`

## 破坏性操作检测

某些操作需要更高的审查级别。自动检测并标记：`rm -rf`、`git push --force`、`git reset --hard`、`DROP TABLE`、`TRUNCATE` 等。处理方式：
1. 无论权限模式如何，始终要求用户明确批准。
2. 显示描述不可逆性质的清晰警告。
3. 在后台 Agent（无终端）中，自动拒绝破坏性操作。
4. 记录所有破坏性操作尝试以便审计。

## 沙箱集成

沙箱提供权限规则无法覆盖的硬边界。沙箱检查在权限规则之前运行。被沙箱拒绝的操作无法被任何规则、用户批准或模式覆盖。沙箱是最后的防线 -- 没有什么能绕过它。

## 多 Agent 系统中的权限冒泡

当子 Agent 需要权限但无法访问终端时：
- `bubble` 模式：将权限请求转发到父 Agent 的终端。用户可以允许、拒绝或为该 Agent 允许所有同类操作。
- 无 `bubble` 模式：自动拒绝（无终端访问）。

## 关键原则

- **安全闭合。** 权限系统遇到意外状态时，拒绝操作。永远不要默认允许。
- **沙箱是绝对的。** 没有任何权限规则、用户批准或模式覆盖可以绕过沙箱。
- **审计一切。** 每个权限决策都应记录时间戳、工具、输入和决策。
- **会话规则是短暂的。** 它们随会话消亡。项目和全局规则持久化。这防止了意外的永久授权。

## 实现清单

1. 定义权限模式及其默认行为。
2. 实现两阶段验证：validateInput 然后 checkPermissions。
3. 构建具有会话/项目/全局范围和评估顺序的规则引擎。
4. 添加常见危险模式的破坏性操作检测。
5. 将沙箱边界作为权限规则之前的硬层集成。
6. 为多 Agent 场景实现权限冒泡。
7. 添加所有权限决策的审计日志。
8. 使用试图绕过权限的对抗性提示进行测试。
