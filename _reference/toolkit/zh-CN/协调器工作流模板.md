# 协调者工作流模板

构建协调者工作流的实用模板 -- 一个主导 Agent 通过结构化多阶段流程编排多个子 Agent 的模式。这是需要研究、综合、实现和验证的复杂任务最常见的模式。

## 四阶段模式

### 阶段 1：研究（扇出）

协调者并行生成多个 Agent 来调查问题的不同方面。每个研究 Agent 获得一个具体的、聚焦的问题。

**关键原则：**
- 每个 Agent 获得一个具体问题，而非模糊的指令。
- Agent 应返回结构化发现，而非意见或建议。
- 使用后台模式 -- 研究 Agent 是独立的。
- 在一个协调者轮次中生成所有研究 Agent（一条消息，多个工具调用）。
- 永远不要求研究 Agent "修复"任何东西。仅研究。

### 阶段 2：综合

所有研究通知到达后，协调者阅读结果并构建具体计划。这是协调者推理最关键的地方。

**关键原则：**
- 协调者自己做综合 -- 永远不要把理解委托给子 Agent。
- 编写具体的、逐文件的计划。模糊的计划产生模糊的实现。
- 按名称引用研究 Agent 的具体发现。
- 识别实现步骤之间的依赖关系。
- 将计划写入共享便签本文件以便追溯。

### 阶段 3：实现

协调者执行计划，直接或通过委托给专门的 Agent。

**关键原则：**
- 小改动（不到 5 个文件），协调者直接实现。为琐碎的编辑生成 Agent 浪费 token 并增加延迟。
- 大改动，委托但要具体。包含确切的文件路径、确切的变更、确切的要遵循的模式。
- 使用前台模式避免文件写入冲突。
- 当涉及重叠文件时，使用顺序而非并行的实现 Agent。

### 阶段 4：验证

一个全新的 Agent -- 未参与研究或实现的 -- 验证工作。

**为什么要全新 Agent？** 执行实现的 Agent 倾向于相信自己的工作是正确的。它会忽略问题，因为它"知道"自己的意图。具有全新上下文的 Agent 会真正运行测试、搜索问题并诚实报告。这是协调者模式中最重要的洞察。

## 便签本：阶段间共享状态

使用临时便签本目录进行阶段间通信。文件优于消息传递的原因：
- 文件跨 Agent 生命周期持久化。消息在 Agent 终止时丢失。
- 文件可以被任何 Agent 在任何时间读取。消息仅限于父子关系。
- 文件提供审计轨迹。你可以审查每个 Agent 发现了什么和决定了什么。
- 文件避免上下文窗口膨胀。

## 常见变体

- **仅研究协调者：** 研究 -> 综合 -> 报告（跳过实现和验证）
- **重实现协调者：** 最小研究 -> 实现 -> 验证（计划已知时）
- **迭代协调者：** 循环直到验证通过（限制 2-3 次最大迭代，防止无限循环和 token 消耗失控）

## 反模式

- 把综合委托给子 Agent（"根据你的发现修复它"产生浅层工作）
- 在同一文件上并行实现（写入冲突，丢失变更）
- 验证者继承上下文（确认偏差，橡皮图章）
- 模糊的研究提示（"调查认证系统"产生不聚焦的结果）
- 跳过验证（Bug 未被检测就上线生产环境）
- 过多阶段（对简单任务来说开销超过收益）
