# 任务系统模板

设计 Agent 架构中任务系统的实用模板。任务是具有生命周期的任何工作单元 -- 它开始、运行、可能产生中间结果，最终完成或失败。任务系统管理这个生命周期，防止僵尸任务，并在并发任务之间提供上下文隔离。

## 任务生命周期状态机

每个任务遵循严格的状态机。转换是单向的；除了重试（failed -> pending），任务不能后退。

### 转换规则

- created -> pending：任务注册到系统（分配 ID，持久化状态，发出事件）
- pending -> running：工作者接手任务（记录开始时间，启动心跳）
- running -> completed：工作成功完成（记录结果和完成时间，通知父级）
- running -> failed：工作抛出错误（记录错误，触发重试或升级）
- running -> cancelled：用户或系统取消（发送 AbortController 信号，清理资源）
- failed -> pending：重试策略触发（增加重试计数，重新入队）

## 全局状态注册

所有任务必须在全局任务注册表中注册。这是任务追踪、UI 显示、清理和僵尸检测的唯一真相来源。注册表必须是进程级单例 -- 多个注册表会导致孤立任务和不一致状态。

核心功能：注册（含重注册支持）、状态转换（带验证守卫）、查询活跃任务、按 Agent 查询、修剪已通知的终端任务。

## 后台执行

后台任务独立于主对话流运行。立即返回任务 ID，任务在后台运行。完成时通过通知传递结果。

**磁盘输出流：** 任务输出流式写入磁盘以避免内存压力。父级使用偏移量追踪增量读取输出。

## 僵尸保护

僵尸任务是卡在 `running` 状态但没有活跃工作者的任务。原因包括工作者崩溃、进程被终止或任务完成逻辑中的 bug。

**何时发生：** 父 Agent 完成或被终止，但其生成的后台 shell 任务仍在运行。

**心跳检测：** 长时间运行的任务发出心跳（建议 30 秒间隔）。如果心跳缺失超过超时阈值（建议 90 秒），任务被视为已死亡。

**僵尸收割者：** 后台进程定期检查（建议 60 秒间隔）。根据重试次数决定重试或永久标记为失败（建议最多 3 次重试）。

**Agent 退出清理：** 每个 Agent 必须在 `finally` 块中终止其孤立任务。

## AsyncLocalStorage 上下文隔离

在 Node.js 中，`AsyncLocalStorage` 提供每任务上下文，无需通过每个函数调用传递上下文对象。这对于隔离同一进程中并发运行的任务之间的状态至关重要。

**优势：**
- **线程安全：** 每个异步任务获得自己的隔离上下文。
- **无参数传递：** 上下文在调用栈中的任何位置可用。
- **自动清理：** 异步范围结束时上下文被垃圾回收。
- **嵌套上下文：** 内部 `run()` 调用创建遮蔽外部上下文的嵌套范围。
- **零开销：** AsyncLocalStorage 是 Node.js 内置功能，性能开销可忽略不计。

## 实现清单

1. 定义具有显式转换规则和守卫函数的任务状态机。
2. 构建具有注册、转换、查询和修剪功能的全局 TaskRegistry 单例。
3. 实现带有 AbortController 和磁盘输出流的后台任务执行。
4. 为长时间运行的任务添加心跳发射（建议 30 秒间隔，90 秒超时）。
5. 构建检测卡住任务并应用重试/失败策略的僵尸收割者。
6. 在每个 Agent 的 `finally` 块中添加清理以终止孤立任务。
7. 设置 AsyncLocalStorage 用于每任务上下文隔离。
8. 构建任务通知系统（注入父级对话的 XML 消息）。
9. 添加监控：活跃任务数、平均持续时间、失败率、僵尸检测次数。
10. 使用并发任务、模拟崩溃、中止和重试场景进行测试。
