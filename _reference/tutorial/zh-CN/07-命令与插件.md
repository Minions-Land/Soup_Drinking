# 第七章：命令与插件 -- 扩展智能体

> 上一章：[第六章：终端 UI](06-terminal-ui.md) | 下一章：[第八章：远程连接](08-remote-bridge.md)

想象你有一个能回答问题、运行工具的个人助手。这已经很强大了。但如果你还能教它新技能呢？比如一个了解你 Git 工作流的 `/commit` 命令，或者一个遵循团队代码审查清单的 `/review-pr` 命令？更进一步，如果任何人都能把这些技能打包成插件，分享给整个社区呢？

本章解释 Claude Code 的**命令和插件系统**如何让智能体无限扩展，而无需修改核心代码。

## 什么是斜杠命令？

当你在 Claude Code 中输入 `/commit` 时，你正在调用一个**斜杠命令**。斜杠命令是触发特定行为的快捷方式。有些命令运行本地 JavaScript 逻辑，有些展开为 AI 模型处理的提示词，还有些直接在终端中渲染交互式的 React UI 组件。

管理这一切的系统是**命令注册中心**，定义在 `src/commands.ts` 中。

## 命令注册中心：一个入口，多个来源

注册中心最有趣的设计决策是命令并非全部来自同一个地方。相反，它在运行时从四个不同来源**聚合**命令：

| 来源 | 描述 | 示例 |
|------|------|------|
| 内置 | 硬编码在 CLI 源码中 | `/exit`、`/help`、`/clear` |
| 技能 | 磁盘上的 Prompt 文件，AI 模型可调用 | `/commit`、`/review-pr` |
| 插件 | 动态加载的扩展 | 来自 `src/plugins/` 的第三方工具 |
| 工作流 | 可脚本化的工具链 | 用户自定义的自动化 |

当你按 Tab 键查看可用命令，或者智能体考虑调用哪个技能时，所有四个来源都被合并成一个统一列表。这就是**多来源聚合模式** -- 一个从多个提供者拉取并呈现统一接口的注册中心。

可以把它想象成餐厅菜单。厨房（内置）提供标准菜品，客座厨师（插件）添加特色菜，季节性食谱卡（技能）添加限时菜品，服务员（注册中心）将一份完整的菜单递给食客。

## 三种命令类型

每个命令都有一个 `type` 字段，决定了它的执行方式：

```typescript
type: 'prompt' | 'local' | 'local-jsx'
```

**`prompt`** 命令最有意思。当你输入 `/commit` 时，并没有 JavaScript 函数直接创建 Git 提交。相反，命令构建一段精心设计的提示词并注入对话。AI 模型读取这个提示词，然后使用工具（如 `BashTool` 运行 `git commit`）来产生结果。命令定义*意图*；模型解决*执行*。

**`local`** 命令完全绕过 LLM。`/clear` 不需要 AI 智能 -- 它只是重置内存中的对话状态并返回确认文本。`/exit` 退出程序。这些是即时发生的确定性操作。

**`local-jsx`** 命令在终端中渲染交互式的 React/Ink UI 组件。`/config` 打开一个设置编辑器，你可以用方向键导航、切换选项、保存 -- 全程不涉及 AI 模型。命令返回的是 React 组件而不是文本。

这种三类型设计优雅地分离了关注点：AI 驱动的行为（`prompt`）、程序化逻辑（`local`）和交互式 UI（`local-jsx`）。添加新命令时，你只需选择匹配需求的类型。

## 命令如何被发现和过滤

当注册中心的 `getCommands()` 函数被调用时，它执行多来源聚合：

```
getCommands() =
    内置（src/commands/ 中的硬编码数组）
  + 技能（从磁盘上的 Prompt 文件扫描）
  + 插件（启动时动态注册）
  + 工作流（用户定义的脚本化链）
  --> 按名称去重
  --> 按认证级别、模式、安全性过滤
  --> 返回统一列表
```

过滤步骤至关重要。命令携带关于谁可以使用它们以及何时可用的元数据：

- **认证级别**：某些命令仅对 `claude-ai` 用户可用，其他对 `console` 用户可用。同一个 CLI 二进制文件服务两种受众，但命令列表会自动适配。
- **运行模式**：当 Claude Code 在远程控制会话中运行时（从手机上的 claude.ai/code），只有 `REMOTE_SAFE_COMMANDS` 集合中的命令会出现。你不希望移动用户意外触发 `/clear` 清除一个长对话。
- **Bridge 安全性**：通过手机或网页远程控制使用的命令经过筛选，防止可能干扰本地环境的操作。

这种分层过滤意味着命令列表自动适应用户的环境，无需任何手动配置。

## 技能系统：AI 遇见命令

技能是命令系统中最新颖的部分。技能是一个打包为命令的**提示词模板**，用户和 AI 模型都可以调用它。

当模型决定需要创建 Git 提交时，它调用 `SkillTool`（定义在 `src/tools/SkillTool/` 中）。该工具查找技能，从磁盘加载其提示词文件，并将提示词注入对话。从模型的角度来看，就好像用户输入了一个详细的请求。

当有人输入 `/commit` 时的流程：

1. 注册中心识别 `/commit` 为技能（类型：`prompt`）
2. 技能的提示词模板从磁盘加载
3. 提示词文本被注入对话
4. 模型读取指令并行动 -- 调用 `Bash` 运行 git 命令，调用 `Read` 检查文件等
5. 模型生成提交

技能的强大之处在于它们只是文本文件。任何人都可以通过创建描述模型应做什么的提示词文件来编写新技能。不需要编译代码。模型的通用智能处理执行。

例如，`/commit` 技能指示模型：检查 `git status`、用 `git diff` 审查差异、查看最近的提交消息了解风格惯例、起草消息、创建提交。`/compact` 技能告诉模型总结对话以回收上下文空间。这些是完全用自然语言描述的复杂工作流。

## 插件架构：技能 + Hook + MCP 服务器

插件是最广泛的扩展机制。一个插件可以提供三种能力的任意组合：

**技能** -- 额外的基于提示词的命令，就像内置的一样。插件可能添加 `/deploy`、`/lint` 或 `/migrate`。

**Hook** -- 在特定生命周期点运行的拦截器。例如，`PermissionRequest` Hook 可以自定义权限提示的处理方式，添加自定义审批逻辑或日志记录。

**MCP 服务器** -- 使用模型上下文协议的外部工具提供者。MCP 服务器可以向智能体暴露全新的工具。插件可能提供数据库查询工具、Jira 集成或 Figma 阅读器。模型将这些视为工具箱中的额外工具，与内置工具无法区分。

### 内置插件 vs 市场插件

Claude Code 在 `plugins/bundled/` 中内置了**捆绑插件**。这些预装可通过 `/plugin` 命令切换开关。它们遵循与任何其他插件相同的注册接口 -- 第一方代码没有特权 API。

**市场插件**从外部来源加载。第三方开发者发布，用户安装使用。因为使用相同的注册接口，它们的工作方式完全相同。

这种双层模型是可扩展软件中经过验证的模式：提供有用的默认值，但让社区无限扩展。VS Code、Webpack 和 Figma 都遵循同样的方法。

## 四个命令的实际运作

让我们追踪四个真实命令来观察各类型的实际效果：

**`/config`**（类型：`local-jsx`）：打开交互式设置编辑器，渲染为 React/Ink 组件。用户用方向键导航、切换设置、保存。不涉及 AI 模型。

**`/commit`**（类型：`prompt`/技能）：生成详细提示词，指导模型用 `git diff` 检查暂存变更、读取最近提交消息以保持风格一致、起草提交信息、执行 `git commit`。模型完成所有思考；命令只提供配方。

**`/compact`**（类型：`prompt`）：告诉模型将当前对话总结为更短形式，保留关键上下文同时减少 Token 使用。纯提示词工程 -- 无需自定义代码。

**`/branch`**（类型：`prompt`/技能）：展开为提示词，指导模型创建或切换 Git 分支，处理未提交更改等边缘情况。

模式很清晰：大多数命令只是提示词。AI 的通用能力承担繁重工作。只有需要直接程序化控制的命令才使用 `local` 或 `local-jsx`。

## 设计你自己的可扩展命令系统

如果你正在构建智能体或聊天机器人，以下是应用这些模式的方法：

1. **定义统一的命令接口。** 保持最小化 -- 名称、类型和处理器。其他一切都应有安全默认值。
2. **构建中央注册表**，从多个来源聚合命令。注册表应该*发现*命令，而不是直接导入。
3. **支持多种命令类型。** 至少支持"prompt"（委托给 LLM）和"local"（直接执行）。
4. **添加上下文过滤。** 不是每个命令在每个上下文中都应可用。按用户角色、会话类型或环境过滤。
5. **使其可插拔。** 定义一个插件接口，让外部代码注册新命令、Hook 和工具提供者。对第一方和第三方扩展使用相同接口。

## 核心要点：多来源聚合模式

Claude Code 命令系统的核心课题是**多来源聚合模式**：

> 不要硬编码所有功能，而是定义标准的命令接口，从多个独立来源（内置代码、技能文件、插件、工作流）聚合实现。在查询时应用上下文感知过滤。这创建了一个 API 统一且功能可无限扩展的系统。

这个模式在设计良好的可扩展系统中随处可见。力量在于约定：只要每个提供者实现相同的接口，注册表就不关心命令来自哪里。
