# 第六章：终端 UI 框架 -- 命令行中的 React

Claude Code 是一个命令行工具，但它的界面远非简单的文本流。它渲染权限对话框、动画加载器、可滚动的代码视图、可点击的按钮和多面板布局 -- 全部在终端内完成。这得益于 Claude Code 使用了一个经过深度定制的 **Ink** 分支，Ink 是一个用于终端应用的 React 渲染器。本章解释终端如何成为声明式 UI 的画布。

## 什么是 Ink？

Ink 是一个开源库，让你可以使用 React 构建终端 UI。你的 React 组件不是渲染到浏览器 DOM，而是使用 ANSI 转义序列渲染到终端字符单元格。你编写 `<Box>` 和 `<Text>` 而不是 `<div>` 和 `<span>`，Ink 处理布局、样式和输出。

Claude Code 分叉了 Ink 并重建了其主要部分以获得极致性能。原版 Ink 在每次变更时重新渲染整个屏幕。Claude Code 的版本只重新渲染发生变化的部分，使用零分配缓冲区，并产生最少的终端输出。结果是即使渲染数千行工具输出，UI 也感觉瞬时响应。

整个框架位于 `src/ink/` 中。

## 渲染管线

Claude Code 中的每一帧都遵循五个阶段的管线：

```
React 状态变更
     |
     v
Reconciler (reconciler.ts)
  - React Fiber 对比虚拟树
  - 更新 Ink DOM 节点
  - 在变更的节点上设置 dirty 标志
     |
     v
布局 (Yoga)
  - 在 DOM 树上进行 Flexbox 计算
  - 将 Ink 样式转换为 Yoga 属性
  - 产生精确到字符单元格的位置
     |
     v
渲染 (render-node-to-output.ts)
  - 遍历 DOM 树
  - 跳过干净的子树（从上一帧 blit）
  - 将文本、边框、背景写入 Screen 缓冲区
     |
     v
屏幕缓冲 (screen.ts)
  - Int32Array 打包的单元格（每个单元格 2 个字）
  - 与上一帧的缓冲区进行 diff
  - 产生最小的 ANSI 转义补丁列表
     |
     v
终端输出 (terminal.ts)
  - 用 DEC 2026 同步输出包裹补丁
  - 单次原子写入 stdout
```

关键洞见是大多数帧只有很小的变化 -- 加载器转动、一行文本被追加、进度条前进。通过跟踪 dirty 标志并在单元格级别进行 diff，系统避免了重新计算或重写任何未变化的内容。

## 核心优化

### O(dirty) 渲染

标准方法是每帧重新渲染整个树。Claude Code 的 reconciler 在每个变更的 DOM 节点上设置 `dirty` 标志。渲染过程检查此标志并跳过干净的整个子树。对于一个有 100 个组件但只有 1 个变化的屏幕，这意味着处理 1 个节点而不是 100 个。

### 无 GC 缓冲

JavaScript 的垃圾收集器可能在终端 UI 中造成可见的暂停。Claude Code 通过使用 `Int32Array` 作为屏幕缓冲区来消除这一问题。所有单元格数据打包到一个扁平的类型化数组中，没有对象分配就没有 GC 压力。

### Blitting

当子树没有变化时，其上一帧的输出可以直接复制到当前帧的缓冲区。这称为 "blitting"（块传输）。渲染过程通过 `nodeCache` 检测干净节点并复制它们的矩形区域。这对于文件预览等大型静态内容特别有效。

### 同步输出

终端模拟器在输出到达时即刻渲染，大量更新写入时可能导致可见的撕裂。Claude Code 用 DEC 模式 2026 的开始/结束标记包裹每一帧。支持此协议的终端模拟器会缓冲整个更新并原子性地显示，消除闪烁。

### 双缓冲

`Ink` 类维护两个 `Frame` 对象：`frontFrame`（上次显示的内容）和 `backFrame`（正在准备的内容）。渲染后，系统在单元格级别对两个帧进行 diff，仅产生将前帧转换为后帧所需的 ANSI 序列。然后帧交换。

## 打包的单元格格式

屏幕缓冲区中的每个终端单元格在 `src/ink/screen.ts` 中表示为 2 x Int32 字：

- **字 0**：驻留的字符 ID。字符存储在 `CharPool` 中，通过整数索引引用。ASCII 字符获得快速路径直接查找；非 ASCII 字符使用 `Map`。池在所有屏幕间共享，因此 blitting 可以直接复制 ID 而无需重新驻留。

- **字 1**：包含前景色、背景色、文本属性（粗体、斜体、下划线等）和显示宽度的打包位字段。这种编码允许通过单个整数比较进行单元格比较。

还有用于 OSC 8 超链接 URI 的 `HyperlinkPool` 和用于 ANSI SGR 序列的 `StylePool`。所有池都使用相同的驻留模式：存储字符串一次，通过整数 ID 引用，通过 ID 而非字符串相等来比较。

## 事件系统

终端输入与浏览器输入根本不同。终端发送必须解码的原始字节序列。Claude Code 的事件管线如下工作：

**1. 按键解析** (`src/ink/parse-keypress.ts`) -- 原始输入字节被解码为结构化的按键事件。解析器处理多种协议：标准 ANSI 转义序列、Kitty 键盘协议（用于修饰键跟踪）、SGR 鼠标跟踪（用于点击和滚动）和括号粘贴模式。

**2. 事件分发** (`src/ink/events/dispatcher.ts`) -- 按键解析后，进入仿照浏览器 DOM 的捕获/冒泡分发循环：

```
根节点（捕获阶段，自顶向下）
  -> 父节点（捕获）
    -> 目标节点（at_target，捕获和冒泡处理器）
  -> 父节点（冒泡）
根节点（冒泡阶段，自底向上）
```

分发器从目标节点走到根节点，收集捕获处理器（前插）和冒泡处理器（追加）。这让父节点有机会在子节点看到事件之前拦截它们，就像浏览器中带 `capture: true` 的 `addEventListener`。

**3. 焦点管理** (`src/ink/focus.ts`) -- `FocusManager` 跟踪哪个节点拥有焦点。Tab 导航、自动焦点和程序化焦点变更都通过此系统路由。只有焦点节点的处理器会为键盘事件触发。

## 终端的 React Hooks

Claude Code 提供了自定义 Hook，将 React 的声明式模型与终端特定的关注点桥接起来：

| Hook | 文件 | 用途 |
|------|------|------|
| `useInput` | `hooks/use-input.ts` | 订阅焦点元素上的键盘事件 |
| `useTerminalViewport` | `hooks/use-terminal-viewport.ts` | 知道哪些行可见；启用视口裁剪 |
| `useAnimationFrame` | `hooks/use-animation-frame.ts` | 与渲染帧时钟同步运行逻辑 |
| `useTerminalTitle` | `hooks/use-terminal-title.ts` | 通过转义序列设置终端窗口标题 |
| `useDeclaredCursor` | `hooks/use-declared-cursor.ts` | 为 IME 输入定位硬件光标 |
| `useSelection` | `hooks/use-selection.ts` | 在全屏模式下跟踪文本选择状态 |
| `useTabStatus` | `hooks/use-tab-status.ts` | 设置终端标签页状态栏 |

这些 Hook 让组件作者可以使用与 React Web 应用相同的声明式模式来使用终端特性。

## 组件

`src/ink/components/` 中的组件库提供了终端 UI 的构建块：

- **`Box`** -- Flexbox 容器，等同于 `<div>`。支持内边距、外边距、边框和所有 Yoga flex 属性。
- **`Text`** -- 带颜色、粗体、斜体、下划线、删除线和变暗的样式化文本。
- **`ScrollBox`** -- 带虚拟滚动的可滚动容器。处理鼠标滚轮事件、键盘滚动和粘性滚动（自动跟随新内容）。
- **`Button`** -- 带焦点和悬停状态的可点击元素。
- **`Link`** -- 在用户浏览器中打开 URL 的 OSC 8 超链接。
- **`RawAnsi`** -- 不经处理直接传递预格式化的 ANSI 文本，用于显示原始工具输出。
- **`AlternateScreen`** -- 切换到终端的备用屏幕缓冲区（全屏模式）。
- **`Spacer`** -- 用于布局的弹性间距器。

## 工具结果如何渲染

当工具完成时，其结果通过工具特定的 UI 组件渲染。每个工具定义 `renderResultUI()` 和 `renderToolUseProgressUI()` 方法，返回 React 元素。这些元素被组合到对话视图中，由 Yoga 布局，渲染到屏幕缓冲区，并与上一帧进行 diff。

例如，当 `FileReadTool` 返回文件内容时，它在 `ScrollBox` 中渲染语法高亮的文本。当 `BashTool` 返回命令输出时，如果命令失败，它会以适当的错误样式渲染输出。工具不需要了解 ANSI 代码或终端尺寸 -- 它只需返回 React 组件，框架处理其余一切。

## 核心要点

Claude Code 证明了终端 UI 不必是原始的。通过将 React 的声明式组件模型应用于终端渲染 -- 并通过 dirty 跟踪、类型化数组缓冲区、blitting 和同步输出大力投入性能优化 -- 你可以构建在响应性和复杂度上媲美 GUI 应用的界面。框架处理困难的部分（ANSI 转义、布局计算、输入协议解析、GC 回避），让组件作者可以专注于 UI 应该是什么样子，而不是如何逐字符地渲染它。
