# 第六章：终端 UI 框架 — 在命令行里跑 React

## 为什么终端需要 UI 框架

Claude Code 不是一个简单的"打印文本"的命令行工具。它需要实时展示：工具执行进度、多个并行操作的状态、可折叠的搜索结果、权限确认对话框、Todo 列表面板……这些交互式元素用 `console.log` 是搞不定的。

传统方案有 blessed（Node.js 的终端 UI 库）和 ncurses（C 语言的终端库），但它们都是命令式 API（💡 你需要手动告诉程序"在第 3 行第 5 列画一个字符"），写起来痛苦，维护更痛苦。

[Ink](https://github.com/vadimdemedes/ink) 提出了一个激进的想法：**用 React 渲染终端 UI**。你写 JSX 组件，Ink 负责把虚拟 DOM 转换成终端字符。声明式 UI 在终端里也能工作。

Claude Code 使用了 Ink，但没有直接依赖它——而是 **Fork 了整个项目**并做了大量性能优化。为什么？因为原版 Ink 在 Claude Code 的使用强度下会出现性能问题：每次工具调用都会触发 UI 更新，多个工具并行执行时更新频率极高，GC 暂停会导致输出卡顿。

Claude Code 的 Fork 解决了这些问题，最终形成了一个独立的高性能终端渲染引擎。代码在 `src/ink/` 目录下。

## 渲染管线：从 React 到终端字符

整个渲染管线分为六个阶段。理解这条管线，就理解了整个 UI 框架。

### 第 1 阶段：React 状态变更 → Reconciler

和浏览器 React 一样，一切从状态变更开始。当工具返回结果、进度更新、用户输入时，React 组件的 state 改变，触发重渲染。

`reconciler.ts` 是 React 和 Ink DOM 之间的桥梁。它实现了 `react-reconciler`（💡 React 官方提供的自定义渲染器接口，[了解更多](https://www.baidu.com/s?wd=react-reconciler+自定义渲染器)）接口，把 React Fiber 的 create/update/delete 操作映射到 Ink 自定义 DOM 的节点操作。

### 第 2 阶段：Ink DOM 和脏标记

`dom.ts` 定义了轻量级的 DOM 节点：`DOMElement` 和 `TextNode`。关键设计是**脏标记（Dirty Flag）**——每个节点维护一个标志，记录自上次渲染后是否被修改。

当 Reconciler 修改一个节点时，该节点及其所有祖先都被标记为"脏"。渲染阶段只处理脏子树，干净的子树直接跳过。这就是 **O(dirty) 渲染**的核心——修改一个叶子节点不需要重渲染整棵树。

### 第 3 阶段：Yoga 布局

布局计算由 Yoga 引擎完成——这是 Facebook 开发的跨平台 Flexbox（💡 CSS 弹性盒布局，[搜索了解更多](https://www.baidu.com/s?wd=CSS+Flexbox+布局)）实现，编译为 WASM 在 Node.js 中运行。

`layout/yoga.ts` 负责将 Ink 的样式属性（margin、padding、flex、width）转换为 Yoga 的属性调用。`layout/node.ts` 做了一个关键的适配：终端中的坐标必须是整数（字符单元格对齐），所以所有 Yoga 计算出的浮点坐标都要取整。这听起来简单，但取整方向的选择（floor 还是 round）会影响布局的视觉一致性。

### 第 4 阶段：渲染树遍历和 Blitting

`render-node-to-output.ts` 是渲染阶段的核心。它遍历 DOM 树，把每个节点的内容写入 `Output` 缓冲区。

**Blitting（💡 位块传送）优化**是这个阶段最重要的性能技巧。当一个子树没有被标记为脏时，渲染器不会重新计算它的输出，而是直接从上一帧的缓冲区中复制（blit）对应区域。对于一个工具结果已经渲染过且没有变化的区域，这意味着零计算成本。

### 第 5 阶段：屏幕缓冲（Screen Buffer）

`screen.ts` 是整个框架中最底层也最精妙的部分。它用 **`Int32Array` 类型化数组**（💡 JavaScript 的固定长度整数数组，直接操作内存，[了解更多](https://www.baidu.com/s?wd=JavaScript+Int32Array+类型化数组)）来存储终端屏幕的每个字符单元格。

**为什么不用普通 JavaScript 对象？** 因为 GC（垃圾回收）。

一个 80x24 的终端有 1920 个字符单元格。如果每个单元格是一个 JS 对象 `{ char: 'A', fg: 'red', bg: 'black', bold: true }`，那每帧你就创建了 1920 个短命对象。在每秒 30 帧的更新频率下，GC 每秒要处理 57600 个对象。GC 暂停直接表现为输出卡顿。

Claude Code 的方案是**打包格式（Packed Format）**：每个单元格占 2 个 Int32（8 字节）。

**Word 0** — 驻留字符 ID：`CharPool` 类维护一个字符串驻留池，每个唯一字符串分配一个整数 ID。ASCII 字符有快速路径——直接用 `Int32Array` 做 charCode → ID 的映射，跳过 `Map.get` 的开销。

源码 `screen.ts:21-53` 展示了 `CharPool.intern()` 的实现：

```
ASCII 字符（0-127）→ 直接数组查找（O(1)，无哈希）
其他字符（CJK、Emoji 等）→ Map 查找
```

**Word 1** — 打包位字段：前景色、背景色、属性（粗体/斜体/下划线等）、显示宽度全部编码到一个 32 位整数中。位操作替代属性访问，速度快且零 GC 压力。

类似的驻留池还有 `HyperlinkPool`（超链接 URL）和 `StylePool`（样式组合）。整个屏幕缓冲的内存占用是固定的——`rows × cols × 8 bytes`——不随渲染次数增长。

### 第 6 阶段：差分输出和同步写入

`terminal.ts` 负责最后一步：比较新旧两帧的屏幕缓冲，生成最小的 ANSI 转义序列差异，然后写入 `stdout`。

**DEC 2026 同步更新**：终端渲染有一个经典问题——如果一帧的 ANSI 序列分多次 write 调用输出，用户可能看到半渲染的中间状态（闪烁）。DEC 私有模式 2026 解决了这个问题：在帧开始前发送 BSU（Begin Synchronized Update）序列，帧结束后发送 ESU（End Synchronized Update），终端会缓冲所有中间输出，只在 ESU 时一次性刷新。

支持 DEC 2026 的终端包括 iTerm2、Kitty、WezTerm 等现代终端。对于不支持的终端，序列会被忽略，退化到普通的逐步输出。

## 五大性能优化总结

| 优化 | 问题 | 方案 | 效果 |
|------|------|------|------|
| O(dirty) 渲染 | 每次状态变更重渲染整棵树 | 脏标记，只处理修改过的子树 | 渲染时间与变更量成正比，不与树大小成正比 |
| GC-free 缓冲 | 每帧创建数千短命对象 | Int32Array 打包格式 + 驻留池 | 渲染阶段零对象分配，消除 GC 暂停 |
| Blitting | 未变化区域的重复计算 | 直接从上一帧复制未变化区域 | 大部分 UI（历史消息）零渲染成本 |
| 同步输出 | 多次 write 造成闪烁 | DEC 2026 BSU/ESU 包裹 | 帧级原子更新，无中间状态 |
| 驻留池 | 重复字符串的内存和比较开销 | CharPool/HyperlinkPool/StylePool | 字符串比较变为整数比较 |

这些优化的组合效果是：即使在高频率更新（多工具并行、流式输出）的场景下，UI 也能保持流畅，不会出现卡顿或闪烁。

## 事件系统：从按键到 React

### 输入解析（parse-keypress.ts）

终端输入是原始字节流，同一个按键在不同终端中可能产生不同的字节序列。`parse-keypress.ts` 是一个精密的状态机，能够解析：

- **标准 ANSI 序列** — 方向键、功能键、Home/End 等
- **Kitty 键盘协议** — 现代终端的扩展协议，支持按键释放事件和修饰键组合
- **SGR 鼠标追踪** — 鼠标点击、移动、滚轮事件
- **粘贴括号** — 区分用户输入和粘贴的文本（防止粘贴内容被当作命令执行）

### 事件分发（Dispatcher）

解析出的事件通过 `events/dispatcher.ts` 分发到 DOM 树。分发模型借鉴了浏览器的**捕获/冒泡**机制：事件先从根节点向下传递（捕获阶段），到达目标节点后再向上冒泡。

焦点管理（`focus.ts`）实现了类似浏览器的 Tab 导航——在终端 UI 的多个可交互元素之间切换焦点。

## React Hooks

Claude Code 提供了一组终端专用的 React Hooks：

| Hook | 用途 | 应用场景 |
|------|------|---------|
| `useInput` | 高级键盘输入处理 | 快捷键、文本输入 |
| `useTerminalViewport` | 检测组件是否在屏幕可见范围内 | 离屏组件跳过渲染（性能优化） |
| `useAnimationFrame` | 同步动画时钟 | 加载动画、进度条 |
| `useTerminalTitle` | 设置终端窗口标题 | 显示当前操作状态 |
| `useTerminalFocus` | 检测终端是否获得焦点 | 焦点丢失时暂停动画 |
| `useSelection` | 文本选择管理 | 复制/粘贴 |

`useTerminalViewport` 是一个特别聪明的优化：当一个消息滚出屏幕后，它的 React 组件仍然存在于虚拟 DOM 中，但 `useTerminalViewport` 让渲染器知道这个组件"不可见"，可以跳过渲染计算。在长对话（几百条消息）中，这个优化的效果非常显著。

## 组件库

Claude Code 的 UI 组件全部基于 Ink 的 React 模型：

- **Box** — Flexbox 容器，支持 margin/padding/border/flex
- **Text** — 带样式的文本（颜色、粗体、斜体、下划线）
- **ScrollBox** — 可滚动容器，自动处理溢出
- **Button** — 可点击按钮（通过键盘焦点或鼠标）
- **Link** — 可点击超链接（在支持的终端中）
- **RawAnsi** — 直接输出 ANSI 序列（用于嵌入外部工具的输出）
- **AlternateScreen** — 替代屏幕缓冲区（全屏模式）

## 工具渲染系统

每个工具的 UI 通过三组方法与 Ink 框架配合：

**renderToolUseMessage(input)** — 工具被调用时显示什么。比如 BashTool 显示命令文本，FileReadTool 显示文件路径。

**renderToolUseProgressMessage(progress)** — 工具执行过程中的进度展示。BashTool 显示实时输出流，AgentTool 显示子 Agent 的思考过程。

**renderToolResultMessage(output)** — 工具完成后的结果展示。可以是折叠的（搜索结果默认折叠，点击展开）或完整的（文件内容直接显示）。

还有一个 `renderGroupedToolUse` 方法，处理多个相同工具并行执行的分组显示——比如同时搜索多个文件时，结果合并为一个折叠面板。

这种"每个工具自带渲染逻辑"的设计使得添加新工具时，UI 展示是工具自身的职责，不需要修改框架代码。

## 关键收获

Claude Code 的终端 UI 框架证明了一件事：**声明式 UI 和极致性能不矛盾**。通过脏标记、打包缓冲、位块传送和同步输出的组合，React 在终端中可以达到和手写 ANSI 序列相当的性能，同时保持了声明式编程的开发效率。

如果你要在终端中构建复杂的交互式 UI，这个架构值得认真研究。关键不是"用 React"本身，而是它在底层做的那些性能优化——这些优化在任何高频更新的 UI 系统中都适用。
