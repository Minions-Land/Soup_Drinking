# 第四章：任务生命周期

## 什么是任务？

在 Claude Code 中，**任务（Task）** 是可以独立于主对话循环运行的工作单元。查询循环（第一章）处理同步工具调用 -- 模型请求工具，等待结果，然后继续。而任务处理需要在后台运行、跨回合持续或作为一等实体管理的工作。

这样理解区别：工具调用就像调用函数。任务就像生成线程。

## 任务抽象

Task 接口定义在 `src/Task.ts` 中。每个任务都有核心属性：

```typescript
interface Task {
  id: string;                      // 唯一标识符
  type: TaskType;                  // 任务类别
  status: TaskStatus;              // 当前生命周期状态
  description: string;             // 人类可读描述
  createdAt: number;               // 时间戳
  result?: TaskResult;             // 最终输出（完成时）
  error?: string;                  // 错误消息（失败时）
  
  start(): Promise<void>;
  cancel(): Promise<void>;
  getProgress(): TaskProgress;
}
```

## 任务类型

| 类型 | 目录 | 描述 |
|------|------|------|
| `LocalShellTask` | `src/tasks/LocalShellTask/` | 后台 Bash 进程 |
| `LocalAgentTask` | `src/tasks/LocalAgentTask/` | 本地子智能体执行 |
| `RemoteAgentTask` | `src/tasks/RemoteAgentTask/` | 远程智能体执行 |
| `InProcessTeammateTask` | `src/tasks/InProcessTeammateTask/` | 同进程 Swarm 成员 |
| `DreamTask` | `src/tasks/DreamTask/` | 后台"梦境"处理 |

## 生命周期状态

```
  +---------+     start()     +-----------+
  | PENDING | --------------> | RUNNING   |
  | 等待中   |                 | 运行中     |
  +---------+                 +-----------+
                                |       |
                       complete |       | error/cancel
                                v       v
                          +-----------+  +--------+
                          | COMPLETED |  | FAILED |
                          | 已完成     |  | 已失败  |
                          +-----------+  +--------+
                                         |
                                         v
                                     +--------+
                                     | KILLED |
                                     | 已终止  |
                                     +--------+
```

状态转换是强制的 -- 你不能从 COMPLETED 回到 RUNNING。这防止了竞态条件并使任务状态可预测。

## AppState 注册

任务创建时立即在 `AppState.tasks` 中注册。目的包括：发现（模型可以查询运行中的任务）、状态报告（UI 展示进度）、生命周期管理（关闭时取消所有任务）、持久化（跨会话保存和恢复）。

## 后台执行

后台任务在独立的异步（💡 不等待某个操作完成就继续做其他事，就像发邮件后不用守着等回复）上下文中运行，拥有自己的 AsyncLocalStorage 上下文、错误边界、取消范围和输出缓冲区。主对话继续进行，模型通过 `TaskGetTool`、`TaskOutputTool`、`TaskStopTool` 管理任务。完成时通过 `<task-notification>` XML 通知。

## 僵尸防护

防御措施包括：超时强制、健康检查、会话清理、孤儿检测，以及 `killShellTasksForAgent()` 确保智能体被终止时所有派生的 bash 进程也被清理。

## AsyncLocalStorage 隔离

每个任务通过 `AsyncLocalStorage`（💡 Node.js提供的"隔离储物柜"，每个程序执行路径有自己的独立存储，互不干扰——类似每位员工有各自抽屉，不会拿错别人的东西）获得隔离上下文。上下文自动流过异步操作，无需手动传递参数。

```typescript
const taskStorage = new AsyncLocalStorage<TaskContext>();

function startTask(task: Task) {
  taskStorage.run({ taskId: task.id, state: {} }, async () => {
    await task.start();
  });
}
```

这给每个任务自己的"线程本地"存储，而没有实际线程的复杂性。

> 💡 **新手提示：AsyncLocalStorage 是什么？**  
> 想象一个大厅里同时有多位服务员（多个异步任务）。普通变量就像大厅的公告板，所有人都能看到，很容易混乱。`AsyncLocalStorage` 就像给每位服务员发一个私人小本本，只属于自己的那条服务流程，不会和其他人的记录混在一起。  
> 🔍 想深入了解？[搜索"Node.js AsyncLocalStorage 是什么"](https://www.baidu.com/s?wd=Node.js+AsyncLocalStorage+是什么)

## 持久化

任务状态持久化到 `.claude/projects/` 目录，支持崩溃恢复和会话恢复。进行中的任务以当前状态保存，已完成的结果被缓存。

## 设计要点

1. **将同步工具与异步工作分开。** 任务为后台工作提供干净的模型。
2. **集中注册所有工作。** 任务注册表使监控、管理和清理成为可能。
3. **强制生命周期状态。** 明确的状态机防止不可能的转换。
4. **防护僵尸。** 任何生成后台工作的系统都必须有超时、健康检查和清理机制。
5. **使用 AsyncLocalStorage 进行隔离。** 提供每任务上下文，不污染函数签名。
