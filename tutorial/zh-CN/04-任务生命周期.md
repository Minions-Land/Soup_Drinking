# 第四章：任务生命周期

## 1. Task 抽象的设计动机

第一章讲的查询循环（Query Loop）是**同步**的——模型请求一个工具，等待结果，继续推理。这个模式在大多数情况下工作得很好，但它有一个根本限制：调用者被阻塞了。

现实中很多工作不应该阻塞主对话：

- 用户要求运行一个耗时 30 秒的测试套件——没人愿意盯着进度条看 30 秒
- 验证 Agent 需要独立运行构建、测试、代码审查——这些可以和实现并行进行
- 主 Agent 的查询本身可能需要被"后台化"——用户按两次 Ctrl+B 后继续输入新问题
- 多个 Agent 在 Swarm 模式下同进程并发执行——它们的工作互不阻塞
- 后台记忆整理（Dream）在空闲时运行——用户完全不需要知道它的存在

所有这些场景都有一个共同需求：一个统一的接口来追踪"在某个地方有一件事在运行"的状态。这就是 `Task` 抽象存在的原因。

类比思考：如果工具调用像**函数调用**（同步、返回值），那任务就像**线程/goroutine**（异步、通过状态查询结果）。但和真正的线程不同，Task 没有 OS 级别的上下文切换开销——它利用 Node.js 的事件循环和 `AsyncLocalStorage` 实现协作式并发（Cooperative Concurrency）💡 *各任务在 await 点主动让出控制权，而非被抢占式调度*。

一个关键的设计决策是：**所有**后台操作都必须通过 Task 接口注册。哪怕是最简单的后台 bash 命令，也需要创建一个 `LocalShellTask`。这带来了管理成本，但换来了两个关键能力：(1) 全局可见性——UI 可以展示所有正在进行的工作；(2) 统一清理——会话结束时可以可靠地终止一切。

## 2. Task 接口的核心结构

Task 的类型定义在 `src/Task.ts` 中。这个文件出乎意料地精简——不到 130 行，定义了整个任务系统的基础。

### 2.1 TaskType：七种任务类型

```typescript
type TaskType =
  | 'local_bash'
  | 'local_agent'
  | 'remote_agent'
  | 'in_process_teammate'
  | 'local_workflow'
  | 'monitor_mcp'
  | 'dream'
```

注意这里用的是**字面量联合类型**（Literal Union Type），而非枚举。字面量联合在 JSON 序列化/反序列化时更友好，且支持更好的类型收窄（Type Narrowing）。

### 2.2 TaskStatus：五种生命周期状态

```typescript
type TaskStatus =
  | 'pending'
  | 'running'
  | 'completed'
  | 'failed'
  | 'killed'
```

### 2.3 TaskStateBase：基础状态字段

每个具体任务类型都通过**交叉类型**（Intersection Type）扩展 `TaskStateBase`：

```typescript
type TaskStateBase = {
  id: string           // 唯一标识（前缀+8位随机字符）
  type: TaskType       // 任务类型
  status: TaskStatus   // 当前状态
  description: string  // 人类可读描述
  toolUseId?: string   // 触发该任务的工具调用 ID（可选）
  startTime: number    // 创建时间戳
  endTime?: number     // 结束时间戳（完成/失败/终止时设置）
  totalPausedMs?: number // 累计暂停时长
  outputFile: string   // 输出文件路径
  outputOffset: number // 已报告的输出偏移量
  notified: boolean    // 是否已通知父级
}
```

几个值得关注的字段：

**`id` 的生成规则**。`generateTaskId()` 使用类型前缀 + 8 位随机字符。前缀映射表（`TASK_ID_PREFIXES`）为：bash → `b`、agent → `a`、remote → `r`、teammate → `t`、workflow → `w`、monitor → `m`、dream → `d`。使用 `0-9a-z` 字母表（36^8 ≈ 2.8 万亿组合），注释中提到这是为了"resist brute-force symlink attacks"——防止沙箱内的恶意代码猜测任务 ID 来创建符号链接攻击。

**`outputFile`**。每个任务的输出被写入磁盘文件，路径由 `getTaskOutputPath()` 生成，位于项目临时目录下的 `tasks/` 子目录。这个设计让大输出（比如长时间运行的 bash 命令输出）不会撑爆内存——AppState 只存储偏移量，需要时从磁盘读取。

**`notified`**。防止重复通知的标志。一个任务完成后应该且只应该通知父级一次。`enqueueAgentNotification()` 中有一个原子性检查（`LocalAgentTask.tsx` 第 227-240 行）：先检查 `notified` 是否为 false，是则设为 true 并入队通知，否则跳过。这避免了竞态条件下的重复消息。

### 2.4 Task 调度接口

```typescript
type Task = {
  name: string
  type: TaskType
  kill(taskId: string, setAppState: SetAppState): Promise<void>
}
```

这个接口**只有 kill 方法**。早期版本还有 `spawn` 和 `render`，但在重构中被移除了（代码注释引用了 PR #22546）。当前的 `Task` 接口是纯粹的"注册 + 终止"模型——各具体任务类型自行管理创建和渲染。

`getAllTasks()` 函数（`src/tasks.ts`）返回所有任务类型的调度器列表。`getTaskByType()` 根据类型查找对应调度器——当需要终止一个任务时，框架通过类型找到调度器，调用其 `kill()` 方法。

## 3. 七种任务类型详解

### 3.1 LocalShellTask：后台 Bash 进程

**源文件**：`src/tasks/LocalShellTask/`

**关系**：BashTool 调用 `ShellCommand` 执行命令。当 `run_in_background: true` 时，命令不阻塞主循环，而是创建一个 `LocalShellTask` 来追踪它。

**状态扩展**（`guards.ts`）：
```
LocalShellTaskState = TaskStateBase & {
  type: 'local_bash'
  command: string            // 执行的命令
  result?: { code, interrupted }  // 退出码和中断标志
  shellCommand: ShellCommand | null  // Shell 进程引用（运行时）
  isBackgrounded: boolean    // 是否已后台化
  agentId?: AgentId          // 生成该任务的 Agent ID
  kind?: 'bash' | 'monitor'  // UI 显示变体
}
```

`agentId` 字段是僵尸防护的关键——当一个 Agent 退出时，`killShellTasksForAgent()` 通过这个字段找到所有该 Agent 生成的 bash 任务并终止它们。

`kind` 字段区分普通 bash 任务和监控任务。`'monitor'` 类型在 UI 中显示描述而非命令文本，使用不同的对话框标题和状态栏样式。

**输出处理**：bash 进程的 stdout/stderr 被写入磁盘文件（`outputFile`）。一个看门狗（Watchdog）轮询文件大小，超过 `MAX_TASK_OUTPUT_BYTES`（5GB）时终止进程。这个限制在 `diskOutput.ts` 中定义，注释说明了 file 模式（bash）和 pipe 模式（hooks）共享同一个上限。

### 3.2 LocalAgentTask：本地子智能体

**源文件**：`src/tasks/LocalAgentTask/LocalAgentTask.tsx`

**关系**：AgentTool 在后台模式下创建 `LocalAgentTask`。每个后台运行的子 Agent 对应一个 `LocalAgentTask` 实例。

**状态扩展**：
```
LocalAgentTaskState = TaskStateBase & {
  type: 'local_agent'
  agentId: string            // Agent 唯一 ID
  prompt: string             // 任务提示
  selectedAgent?: AgentDefinition  // Agent 定义
  agentType: string          // Agent 类型名称
  model?: string             // 模型覆盖
  abortController?: AbortController  // 取消控制器
  progress?: AgentProgress   // 进度信息
  retrieved: boolean         // 结果是否已被父级取走
  messages?: Message[]       // 对话历史（用于 UI 查看）
  isBackgrounded: boolean    // 是否已后台化
  pendingMessages: string[]  // 排队中的消息
  retain: boolean            // UI 是否正在持有此任务
  diskLoaded: boolean        // 磁盘对话记录是否已加载
  evictAfter?: number        // 面板可见性截止时间
}
```

**进度追踪**。`ProgressTracker` 和 `updateProgressFromMessage()` 是精心设计的：

```typescript
type ProgressTracker = {
  toolUseCount: number       // 工具调用计数
  latestInputTokens: number  // 最新输入 token 数（累计值）
  cumulativeOutputTokens: number  // 累计输出 token 数
  recentActivities: ToolActivity[]  // 最近 5 个工具活动
}
```

注意 `latestInputTokens` 和 `cumulativeOutputTokens` 的不同处理方式。Anthropic API 的 `input_tokens` 是**每轮累计的**（包含所有之前的上下文），所以取最新值而非累加。`output_tokens` 是每轮独立的，所以需要累加。这个细节如果搞错会导致进度显示的 token 数翻倍。

**消息队列**。`pendingMessages` 和 `queuePendingMessage()` 用于"mid-turn"消息注入。当用户通过 `SendMessageTool` 给正在运行的 Agent 发消息时，消息先进入队列，在下一个工具轮次边界被 `drainPendingMessages()` 取出并注入。

**通知构建**。`enqueueAgentNotification()` 构建 `<task-notification>` XML：
```xml
<task-notification>
  <task-id>{taskId}</task-id>
  <tool-use-id>{toolUseId}</tool-use-id>
  <output-file>{outputPath}</output-file>
  <status>completed|failed|killed</status>
  <summary>{人类可读摘要}</summary>
  <result>{Agent 最终文本响应}</result>
  <usage>
    <total_tokens>N</total_tokens>
    <tool_uses>N</tool_uses>
    <duration_ms>N</duration_ms>
  </usage>
</task-notification>
```

这个 XML 被 `enqueuePendingNotification()` 放入全局消息队列，在父 Agent 的下一次工具轮次间隙作为 user 消息注入。

**面板显示和内存管理**。`retain`、`diskLoaded`、`evictAfter` 三个字段共同管理 UI 生命周期：

- `retain = true` 时任务不被回收（用户正在查看）；`diskLoaded = true` 表示已从磁盘懒加载历史对话；`evictAfter` 在终态时设置，给用户 `PANEL_GRACE_MS`（30 秒）查看时间后才允许回收

### 3.3 RemoteAgentTask：远程智能体

**源文件**：`src/tasks/RemoteAgentTask/RemoteAgentTask.tsx`

**关系**：当任务通过 `isolation: 'remote'` 或 `teleportToRemote()` 被发送到远程 CCR（Claude Code Remote）环境时创建。这主要是 Anthropic 内部使用的功能。

**状态扩展**包括：
- `remoteTaskType`：`'remote-agent'`、`'ultraplan'`、`'ultrareview'`、`'autofix-pr'`、`'background-pr'` 等
- `sessionId`：远程会话 ID，用于 API 轮询
- `log: SDKMessage[]`：远程会话的消息日志
- `todoList: TodoList`：远程 Agent 的 todo 列表
- `isLongRunning`：长时运行标志，不在第一个 `result` 后标记完成
- `pollStartedAt`：本地轮询开始时间，防止恢复会话时立即超时

远程任务通过 `pollRemoteSessionEvents()` 轮询远程会话状态。它有一个注册的 `RemoteTaskCompletionChecker` 回调机制——不同的远程任务类型可以注册自定义的完成检测逻辑（比如 PR autofix 任务在 PR 合并后才算完成）。

### 3.4 InProcessTeammateTask：同进程 Swarm 成员

**源文件**：`src/tasks/InProcessTeammateTask/types.ts`

**关系**：在 Agent Swarm（多智能体群组）模式下，多个 Agent 作为"队友"在**同一个进程**中并发运行。每个队友对应一个 `InProcessTeammateTask`。

这是最复杂的任务类型。**状态扩展**：

```
InProcessTeammateTaskState = TaskStateBase & {
  type: 'in_process_teammate'
  identity: TeammateIdentity   // 身份信息（名称、团队、颜色）
  prompt: string               // 初始任务提示
  model?: string               // 模型覆盖
  selectedAgent?: AgentDefinition  // Agent 定义
  abortController?: AbortController  // 终止整个队友
  currentWorkAbortController?: AbortController  // 终止当前轮（不杀队友）
  awaitingPlanApproval: boolean  // 是否等待计划审批
  permissionMode: PermissionMode  // 独立的权限模式
  progress?: AgentProgress     // 进度
  messages?: Message[]         // 对话历史（UI 查看用）
  isIdle: boolean              // 是否空闲
  shutdownRequested: boolean   // 是否请求关闭
  pendingUserMessages: string[]  // 待处理的用户消息
  onIdleCallbacks?: Array<() => void>  // 空闲回调
  lastReportedToolCount: number  // 进度增量计算
  lastReportedTokenCount: number
}
```

几个值得深入的设计：

**双 AbortController**。`abortController` 终止整个队友的生命周期，`currentWorkAbortController` 只终止当前正在进行的工作轮次。这使得用户可以打断队友的当前操作（比如取消一个长时间运行的命令）而不杀死整个队友。

**计划模式**。`awaitingPlanApproval` 和 `identity.planModeRequired` 实现了"plan-then-execute"工作流：队友先生成实现计划，等团队领导（Team Lead）审批后才开始执行。审批通过 `SendMessageTool` 的 `plan_approval_response` 结构化消息完成。

**空闲机制**。`isIdle` 标志和 `onIdleCallbacks` 实现了高效的"等待空闲"模式。团队领导想等某个队友完成当前工作时，不需要轮询——注册一个 `onIdleCallback`，队友空闲时会直接调用。

**消息上限**。`TEAMMATE_MESSAGES_UI_CAP = 50` 限制了 `messages` 数组的长度。注释中的分析数据很有说明性：

> BQ analysis (round 9, 2026-03-20) showed ~20MB RSS per agent at 500+ turn sessions and ~125MB per concurrent agent in swarm bursts. Whale session 9a990de8 launched 292 agents in 2 minutes and reached 36.8GB.

36.8GB 的内存使用主要来自消息数组的重复存储。`TEAMMATE_MESSAGES_UI_CAP` 就是为了解决这个问题——完整的对话历史存在于 Agent 的 `runAgent()` 本地变量和磁盘 transcript 中，AppState 只保留最近 50 条供 UI 显示。

### 3.5 DreamTask：后台"梦境"处理

**源文件**：`src/tasks/DreamTask/DreamTask.ts`

"Dream"（梦境）是一个很有意思的概念：当用户空闲时，系统在后台生成一个子 Agent 来整理记忆——回顾最近的会话、提取关键信息、更新记忆文件（MEMORY.md）、清理过时的条目。这个过程类似于人类在睡眠中整理记忆，因此得名"梦境"。

**状态扩展**：
```
DreamTaskState = TaskStateBase & {
  type: 'dream'
  phase: 'starting' | 'updating'  // 当前阶段
  sessionsReviewing: number    // 正在回顾的会话数
  filesTouched: string[]       // 修改过的文件路径
  turns: DreamTurn[]           // 对话轮次（最多 30 个）
  abortController?: AbortController
  priorMtime: number           // 整理锁的前一次修改时间
}
```

**阶段检测**。Dream 提示词有 4 阶段结构（orient/gather/consolidate/prune），但系统不解析这些阶段。它只检测两个状态：`'starting'`（初始阶段）和 `'updating'`（检测到第一个 Edit/Write tool_use 时切换）。这是一个务实的选择——与其构建脆弱的阶段解析器，不如只检测最重要的转变。

**`filesTouched` 的局限性**。注释中明确警告这是"INCOMPLETE reflection"——它只捕获通过 Edit/Write 工具调用修改的文件，bash 写入操作不在追踪范围内。这种诚实的文档是好的工程实践——下游代码知道不能依赖这个列表的完整性。

**杀死 Dream 时的锁回滚**。`priorMtime` 存储了整理锁的前一次修改时间。如果 Dream 任务被杀死（用户取消或出错），可以回滚锁状态，允许下一次空闲时重新触发。这通过 `rollbackConsolidationLock()` 实现。

**无模型通知**。Dream 完成时设置 `notified: true`，但**不**通过 `<task-notification>` 通知模型——它是纯 UI 的。用户在界面上看到 Dream 完成的提示，但主 Agent 不知道也不需要知道 Dream 的存在。

### 3.6 LocalWorkflowTask 和 MonitorMcpTask

这两个任务类型通过特性门控制（`WORKFLOW_SCRIPTS` 和 `MONITOR_TOOL`），可能尚未完全稳定。

**LocalWorkflowTask** 管理本地工作流脚本的执行。前缀为 `w`。

**MonitorMcpTask** 管理 MCP 监控任务。前缀为 `m`。它也有一个类似 `killShellTasksForAgent()` 的清理函数 `killMonitorMcpTasksForAgent()`，在 `runAgent()` 的 finally 块中调用。

## 4. 任务状态机

### 4.1 状态转换规则

```
pending ──start()──→ running ──complete()──→ completed
                       │
                       ├──error()──→ failed
                       │
                       └──cancel()──→ killed
```

`isTerminalTaskStatus()` 函数（`Task.ts` 第 27-29 行）定义了终态判断：

```typescript
function isTerminalTaskStatus(status: TaskStatus): boolean {
  return status === 'completed' || status === 'failed' || status === 'killed'
}
```

这个函数被用于：
- **防止给死任务注入消息**——`SendMessageTool` 在发送前检查目标任务是否已终止
- **触发任务回收**——`evictTerminalTask()` 只回收终态任务
- **清理逻辑**——关闭时只处理非终态任务

### 4.2 没有 "暂停" 状态

状态机中没有 `paused` 状态——这是一个有意的简化。暂停语义通过 `totalPausedMs` 字段和外部控制（AbortController 暂停、消息队列冻结）实现，而不是引入额外的状态转换。这减少了状态机的复杂度，避免了 `paused → running` 恢复路径中的 edge case。

### 4.3 幂等的终态转换

`killTask()`（`killShellTasks.ts` 第 16-46 行）中有一个关键的 guard：

```typescript
if (task.status !== 'running' || !isLocalShellTask(task)) {
  return task  // 返回相同引用，触发 no-op
}
```

如果任务已经不在 `running` 状态，kill 操作是 no-op。`updateTaskState()` 检测到相同引用时跳过 AppState 更新：

```typescript
if (updated === task) {
  return prev  // 引用相等 → 跳过 spread
}
```

这个"引用相等优化"贯穿了整个任务系统——避免在无变化时触发 React re-render。

## 5. AppState 全局状态注册

### 5.1 为什么所有任务都要注册

`registerTask()`（`framework.ts` 第 77-117 行）将新创建的任务写入 `AppState.tasks`：

```typescript
function registerTask(task: TaskState, setAppState: SetAppState): void {
  setAppState(prev => ({
    ...prev,
    tasks: { ...prev.tasks, [task.id]: task }
  }))
  // 发送 SDK 事件
  enqueueSdkEvent({ type: 'system', subtype: 'task_started', ... })
}
```

注册之后，这些信息被多个消费者使用：

1. **UI 组件**。后台任务指示器（`BackgroundTasksDialog.tsx`）遍历 `AppState.tasks` 显示所有活跃任务。`isBackgroundTask()` 过滤函数（`tasks/types.ts` 第 37-46 行）决定哪些任务出现在指示器中——必须是 `running` 或 `pending` 状态，且 `isBackgrounded !== false`
2. **模型查询**。模型可以通过 `TaskGet`、`TaskList`、`TaskOutput` 工具查询任务状态和输出
3. **通知系统**。`generateTaskAttachments()` 扫描所有任务，为有新输出的任务生成 attachment，注入到模型的下一轮输入中
4. **清理系统**。会话关闭时遍历 `AppState.tasks`，终止所有非终态任务
5. **SDK 事件**。通过 `enqueueSdkEvent()` 向外部 SDK 消费者发送 `task_started`、`task_terminated` 事件

### 5.2 替换注册（Resume）

`registerTask()` 有一个"替换注册"路径（第 80-97 行）——如果相同 ID 的任务已存在（Agent 恢复场景），会保留一些 UI 状态：

```typescript
const merged = existing && 'retain' in existing
  ? { ...task, retain: existing.retain, startTime: existing.startTime,
      messages: existing.messages, diskLoaded: existing.diskLoaded,
      pendingMessages: existing.pendingMessages }
  : task
```

这确保了 Agent 恢复时，用户已经打开的对话视图不会被重置。

### 5.3 任务回收

`evictTerminalTask()`（第 125-144 行）是主动回收路径——在任务达到终态且通知已发送后，从 `AppState.tasks` 中删除。`generateTaskAttachments()` 中的懒惰 GC 是安全网：

```typescript
case 'completed':
case 'failed':
case 'killed':
  evictedTaskIds.push(taskState.id)
  continue
```

面板显示的 Agent 任务有 `evictAfter` 保护——终态后 30 秒内（`PANEL_GRACE_MS`）不被回收，给用户查看结果的时间。已终止任务显示 3 秒后消失（`STOPPED_DISPLAY_MS`）。

## 6. 后台任务执行流

一个完整的后台 Agent 任务（最常见的后台任务类型）的生命周期分为以下步骤：

**步骤 1：AgentTool 决定后台运行**

模型调用 `AgentTool` 时设置 `run_in_background: true`，或 Agent 定义中 `background: true`，或系统在 Coordinator 模式下强制所有 Agent 后台运行。

**步骤 2：创建 LocalAgentTask 状态**

`registerAsyncAgent()` 创建 `LocalAgentTaskState`，设置 `status: 'pending'`，通过 `registerTask()` 写入 AppState。

**步骤 3：创建独立 AbortController**

后台 Agent 获得与父级**不关联**的 AbortController。这确保了：
- 父 Agent 被取消不影响后台 Agent
- 后台 Agent 可以被独立取消（通过 `TaskStopTool`）

**步骤 4：启动 runAgent() 异步循环**

在独立的 `runWithAgentContext()` 异步上下文中启动。`agentContext` 通过 `AsyncLocalStorage` 传播，所有在这个异步链中的操作都能通过 `getAgentContext()` 获取当前 Agent 的身份信息。

**步骤 5：状态转为 running**

`runAgent()` 开始执行后，任务状态更新为 `running`。

**步骤 6：进度更新**

每次 `runAgent()` yield 一条消息，`updateProgressFromMessage()` 更新进度信息——工具调用计数、token 消耗、最近活动。这些进度信息通过 `generateTaskAttachments()` 定期推送给父 Agent。

**步骤 7：父级轮询（按需）**

父 Agent 可以通过 `TaskOutput` 工具主动查询后台任务的最新输出（从磁盘文件读取增量）。`TaskGet` 查询状态，`TaskList` 列出所有任务。

**步骤 8：完成/失败/终止**

`runAgent()` 正常结束 → `completeAsyncAgent()` → 状态转为 `completed`
`runAgent()` 抛异常 → `failAsyncAgent()` → 状态转为 `failed`
用户调用 `TaskStop` → `killAsyncAgent()` → `abortController.abort()` → 状态转为 `killed`

**步骤 9：通知入队**

`enqueueAgentNotification()` 构建 `<task-notification>` XML，放入 `pendingNotifications` 队列。同时调用 `abortSpeculation()` 取消任何基于旧状态的推测性响应。

**步骤 10：通知注入和回收**

父 Agent 在下一次工具轮次边界取出通知，作为 user 消息注入对话。任务在通知发送后被标记为可回收，在下一次 `generateTaskAttachments()` 扫描或 `evictTerminalTask()` 调用时从 AppState 移除。

## 7. 任务管理工具

系统提供五个工具让模型管理后台任务：

**TaskCreate**（`TaskCreateTool`）：直接创建后台 bash 任务。**TaskGet**（`TaskGetTool`）：查询单个任务的状态、进度和最近活动。**TaskOutput**（`TaskOutputTool`）：通过 `outputOffset` 增量读取任务输出，每次上限 8MB。**TaskStop**（`TaskStopTool`）：终止运行中的任务。**TaskList**（`TaskListTool`）：列出所有活跃任务。

这五个工具的权限都是非破坏性的——它们不修改代码，不会引发安全风险，所以不需要用户权限确认。

## 8. 僵尸防护（Zombie Protection）

"僵尸"是指 Agent 或进程已经不再需要，但没有被正确清理，继续消耗资源的情况。在长时间运行的会话中（Whale Session，注释中提到的一个 session 生成了 292 个 Agent），僵尸累积会导致严重的资源耗尽。

### 8.1 killShellTasksForAgent 的实现

`killShellTasksForAgent()`（`killShellTasks.ts` 第 53-76 行）是最具体的僵尸防护机制。

逻辑很直白：
1. 遍历 `AppState.tasks` 中的所有条目
2. 筛选条件：是 `LocalShellTask` + `agentId` 匹配 + 状态为 `running`
3. 对每个匹配的任务调用 `killTask()`
4. 清除该 Agent 的所有排队通知：`dequeueAllMatching(cmd => cmd.agentId === agentId)`

最后一步很容易被忽略但很重要——`killTask` 会异步地生成 `'killed'` 通知，但此时 Agent 的查询循环已经退出了，没有消费者会处理这些通知。`dequeueAllMatching()` 把已排队的和即将排队的通知都清除掉。

### 8.2 其他僵尸防护机制

**超时强制**。后台 bash 任务可以设置 `timeout`，超时后被 watchdog 终止。

**Agent 级别清理**。`runAgent()` 的 `finally` 块（第 816-858 行）是一个全面的清理清单——MCP 连接、session hooks、Perfetto trace、todo 条目、文件状态缓存、初始消息数组——每一项都有对应的释放操作。`initialMessages.length = 0` 这行看似奇怪，但它释放了 fork 上下文消息数组占用的内存（可能很大，因为包含了完整的父级对话历史）。

**进程退出时的全局清理**。`registerCleanup()` 在进程退出时终止所有未完成的任务。**输出文件回收**。`evictTaskOutput()` 在任务终止或回收时删除磁盘输出文件。

## 9. AsyncLocalStorage 隔离

### 9.1 为什么 AsyncLocalStorage 适合多 Agent 场景

[`AsyncLocalStorage`](https://nodejs.org/api/async_context.html) 💡 *Node.js 提供的异步上下文传播 API，可以在异步调用链中自动携带上下文数据，无需手动传参* 是 Node.js `async_hooks` 模块提供的一个 API，它解决了"在异步操作中追踪请求上下文"的问题。

在多 Agent 场景中，同一个进程中可能同时运行多个 Agent（后台 Agent、Swarm 队友、Dream 任务）。每个 Agent 需要知道"我是谁"——用于日志记录、分析事件归因、权限检查等。

**手动传参方案的问题**：

```typescript
// 方案 A：每个函数都传 agentId
function readFile(path: string, agentId: string) { ... }
function logEvent(event: string, agentId: string) { ... }
function checkPermission(tool: string, agentId: string) { ... }
```

这种方案要求每个被 Agent 调用的函数都增加 `agentId` 参数——也就是"参数穿透"（Parameter Drilling）。在一个有几百个函数的系统中，这意味着大量的签名修改，而且容易遗漏（某个函数忘了传参，运行时才发现）。

**AsyncLocalStorage 方案**：

```typescript
// 设置上下文（一次性）
runWithAgentContext({ agentId, agentType: 'subagent', ... }, async () => {
  // 这个闭包内的所有异步操作自动携带上下文
  await readFile(path)    // readFile 内部调用 getAgentContext() 就能拿到 agentId
  await logEvent(event)   // 同理
  await checkPermission(tool)  // 同理
})
```

上下文在 `runWithAgentContext()` 调用时设置，之后**自动流过**整个异步调用链——包括 `await`、`Promise.then()`、`setTimeout()` 等。不需要修改任何被调用函数的签名。

### 9.2 agentContext 和 teammateContext 的区别

系统维护两个独立的 `AsyncLocalStorage` 实例：

**`agentContext`**（`src/utils/agentContext.ts`）：用于**分析归因**（Analytics Attribution）。它有两种子类型：

- `SubagentContext`：普通子 Agent（通过 AgentTool 生成），包含 `agentId`、`subagentName`、`isBuiltIn` 等
- `TeammateAgentContext`：Swarm 队友，包含 `agentId`、`agentName`、`teamName`、`agentColor`、`planModeRequired` 等

注释中明确解释了为什么用 AsyncLocalStorage 而非 AppState：

> When agents are backgrounded (ctrl+b), multiple agents can run concurrently in the same process. AppState is a single shared state that would be overwritten, causing Agent A's events to incorrectly use Agent B's context. AsyncLocalStorage isolates each async execution chain, so concurrent agents don't interfere with each other.

**`teammateContext`**（`src/utils/teammateContext.ts`）：用于**运行时身份**。只有 `InProcessTeammate` 使用这个上下文。它包含 `agentId`、`agentName`、`teamName`、`color`、`abortController` 等。

关键区别在于：
- `agentContext` 服务于分析和日志——"谁产生了这个 API 调用"
- `teammateContext` 服务于业务逻辑——"我在哪个团队，我的名字是什么，我应该用什么颜色显示"

`teammate.ts` 中的辅助函数（`getAgentId()`、`getAgentName()` 等）按优先级查找：先检查 `AsyncLocalStorage`（同进程队友），再检查动态注册的 `dynamicTeamContext`（运行时加入的队友），最后检查环境变量（tmux/iTerm2 启动的独立进程队友）。

### 9.3 第三个 AsyncLocalStorage：workloadContext

`workloadContext.ts` 存储一个 `workload` 标签（目前只有 `'cron'`），标记当前执行链是否由 cron 定时任务触发。不能用全局变量替代——后台 Agent 的 `async` 函数在第一个 `await` 处让出控制权，父级的同步代码会在 Agent 恢复前覆盖全局变量。`AsyncLocalStorage` 在 `runWithWorkload()` 调用时创建快照，不受后续覆盖影响。

### 9.4 实际使用中的注意事项

**上下文泄漏**。`workloadContext.ts` 中记录了一个真实案例：cron 执行的 ALS 上下文通过 `queryGuard.end() → _notify() → React subscriber → scheduled re-render` 路径泄漏到后续非 cron 执行。修复方式是始终创建新的 ALS 边界，即使 `workload` 为 `undefined`。

**不要在 ALS 回调外访问上下文**。定时器、事件监听器等如果不在原始 `run()` 闭包内，无法访问上下文。

**性能开销**。`AsyncLocalStorage` 基于 V8 的 `PromiseHooks`，每次 `async` 操作有微小开销，对 Agent 系统的工作负载可忽略。

## 10. 持久化和崩溃恢复

### 10.1 任务状态存储在哪里

任务状态主要存在两个地方：

**内存中的 AppState**。`AppState.tasks` 是所有任务状态的权威来源。React 组件通过订阅 AppState 变化来更新 UI。但 AppState 是纯内存结构——进程退出就没了。

**磁盘上的 transcript 和输出文件**。
- Agent 任务的对话历史通过 `recordSidechainTranscript()` 写入 `subagents/` 目录下的 JSONL 文件
- 任务输出通过 `TaskOutput` 系统写入 `tasks/` 目录下的文件
- Agent 元数据（`agentType`、`worktreePath`、`description`）通过 `writeAgentMetadata()` 持久化
- 远程 Agent 的元数据通过 `writeRemoteAgentMetadata()` 存储，支持跨会话恢复

### 10.2 恢复机制

**Agent 恢复**。Agent 任务可以通过 `resumeAgentBackground()`（`resumeAgent.ts`）从 sidechain transcript 恢复。它读取 JSONL 文件重建消息历史，重新启动 `runAgent()` 查询循环。`registerTask()` 的替换逻辑保留 UI 状态。

**远程 Agent 恢复**。`listRemoteAgentMetadata()` 扫描持久化的元数据，`pollRemoteSessionEvents()` 重新连接。`pollStartedAt` 记录本次轮询开始时间，防止恢复后立即超时。

**Session 模式恢复**。`matchSessionMode()` 检查会话模式，必要时翻转环境变量。

### 10.3 不完美的恢复

诚实地说，当前的恢复能力是有限的：

- **Shell 任务不可恢复**。进程退出后 bash 进程也退出
- **AbortController 丢失**。恢复的 Agent 获得新的 AbortController
- **InProcessTeammate 不跨会话恢复**。Swarm 的完整状态太复杂
- **Dream 任务不跨会话恢复**。失败后回滚锁状态，下次空闲时重新触发

这些限制反映了一个工程权衡：完美的恢复需要完整的状态序列化（运行时引用、AbortController、进程句柄），成本极高。当前的策略是"能恢复的恢复，不能恢复的优雅终止"——通过 finally 块确保资源被清理，通过 `isTerminalTaskStatus()` 确保不会操作已死的任务。

## 11. 设计要点

**1. Task 接口极简——只有 kill**

`Task` 接口不包含 `start()`、`getProgress()`、`render()` 等方法。每种任务类型自行管理创建和渲染。Shell 任务和 Agent 任务的创建流程差异太大，强制统一接口只会增加抽象成本。

**2. 输出走磁盘，不走内存**

任务输出写入磁盘文件，AppState 只存储偏移量。这让系统可以处理 5GB 的 bash 输出而不 OOM。代价是读取输出需要磁盘 I/O，但通过增量读取和缓存友好的顺序访问模式，影响很小。

**3. 通知的"最多一次"语义**

`notified` 标志 + 原子性检查确保每个任务只发送一次完成通知。这比"至少一次"更简单——不需要去重逻辑。如果通知丢失，模型可以通过 `TaskGet` 主动查询。

**4. 引用相等优化无处不在**

`updateTaskState()` 中的 `if (updated === task) return prev` 模式避免了无变化时的 React re-render。在几十个并发任务、每秒多次状态更新的场景下，这直接影响 UI 流畅度。

**5. AsyncLocalStorage 解耦了"谁在执行"和"执行什么"**

分析归因（Agent A 产生了这个 API 调用）和业务逻辑（读取这个文件）被彻底解耦。添加新的分析维度不需要修改工具代码。

**6. 僵尸防护是运行时契约，不是可选优化**

任何生成后台工作的代码都必须在 finally 块中清理。Whale Session 的 36.8GB 内存使用证明了不做清理的后果。
