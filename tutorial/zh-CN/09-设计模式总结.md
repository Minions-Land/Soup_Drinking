# 第九章：12 个可复用设计模式 -- 你的智能体工程手册

> 上一章：[第八章：远程连接](08-远程连接.md)

这一章把前八章分散在各处的工程决策提炼成 12 个独立的设计模式。每个模式都经过 Claude Code 这样一个 1332 个 TypeScript 文件、30+ 工具、6 种内置智能体的生产系统验证。

不是所有模式都适合所有项目。每个模式条目都包含"适用条件"和"不适用条件"，帮你判断该不该用。

---

## 模式 1：工具接口 + 安全默认值

### 问题场景

你的智能体系统有 30 个工具。每个工具都要处理输入验证（Input Validation）、权限检查（Permission Check）、结果截断（Result Truncation）、UI 渲染（UI Rendering）、并发安全声明（Concurrency Safety Declaration）。如果让每个工具作者自己写这些逻辑，两周内就会出现：

- 有人忘了做权限检查，工具默认能写任意文件
- 有人返回了 500KB 的结果字符串，撑爆上下文窗口
- 工具之间的错误格式不统一，模型无法从错误中学习

### Claude Code 的解法

`src/Tool.ts` 定义了统一的 `Tool<Input, Output, Progress>` 泛型接口。`buildTool()` 工厂函数为所有方法填充安全默认值，工具作者只覆盖自己关心的部分。

```typescript
// src/Tool.ts — buildTool 的核心思路
const MyTool = buildTool({
  name: 'MyTool',
  inputSchema: z.object({ ... }),
  maxResultSizeChars: 100_000,
  call: async (args, ctx) => ({ data: result }),
  prompt: async () => "Instructions for the model...",
  // 以下全部自动填充安全默认值：
  // isReadOnly: false（假定危险）
  // isConcurrencySafe: false（假定不安全）
  // checkPermissions: 默认 ask（询问用户）
  // isEnabled: true
})
```

### 核心原理

关键词是 **Fail-Closed**（失败时关闭）。💡 Fail-Closed 指系统遇到未知状态时默认拒绝操作，与 Fail-Open（默认允许）相反。

工程权衡：安全默认值意味着新工具"开箱即用"时功能受限——它不能写文件、不能并发运行。这增加了工具作者的工作量（需要显式声明 `isReadOnly: true`），但消除了一整类安全漏洞。在智能体系统中，模型会以你没预料到的方式调用工具。你无法用测试覆盖所有调用路径，所以默认值必须是安全的。

### 适用条件

- 工具数量超过 10 个，且有多人协作开发
- 系统中有读写混合的工具（纯只读系统可以简化）
- 你希望新开发者写工具时不需要理解完整的安全模型

### 不适用条件

- 工具数量少于 5 个，且都是你一个人写的——直接在每个工具里写逻辑更清晰
- 所有工具都是纯计算，没有任何副作用（Side Effect）

### 实现要点

1. **接口设计先行**。先定义 `Tool` 接口的全部字段，然后才实现 `buildTool()`。接口一旦发布就很难改，默认值可以随时调整。
2. **默认值文档化**。在 `buildTool()` 的 JSDoc 中明确写出每个字段的默认值。工具作者不应该需要读源码才知道默认行为。
3. **`maxResultSizeChars` 是必填字段**。不设上限的工具迟早会返回一个巨大结果，撑爆上下文窗口。Claude Code 用 `100_000` 字符作为默认上限，超过时自动持久化到文件并返回摘要。

### 反模式

- **"超级工具"**：一个工具做 5 件事（读文件、写文件、删文件、列目录），用 `action` 参数区分。这让权限检查变得极其复杂，因为 `isReadOnly` 取决于参数值。Claude Code 的做法是 `ReadFile` 和 `WriteFile` 分开。
- **默认 allow**：`checkPermissions` 默认返回 `allow` 而不是 `ask`。看起来省事，但意味着任何忘记配置权限的工具都自动获得完全写入权限。

## 模式 2：上下文 Fork（Prompt Cache 共享）

### 问题场景

你的协调者智能体（Coordinator Agent）已经和用户对话了 20 轮，积累了大量上下文：读过哪些文件、理解了什么架构、用户表达了什么偏好。现在需要生成一个子智能体去做深入研究。

问题是：如果子智能体从空白开始，它要重新读一遍所有文件，重新理解架构。如果把父级的完整对话复制给子级，API 调用的 Token 费用会翻倍——因为 LLM 提供商（如 Anthropic）的 Prompt Cache 是基于字节前缀匹配（Byte-Prefix Matching）的，即使内容相同，字节序列不同就无法命中缓存。

### Claude Code 的解法

`src/tools/AgentTool/forkSubagent.ts` 实现了上下文分叉。核心思路：子智能体复制父级的 `renderedSystemPrompt`（渲染后的系统提示词）和对话前缀的**精确字节副本**，从而复用 LLM 的 Prompt Cache，实现零重新分词（Zero Re-tokenization）成本。

```
父级上下文：
  [System Prompt][Msg1][Msg2][Msg3]
                  ↓ 在 Msg2 处分叉
子级上下文：
  [System Prompt][Msg1][Msg2][新任务指令]
  ↑ 共享 Prompt Cache — 零重新分词
```

### 核心原理

💡 **Prompt Cache** 是 LLM API 提供商的一项优化：如果两次请求的输入前缀完全相同（字节级），提供商可以跳过对前缀部分的重新分词和计算，大幅降低延迟和费用。Anthropic API 的 Prompt Cache 要求前缀严格字节匹配。

工程权衡：字节精确复制意味着子智能体**必须使用和父级相同的模型**（不同模型的系统提示格式不同，无法匹配缓存）。这是一个硬约束。如果你需要子智能体用更便宜的模型，就不能用 Fork 模式——应该用普通的后台生成（Background Spawn），接受缓存不命中的成本。

### 适用条件

- 子智能体的任务建立在父级已有的研究成果之上
- 父级上下文较大（超过 10K Token），重新获取成本高
- 父子使用相同模型

### 不适用条件

- 子智能体的任务与父级上下文无关——复制上下文是纯浪费
- 需要使用更便宜/更快的模型做子任务
- 子任务很简单，不需要父级的积累

### 实现要点

1. **复制 `renderedSystemPrompt`**，不是重新渲染。重新渲染可能因为时间戳、随机值等导致字节不同，缓存失效。
2. **克隆 `contentReplacementState`**（内容替换状态）。这是 Claude Code 用来追踪哪些文件内容已经发送给模型的状态。不克隆会导致子级重复发送文件。
3. **创建独立的 `AbortController`**。子级必须可以被独立取消，不影响父级。这是隔离的底线。

### 反模式

- **共享 AbortController**：父级取消时子级也被取消，看起来合理——但如果子级正在写文件，中途取消会留下半写的文件。每个智能体需要独立的取消控制。
- **在 Fork 后修改系统提示词**：任何修改都会破坏 Prompt Cache 的字节匹配，让 Fork 的核心优势归零。

## 模式 3：扇出 → 综合 → 实现 → 验证

### 问题场景

用户说："把我们的认证系统从 Session-Based 迁移到 JWT。" 这个任务涉及后端路由、前端存储、中间件、数据库 Schema、测试——一个智能体按顺序做所有事情，要么太慢，要么在处理后端时已经忘了前端的细节。

### Claude Code 的解法

`src/coordinator/coordinatorMode.ts` 实现了四阶段协调器工作流（Four-Phase Coordinator Workflow）。

**阶段 1：研究（扇出 / Fan-Out）**
协调者并行生成多个研究智能体，每个只回答一个具体问题。

```
协调者 →
  研究员 A："后端有哪些文件用了 session？列出文件路径和用法"
  研究员 B："前端哪些组件依赖 cookie-based 认证？"
  研究员 C："现有的测试覆盖了哪些认证场景？"
```

关键：研究员**只收集事实，不提建议**。"调查认证系统"太模糊——"列出 `src/auth/` 下所有读取 `req.session` 的文件路径"才是好的研究指令。

**阶段 2：综合（Synthesis）**
所有研究通知到达后，协调者**自己**分析结果，写出逐文件的实现计划。综合不能委托给子智能体——这是协调者存在的核心价值。

**阶段 3：实现（Implementation）**
根据计划执行。小改动（< 5 文件）协调者直接做。大改动委托给专门的实现智能体，但要给出**精确的指令**：改哪个文件、改什么、遵循什么模式。

**阶段 4：验证（Verification）**
一个**全新的**智能体——没有参与过研究或实现——审查变更、运行测试、检查回归（Regression）。

### 核心原理

验证智能体必须是全新的，这是整个模式最重要的洞察。做了实现的智能体会倾向于相信自己的工作是正确的——它"知道"自己的意图，所以会忽略实际的问题。心理学上叫确认偏差（Confirmation Bias）。全新的智能体没有这个包袱，它会真正运行测试、真正搜索问题。

### 适用条件

- 任务涉及多个子系统（横切变更 / Cross-Cutting Change）
- 任务复杂到需要先理解再行动
- 对正确性要求高（不能靠"大概对了"）

### 不适用条件

- 简单的单文件修改——四个阶段的开销远超收益
- 任务已经有明确的实现计划——跳过研究和综合，直接实现 + 验证
- 时间敏感的快速修复——四阶段太慢

### 实现要点

1. **研究指令要具体**。不是"调查 X"，而是"在 `src/` 目录下找到所有导入了 `Y` 的文件，列出文件路径和具体的 import 语句"。
2. **在一个协调者轮次中生成所有研究智能体**。不要一个一个生成——这是不必要的串行化。
3. **计划写入草稿板（Scratchpad）文件**。不要只放在对话上下文里——文件可以被后续阶段的智能体独立读取。

### 反模式

- **综合委托**："根据你的研究结果修复它"——研究员没有全局视角，它的"修复"会是局部最优。
- **验证者继承上下文**：让实现智能体"顺便"验证自己的工作。这等于没有验证。
- **在同一文件上并行实现**：两个智能体同时修改 `auth.ts`，后写的覆盖先写的。涉及重叠文件时必须串行。

## 模式 4：两阶段权限验证

### 问题场景

用户让智能体"删除 `tmp/` 下所有 `.log` 文件"。你需要检查两件事：
1. `tmp/` 目录存在吗？有 `.log` 文件吗？（逻辑验证）
2. 用户允许删除文件吗？配置规则怎么说？（授权检查）

很多系统把这两件事混在一起。结果是：用户看到"权限被拒绝"的提示，但实际上目录根本不存在——错误消息误导了用户。

### Claude Code 的解法

`src/Tool.ts` 的工具生命周期将验证拆成两个阶段：

**阶段 1：`validateInput(input, context)`** — 纯逻辑检查。文件存在吗？正则表达式合法吗？参数在有效范围内吗？返回 `{ result: true }` 或 `{ result: false, message: "文件不存在: /tmp/xxx" }`。

**阶段 2：`checkPermissions(input, context)`** — 纯授权检查。用户的权限模式（Permission Mode）允许这个操作吗？有匹配的 Always-Allow 规则吗？返回三种行为之一：
- `allow`：预先批准，立即执行
- `ask`：需要用户明确批准
- `deny`：静默阻止

### 核心原理

有效性（Validity）是客观的——文件存在或不存在，这是事实。授权（Authorization）是策略——用户配置决定允许或拒绝。分离它们有三个好处：

1. **错误消息更准确**。"文件不存在"和"权限被拒绝"是完全不同的信息。
2. **避免不必要的权限提示**。如果输入本身就无效，问用户"你允许这个操作吗"毫无意义——用户允许了也执行不了。先验证，验证通过了再检查权限。
3. **两个检查可以独立演化**。输入验证逻辑通常工具特定的，权限检查逻辑通常是全局策略。

### 适用条件

- 系统有读写混合的工具
- 需要给用户提供精确的错误信息
- 权限策略可能经常变化（不想每次改策略都修改验证逻辑）

### 不适用条件

- 所有工具都是只读的——不需要权限检查
- 系统运行在完全受信任的环境中（CI/CD bypassPermissions 模式）

### 实现要点

1. **`validateInput` 在 `checkPermissions` 之前运行**。顺序不可颠倒。
2. **`checkPermissions` 的规则评估有优先级**：会话规则（Session Rules）> 项目规则（`.claude/settings.json`）> 全局规则（`~/.claude/settings.json`）> 权限模式默认值 > 回退到 `ask`。
3. **破坏性操作（Destructive Operations）绕过所有规则**。`rm -rf`、`git push --force`、`DROP TABLE` 等命令，无论权限模式如何，始终要求用户明确批准。在后台智能体（无终端）中自动拒绝。

### 反模式

- **在 `call()` 里做验证**：验证失败时工具已经在"执行中"状态，UI 显示错误信息的上下文不对。
- **`checkPermissions` 返回布尔值**：`true/false` 不够——你还需要第三种状态 `ask`（需要用户决定）。

## 模式 5：AsyncLocalStorage 隔离

### 问题场景

你的 Node.js 进程里有 5 个智能体并发运行。每个智能体有自己的状态：当前工作目录、权限上下文、文件读取缓存、API 配额计数器。你需要在任意深度的函数调用中访问"当前智能体"的状态，但不想在每个函数签名里加一个 `agentContext` 参数——因为调用链可能有 20 层深，中间大部分函数根本不关心这个参数。

### Claude Code 的解法

利用 Node.js 内置的 `AsyncLocalStorage`（异步本地存储），创建自动流经异步调用链的每智能体上下文。

💡 **AsyncLocalStorage** 是 Node.js `async_hooks` 模块提供的 API。它的原理类似于 Java 的 `ThreadLocal`，但针对 Node.js 的异步模型：每次 `run()` 调用创建一个新的存储范围（Store Scope），在这个范围内发起的所有异步操作（`await`、`setTimeout`、`Promise`）都能通过 `getStore()` 访问到同一个存储对象，且不同范围之间互不干扰。

```typescript
// 概念示意
const agentContext = new AsyncLocalStorage<AgentState>()

// 每个智能体在自己的上下文中运行
agentContext.run(agentState, async () => {
  // 调用链中任意深度都能获取当前智能体的状态
  const state = agentContext.getStore()
  await doWork()  // doWork 内部也能通过 getStore() 访问
})
```

在 Claude Code 中，`src/utils/agentContext.ts` 管理智能体级别的隔离，每个通过 `AgentTool` 或 `inProcessRunner.ts` 启动的智能体都在自己的 `AsyncLocalStorage.run()` 范围内运行。

### 核心原理

这和 Express/Koa 用 `AsyncLocalStorage` 管理请求作用域数据（Request-Scoped Data）是同一个机制。区别是：Express 用它存请求 ID 和用户信息，Claude Code 用它存智能体的完整运行状态。

工程权衡：消除了参数传递的负担，但代码变得**隐式**了——函数的行为依赖于它被调用时的 AsyncLocalStorage 上下文，而不是显式传入的参数。这让单元测试更难写（需要手动 `run()` 来设置上下文），也让调试时的数据流更难追踪。

### 适用条件

- 进程内多智能体并发（不是多进程，不是多线程）
- 调用链深度 > 5 层，中间层不需要感知上下文
- Node.js 运行时（浏览器没有 `AsyncLocalStorage`）

### 不适用条件

- 每个智能体是独立进程——进程本身就是隔离边界，不需要 AsyncLocalStorage
- 调用链短且扁平——显式参数传递更清晰
- 你需要在非 Node.js 环境运行（Python 有 `contextvars`，概念类似但 API 不同）

### 实现要点

1. **嵌套上下文**。内层 `run()` 会遮蔽（Shadow）外层上下文。如果子智能体在父智能体的 `run()` 范围内再调用 `run()`，子智能体的 `getStore()` 返回子上下文，不是父上下文。这是正确的行为——不需要手动清理。
2. **不要在 `run()` 外调用 `getStore()`**。它会返回 `undefined`。加类型守卫（Type Guard）或在应用启动时设置一个默认上下文。
3. **性能开销可忽略**。`AsyncLocalStorage` 基于 Node.js 的 `executionAsyncResource`，不是 `async_hooks` 的 `init/before/after/destroy`。在 Node.js 16+ 上开销极低。

### 反模式

- **把 AsyncLocalStorage 当全局变量用**：存放不属于当前智能体的状态（如应用级配置）。应用级状态用模块级变量或单例。
- **跨 `run()` 边界共享可变对象**：父子智能体的 Store 里引用同一个可变对象，修改时互相干扰。Store 里的对象应该是每个智能体独立拥有的。

## 模式 6：渐进式结果交付

### 问题场景

用户让智能体"搜索整个代码库中所有用到了废弃 API 的地方"。这个操作需要 30 秒。在这 30 秒里，用户看到的 UI 是一片空白或一个旋转的加载图标。用户开始怀疑：它在干什么？卡了吗？要不要中断？

### Claude Code 的解法

`src/Tool.ts` 的 `call()` 方法接受一个 `onProgress` 回调参数，类型为 `ToolCallProgress<P>`。工具在执行过程中通过这个回调报告增量更新，UI 层订阅并渐进渲染。

```typescript
// 工具实现中使用 onProgress
async call(args, ctx, canUse, msg, onProgress) {
  onProgress?.({ toolUseID: id, data: { type: 'search', status: 'scanning src/' } })
  // ... 扫描 src/ ...
  onProgress?.({ toolUseID: id, data: { type: 'search', found: 23, status: 'scanning lib/' } })
  // ... 扫描 lib/ ...
  return { data: finalResult }
}
```

在 Claude Code 的终端 UI 中，`BashTool` 和 `GrepTool` 都实现了渐进式输出——`BashTool` 实时流式显示命令的 stdout，`GrepTool` 在扫描过程中逐步显示匹配数量。

### 核心原理

进度更新有**双重作用**：

1. **对 UI**：表明系统在正常工作，用户不需要中断。
2. **对模型**：携带部分结果。在某些架构中，模型可以在工具完成前看到中间结果，实现早期路线修正（Early Course Correction）——如果搜索方向明显错误，可以提前中止而不是等 30 秒后才发现白搜了。

工程权衡：进度回调增加了工具实现的复杂度。一个纯函数 `input → output` 变成了有副作用的过程。测试也更复杂——需要验证进度事件的顺序和内容。对于执行时间 < 2 秒的工具，这个复杂度不值得。

### 适用条件

- 工具执行时间 > 2 秒
- 工具有可度量的中间状态（扫描了多少文件、处理了多少行）
- UI 需要给用户实时反馈

### 不适用条件

- 亚秒级完成的工具
- 工具是原子操作，没有有意义的中间状态
- 无 UI 的纯 API 场景（headless 模式通常不需要视觉进度）

### 实现要点

1. **进度更新不要太频繁**。每秒 2-5 次足够。太频繁会给 UI 渲染造成压力，也会产生大量无用的事件对象。
2. **进度数据要类型化**。用 `ToolProgress<P>` 泛型，每种工具定义自己的进度数据类型。不要用 `string` 或 `any`。
3. **`onProgress` 可能是 `undefined`**。用 `?.` 可选链调用。工具不应该因为没有进度监听器就崩溃。

### 反模式

- **进度更新包含完整结果**：每次进度事件都携带"到目前为止的所有结果"。这会导致 O(n²) 的数据传输。只发增量。
- **把进度百分比硬编码**：`onProgress({ percent: 50 })` 但实际工作量无法预测。宁可报告已完成的绝对数量，不要报告百分比——用户看到"已扫描 150 个文件"比"50%"更有信息量。

## 模式 7：延迟工具加载

### 问题场景

你的系统有 35 个工具。每个工具的 JSON Schema 大约 200-500 个 Token。如果在系统提示词（System Prompt）中包含所有工具的 Schema，仅工具描述就占掉 7,000-15,000 个 Token。但实际上，一个典型对话只会用到 8-12 个工具。那些 NotebookEdit、MemoryTool、ToolSearch 等低频工具在 90% 的对话里一次都不会被调用。

### Claude Code 的解法

在 `src/Tool.ts` 中，工具可以声明 `shouldDefer: true`。被延迟的工具不会出现在初始系统提示词中。系统保留一个轻量的 `ToolSearchTool`——模型需要某种能力时，调用 `ToolSearchTool("jupyter notebook")`，系统根据 `searchHint` 关键词匹配到 `NotebookEditTool`，将其 Schema 加载到下一轮对话中。

```
第 1 轮：模型有 15 个核心工具 + ToolSearchTool
第 2 轮：模型调用 ToolSearchTool("jupyter notebook")
  → 匹配到 NotebookEditTool（已延迟）
  → Schema 加载到第 3 轮
第 3 轮：模型可以使用 NotebookEditTool
```

### 核心原理

这是一个经典的**按需加载**（Lazy Loading）策略，从 Web 开发中的代码分割（Code Splitting）借鉴而来。核心权衡：

- **节省的**：在不需要该工具的每一轮，节省 200-500 Token 的 Schema 空间
- **付出的**：需要该工具时，多花一轮对话来发现和加载

在 Claude Code 中，大约 15 个工具被标记为 `shouldDefer`，节省了约 5,000-8,000 Token 的系统提示空间。考虑到绝大多数对话不会用到这些工具，这是一个高杠杆的优化。

### 适用条件

- 工具数量 > 20
- 系统提示词空间紧张（模型上下文窗口有限）
- 部分工具使用频率明显低于其他工具

### 不适用条件

- 工具数量 < 15——全部放入系统提示没有明显负担
- 所有工具使用频率相近——没有明确的"核心工具"和"低频工具"之分
- 对延迟敏感——每次发现延迟工具都要多花一轮对话

### 实现要点

1. **`searchHint` 要覆盖用户可能的描述方式**。`NotebookEditTool` 的 hint 应该包含 "jupyter", "notebook", "ipynb", "cell" 等关键词。用户不会用工具的内部名称来描述需求。
2. **核心工具不能延迟**。`ReadFile`、`WriteFile`、`Bash`、`Grep` 这些几乎每次对话都用的工具必须在初始提示中。延迟加载一个核心工具会导致模型在第一轮就调用 ToolSearch，反而浪费了一轮。
3. **ToolSearchTool 本身不能被延迟**。如果用来搜索工具的工具本身需要被搜索才能发现，就陷入了鸡生蛋的死循环。

### 反模式

- **延迟太多工具**：把 25 个工具延迟，只保留 5 个核心工具。模型会频繁调用 ToolSearch，每次浪费一轮，总成本可能比全部加载还高。
- **`searchHint` 和 `prompt()` 内容重复**：hint 应该是**搜索关键词**，prompt 应该是**使用说明**。它们的受众不同。

## 模式 8：XML 异步通知

### 问题场景

协调者智能体生成了 3 个后台研究智能体。这些智能体在不可预测的时间完成——可能 5 秒，可能 2 分钟。主对话需要在结果到达时被告知，但你不能让主对话"轮询"后台状态（那样主对话就被阻塞了），也不能用回调函数（LLM 不理解回调）。

### Claude Code 的解法

当后台智能体完成时，系统将结果打包为 XML 标签，注入到主对话的消息流中。模型在下一轮看到这些通知，自然地据此行动。

```xml
<task-notification task_id="abc123" status="completed">
  Agent task "Research API endpoints" completed.
  Found 12 endpoints in src/routes/, 3 of which use deprecated auth middleware.
</task-notification>
```

Claude Code 的实现在 `src/tasks/` 中，`LocalAgentTask` 完成时触发通知注入到父级对话。

### 核心原理

XML 格式的选择不是随意的。XML 有一个独特的特性：它**同时可被代码解析和被模型阅读**。

- 代码可以通过标签名和属性做路由和状态跟踪（`task_id`、`status`）
- 模型可以理解标签内的自然语言内容（"Found 12 endpoints..."）
- 不需要在"机器可读格式"和"模型可读格式"之间做转换

JSON 也能被代码解析，但对模型来说 JSON 的嵌套结构比 XML 的标签结构更难推理。自然语言消息能被模型理解，但代码无法可靠地从自然语言中提取结构化信息。XML 是两者的交集。

### 适用条件

- 多智能体系统中的异步通信
- 后台任务在不可预测的时间完成
- 需要在对话上下文中保留结构化的元数据

### 不适用条件

- 同步工具调用——直接返回结果即可
- 纯程序间通信（不涉及 LLM）——JSON 或 Protobuf 更高效
- 通知内容是大型二进制数据——写入文件，通知中只放文件路径

### 实现要点

1. **通知注入的时机**。在主对话的下一轮开始前注入，不是在消息流的中间。中间注入会打断模型当前的推理。
2. **包含足够的上下文**。通知不只是"完成了"——要包含摘要性的发现，让模型不需要再调用工具就能理解结果。
3. **状态属性用于代码路由**。`status="completed"` 让代码知道任务完成了，可以更新 UI。标签内的文本让模型知道结果是什么。

### 反模式

- **通知过于冗长**：把后台智能体的完整对话历史塞进通知。通知应该是摘要，详细内容写入草稿板文件。
- **没有 task_id**：多个后台任务同时完成时，模型无法区分哪个通知对应哪个任务。

## 模式 9：共享工作队列（Swarm）

### 问题场景

你需要对一个项目中的 200 个测试文件运行 lint 检查并修复问题。一个智能体按顺序处理需要很长时间。如果用协调者模式，协调者需要把 200 个任务一个一个分配给工作者，然后等每个工作者报告——协调者本身成了瓶颈。

### Claude Code 的解法

`src/utils/swarm/` 实现了共享工作队列模式。核心数据结构是一个所有智能体可见的任务列表（Task List），通常基于文件系统实现。

```
领导者：发布 200 个任务到 TaskList
工作者 1：轮询 → 领取 "Task A" → 执行 → 标记完成
工作者 2：轮询 → 领取 "Task B" → 执行 → 标记完成
工作者 3：轮询 → 领取 "Task C" → 执行 → 标记完成
...
```

领导者不需要微管理（Micro-Manage）。它只做两件事：定义任务列表、等待所有任务完成。工作者完全自主——自己从队列中拉取任务、自己执行、自己标记结果。

### 核心原理

💡 **Swarm 模式**（群体模式）的核心是去中心化协调。与协调者模式（一个领导者分配工作）不同，Swarm 模式让工作者自主决定做什么。这消除了协调者瓶颈——系统吞吐量随工作者数量线性扩展。

关键约束：任务必须是**独立的**（Independent）。如果 Task B 依赖 Task A 的结果，工作者无法自主执行——它需要等 Task A 完成才能开始 Task B，而它不知道 Task A 分配给了谁。有依赖的任务应该用协调者模式。

### 适用条件

- 任务是大规模、独立、同构的（格式检查、测试运行、文件迁移）
- 工作者数量 > 2（2 个工作者用 Swarm 的开销不划算）
- 每个任务的执行时间大致相同（避免一个工作者长期占用一个长任务而其他工作者空闲）

### 不适用条件

- 任务之间有依赖关系
- 任务需要不同的能力或权限（Swarm 假设所有工作者相同）
- 任务数量 < 5——Swarm 的基础设施开销大于收益

### 实现要点

1. **任务领取要原子性**。两个工作者同时看到同一个未领取的任务，只能有一个成功领取。文件系统上可以用原子重命名（Atomic Rename）：`task-pending-001.json` → `task-claimed-worker2-001.json`。
2. **工作者崩溃恢复**。工作者领取了任务后崩溃了。心跳超时（建议 90 秒）后，领导者或其他工作者将任务释放回队列。任务需要一个 `retryCount` 字段，超过阈值（建议 3 次）后标记为永久失败。
3. **基于文件的邮箱（Mailbox）实现点对点通信**。`src/utils/swarm/` 中，每个智能体有一个收件箱文件，其他智能体可以写入消息。用文件锁保证并发安全。

### 反模式

- **领导者轮询任务状态**：领导者每秒检查一次"有没有任务完成了"。应该用通知机制——工作者完成任务时主动通知领导者。
- **单文件队列**：所有任务和状态放在一个 JSON 文件里。工作者数量 > 3 时，文件锁竞争会成为瓶颈。用每任务一文件的方案。
- **无工作者上限**：生成 50 个并发工作者，每个都在调用 LLM API。API 限流（Rate Limiting）会导致大量请求失败。根据 API 配额设置工作者上限。

## 模式 10：多来源命令注册

### 问题场景

你的智能体 CLI 有三种命令来源：
1. 内置命令（`/help`、`/clear`、`/status`）
2. 用户定义的 Skill 文件（`skills/` 目录下的 `SKILL.md`）
3. MCP 服务器暴露的工具（动态注册、随时可能上下线）

用户输入 `/commit`，你需要统一查找、优先级排序、执行。

### Claude Code 的解法

`src/commands/` 实现了中央命令注册表。核心是一个 `getCommands()` 函数，从所有来源聚合命令，按名称去重，应用上下文感知过滤器（Context-Aware Filter）。

```
Registry.getCommands() →
  内置命令（硬编码） +
  Skill 文件（从 skills/ 目录发现） +
  插件命令（动态加载） +
  工作流命令（可脚本化）
  → 按名称去重（后注册的覆盖先注册的）
  → 按认证级别、模式、安全性过滤
  → 返回统一的命令列表
```

### 核心原理

注册表不关心命令来自哪里——只关心它是否实现了标准接口。这是经典的**依赖反转**（Dependency Inversion）：核心系统依赖抽象接口，不依赖具体的命令来源。新增一种命令来源不需要修改核心代码。

工程权衡：统一注册意味着命名冲突不可避免。Claude Code 的策略是**后注册覆盖**——如果内置命令和 Skill 都叫 `/review`，Skill 版本覆盖内置版本。这让用户可以自定义内置行为，但也可能导致意外覆盖。

### 适用条件

- 命令/插件/扩展来自 3 个以上来源
- 需要统一的发现和补全（Tab Completion）
- 扩展可能在运行时动态上下线

### 不适用条件

- 命令来源单一且固定——直接用数组/Map 存储
- 不需要动态发现——硬编码更简单

### 实现要点

1. **接口最小化**。命令接口只需要 `name`、`description`、`execute(args)`。可选字段（`aliases`、`isEnabled`、`completion`）用默认值。
2. **过滤器是注册表的职责，不是命令的职责**。命令不应该知道"当前用户有没有权限执行我"——注册表根据上下文过滤。
3. **支持命令别名（Aliases）**。`/q` 是 `/quit` 的别名。别名在去重时不参与覆盖判断——只有主名称参与。

### 反模式

- **命令自己检查前置条件**。`/deploy` 命令内部检查"我在 CI 环境里吗？"——应该由注册表在过滤阶段就剔除不可用的命令。
- **每个命令来源有自己的执行路径**。内置命令走路径 A，Skill 走路径 B，插件走路径 C。三种路径的错误处理、日志、权限检查各自实现，逻辑漂移不可避免。

## 模式 11：无 GC 打包缓冲区渲染

### 问题场景

你的终端 UI 每秒重渲染 5-10 次。每次渲染，如果用"每个单元格一个对象"的方式（`{ char: 'A', fg: 'red', bg: 'black', bold: true }`），一个 200 列 × 50 行的终端就需要创建 10,000 个对象。每秒 5 帧就是 50,000 个对象/秒。V8 的垃圾回收器（GC）会周期性暂停（GC Pause），用户看到的是终端界面"卡一下"。

### Claude Code 的解法

Claude Code 的终端 UI 框架（一个深度优化的 Ink fork，相关代码在其渲染层）用类型化数组（Typed Array）替代了对象。每个单元格用两个 `Int32` 值表示：

```
单元格 = 2 × Int32：
  Word 0：驻留字符 ID（Interned Character ID）
  Word 1：打包位字段（Packed Bitfield）
           ├── FG 颜色索引（8 bit）
           ├── BG 颜色索引（8 bit）
           ├── 属性标志（粗体/斜体/下划线，4 bit）
           └── 字符宽度（2 bit）
```

💡 **字符驻留**（String Interning）是一种把字符串映射到整数 ID 的技术。相同的字符串只存储一次，引用处用 ID 替代。这样比较两个字符串变成了比较两个整数，赋值也只是复制一个整数。

💡 **打包位字段**（Packed Bitfield）是把多个小数据塞进一个整数的不同位段的技术。一个 32 位整数可以同时存储前景色、背景色、样式属性和字符宽度。用位操作（Bitwise Operation）读写。

### 核心原理

这是经典的游戏引擎渲染技术——Object Pool（对象池）和 Data-Oriented Design（数据导向设计）——应用于终端渲染。

工程权衡：代码变得更难读写。`cell.bold = true` 变成了 `buffer[offset + 1] |= BOLD_BIT`。新开发者需要理解位操作。但在热渲染路径上，性能提升是量级级别的——每帧零对象分配，GC 压力从 50,000 对象/秒降到 0。

### 适用条件

- 高频渲染（每秒 5+ 帧）
- GC 暂停对用户可感知（终端 UI、游戏、动画）
- 单元格数据结构简单且固定（颜色 + 样式 + 字符）

### 不适用条件

- 低频渲染（每秒 < 1 帧）——GC 暂停不可感知
- 单元格数据结构复杂或经常变化——位字段的打包/解包成本超过收益
- 团队不熟悉位操作——维护成本过高

### 实现要点

1. **驻留池（Intern Pool）要支持 eviction**。长时间运行的应用会积累大量不再使用的驻留字符串。需要定期清理或用 LRU 策略。
2. **缓冲区大小预分配**。终端大小变化时重新分配缓冲区。不要每帧动态增长——`TypedArray` 不能原地扩展，增长意味着分配新数组 + 复制。
3. **脏区域追踪（Dirty Region Tracking）**。不是每帧重渲染整个缓冲区——只重渲染变化的区域。这进一步减少了每帧的计算量。

### 反模式

- **部分用对象、部分用缓冲区**：混合方案意味着两套渲染路径，复杂度翻倍但收益减半。要么全用对象（简单），要么全用缓冲区（快）。
- **在位字段中预留"未来扩展位"**：32 位整数的每一位都很宝贵。预留位浪费空间且增加移位计算的心智负担。需要更多字段时，把单元格从 2 × Int32 扩展到 3 × Int32。

## 模式 12：协调者 + 草稿板

### 问题场景

3 个研究智能体分别分析了代码库的不同部分，每个产出了一份 2000 行的报告。协调者需要看到所有报告才能做综合。如果通过对话上下文传递这些报告（把报告内容放进消息），6000 行的文本会瞬间耗掉大量上下文窗口（Context Window）Token。而且协调者的后续轮次都要为这些已读过的报告付费。

### Claude Code 的解法

`src/coordinator/` 和 `src/utils/swarm/` 使用了一个所有智能体可访问的共享文件系统目录——草稿板（Scratchpad）。协调者在启动时创建草稿板目录，将其路径注入到每个工作者的上下文中。工作者将发现写入草稿板文件；协调者按需读取。

```
协调者 → 创建 /tmp/task-abc/scratchpad/
工作者 A → 写入 scratchpad/research-backend.md（2000 行）
工作者 B → 写入 scratchpad/research-frontend.md（1800 行）
工作者 C → 写入 scratchpad/research-tests.md（1500 行）
协调者 → 按需读取，综合成 scratchpad/implementation-plan.md
实现者 → 读取 implementation-plan.md，执行
```

### 核心原理

文件系统是最简单的进程间通信（IPC / Inter-Process Communication）机制之一，但它有一个被低估的优势：**持久性**。

对话消息在智能体终止时丢失。文件不会。工作者可以崩溃、重启，通过读取草稿板从上次中断处继续。这对长时间运行的多智能体任务至关重要——不是所有工作者都能一次成功完成。

另一个优势：**完全绕过上下文窗口限制**。2000 行的报告存成文件后，协调者可以用 ReadFile 工具选择性地只读取相关部分，而不是在上下文窗口中一直携带全文。

### 适用条件

- 智能体间需要传递大量中间数据（> 1000 行文本）
- 多阶段工作流，后续阶段需要前序阶段的完整产出
- 需要审计跟踪——事后可以检查每个智能体写了什么

### 不适用条件

- 中间数据量小（< 50 行）——直接放在通知消息里更简单
- 安全敏感数据——文件系统上的临时文件可能被其他进程读取
- 只需要传递状态（如"完成/失败"），不需要传递内容

### 实现要点

1. **草稿板路径通过智能体上下文注入**。不要硬编码路径。每次任务创建一个新的临时目录，任务完成后清理。
2. **文件命名有约定**。`research-{agent-name}.md`、`plan-v{version}.md`、`findings-{topic}.md`。好的命名让协调者不需要打开文件就知道内容是什么。
3. **写入和读取的时序管理**。工作者写完文件后通过通知告知协调者。协调者不应该轮询文件系统等待文件出现——那是回到了轮询模型的反模式。配合模式 8（XML 通知）使用。

### 反模式

- **草稿板当数据库用**：多个工作者并发写入同一个文件，用文件锁控制并发。到这个程度就应该用真正的数据库或消息队列了。
- **不清理临时目录**：长时间运行的系统积累了几百个草稿板目录。在 `finally` 块中清理，或用操作系统的临时文件自动清理机制。
- **草稿板内容替代通知**：协调者每秒轮询草稿板目录看有没有新文件，而不是等工作者的完成通知。轮询浪费 CPU 且有延迟。

## 融会贯通：自动修复 100 个 Lint 告警

以下用一个具体场景演示如何组合多个模式。场景：一个大型项目有 100 个 ESLint 告警分布在 60 个文件中，你的智能体系统需要自动修复它们。

### 第一步：评估和分类（模式 3 的阶段 1 + 模式 12）

协调者先生成 3 个研究智能体（模式 3：扇出），每个分析一部分告警：

```
研究员 A → "分析 src/components/ 下的 35 个告警，分类为：可自动修复 / 需要上下文判断 / 可能影响运行时行为"
研究员 B → "分析 src/utils/ 下的 40 个告警，同上"
研究员 C → "分析 src/api/ 下的 25 个告警，同上"
```

每个研究员将分类结果写入草稿板（模式 12），例如 `scratchpad/analysis-components.json`。

### 第二步：综合和规划（模式 3 的阶段 2）

协调者读取三份分析报告，发现：
- 70 个告警是纯格式问题（missing semicolons、unused imports），可以安全自动修复
- 20 个告警涉及类型断言和 `any` 类型，需要理解上下文
- 10 个告警是关于 `no-explicit-any` 规则，修复可能改变运行时行为

协调者将修复计划写入 `scratchpad/fix-plan.md`。

### 第三步：批量修复（模式 9：Swarm）

对于 70 个纯格式问题，用 Swarm 模式处理——它们彼此独立、不需要上下文判断。

```
领导者发布 70 个任务到 TaskList
工作者 1-5 并行领取、修复、标记完成
```

每个工作者使用 `buildTool()` 创建的 `EditFile` 工具（模式 1：安全默认值），通过两阶段验证（模式 4）确保修改的文件路径合法且用户允许编辑。

工作者在修复过程中通过 `onProgress` 报告进度（模式 6：渐进式交付），用户在终端看到"已修复 23/70 个格式告警"。

### 第四步：上下文修复（模式 3 的阶段 3）

对于 20 个需要上下文判断的告警，协调者在前台模式（Foreground Mode）下逐个生成实现智能体——这些修复涉及类型推断，可能修改相邻代码，需要串行避免冲突。

每个实现智能体继承协调者的对话上下文（模式 2：上下文 Fork），复用 Prompt Cache，不需要重新理解项目结构。

### 第五步：验证（模式 3 的阶段 4）

一个**全新的**验证智能体（没有参与过任何修复）：

1. 运行 `eslint .` 确认修复后的告警数量减少到 10 个（那些被标记为"可能影响运行时行为"的告警被跳过了）
2. 运行完整测试套件确认没有回归
3. 用 `git diff` 审查所有变更，确认没有非预期的修改

验证智能体将结果通过 XML 通知（模式 8）报告给协调者。

### 模式组合总结

| 步骤 | 使用的模式 | 为什么 |
|------|-----------|--------|
| 评估分类 | 3（扇出）+ 12（草稿板） | 并行分析，结果持久化到文件避免上下文膨胀 |
| 综合规划 | 3（综合阶段） | 协调者自己做决策，不委托 |
| 批量修复 | 9（Swarm）+ 1（安全默认值）+ 4（两阶段验证）+ 6（渐进式交付） | 独立任务并行处理，工具有安全保障，用户看到实时进度 |
| 上下文修复 | 3（实现阶段）+ 2（上下文 Fork） | 串行避免冲突，复用 Prompt Cache 节省成本 |
| 验证 | 3（验证阶段）+ 8（XML 通知） | 全新智能体避免确认偏差，异步通知结果 |

## 12 个模式速查表

| # | 模式 | 核心解决的问题 | 源码位置 |
|---|------|---------------|---------|
| 1 | 工具接口 + 安全默认值 | 多工具系统的一致性和安全性 | `src/Tool.ts`, `buildTool()` |
| 2 | 上下文 Fork | 子智能体复用父级研究成果而不付重新分词成本 | `src/tools/AgentTool/forkSubagent.ts` |
| 3 | 扇出→综合→实现→验证 | 复杂任务的结构化分解 | `src/coordinator/coordinatorMode.ts` |
| 4 | 两阶段权限验证 | 精确的错误消息 + 可独立演化的验证和授权 | `src/Tool.ts` 生命周期 |
| 5 | AsyncLocalStorage 隔离 | 进程内多智能体的状态隔离 | `src/utils/agentContext.ts` |
| 6 | 渐进式结果交付 | 长时间工具的用户反馈 | `src/Tool.ts` `onProgress` 回调 |
| 7 | 延迟工具加载 | 系统提示词的 Token 优化 | `src/Tool.ts` `shouldDefer` + `ToolSearchTool` |
| 8 | XML 异步通知 | 异步多智能体间的结果传递 | `src/tasks/` 通知注入 |
| 9 | 共享工作队列（Swarm） | 大规模独立任务的去中心化处理 | `src/utils/swarm/` |
| 10 | 多来源命令注册 | 多扩展点的统一命令发现 | `src/commands/` |
| 11 | 无 GC 打包缓冲区渲染 | 高频终端渲染的 GC 暂停消除 | Ink fork 渲染层 |
| 12 | 协调者 + 草稿板 | 大型中间产物的跨智能体共享 | `src/coordinator/`, `src/utils/swarm/` |

每个模式解决一个特定的反复出现的工程问题。组合使用时，它们覆盖了从单工具设计到大规模多智能体协调的完整光谱。不要为了"用模式"而用模式——先确认你的问题匹配模式描述的场景，再决定是否采用。
