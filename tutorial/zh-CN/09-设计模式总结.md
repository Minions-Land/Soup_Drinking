# 第九章：12 个可复用设计模式 -- 你的智能体设计手册

> 上一章：[第八章：远程连接](08-远程连接.md)

在整个教程系列中，我们解剖了 Claude Code 的架构 -- 它的工具系统、多智能体协调、权限模型、终端 UI、命令注册和远程桥接。一路上，我们遇到了解决智能体工程中根本性问题的反复出现的设计模式。

本章将这些模式提炼为参考手册。每个模式都包含它解决的问题、解决方案、适用场景和关键洞察。把它当作你构建生产级智能体系统的速查表。

---

## 模式 1：带安全默认值的工具接口

**问题**：当系统有 30+ 个工具时，为每个工具编写权限检查、输入验证等代码造成大量样板代码和不一致。

**方案**：创建 `buildTool()` 工厂函数，为所有内容填充安全默认值。工具只覆盖需要的部分。

**适用场景**：任何拥有多个工具或插件需要一致行为的系统。

**关键洞察**：默认值应该是 fail-closed。忘记声明 `isReadOnly` 的工具默认为 `false`（假定危险）。忘记 `checkPermissions` 的工具默认询问用户。安全是默认，灵活通过覆盖。

---

## 模式 2：上下文 Fork 用于 Prompt 缓存共享

**问题**：子智能体需要父级研究上下文，但重新分词代价高昂。

**方案**：Fork 子级时使用父级系统提示词和对话前缀的字节精确副本，复用 LLM 的 Prompt 缓存，零重新分词成本。

**适用场景**：子智能体需要父级积累的研究上下文的多智能体系统。

**关键洞察**：复制 `renderedSystemPrompt`。克隆 `contentReplacementState`。但创建独立的 `AbortController` -- 子级必须可取消而不影响父级。

---

## 模式 3：扇出研究、综合、实现、验证

**问题**：复杂任务在开始写代码前需要从多个视角理解。单个智能体按顺序做所有事情既慢又容易产生隧道视野。

**方案**：四阶段协调器工作流。阶段 1：并行研究智能体收集信息。阶段 2：协调者综合为规范。阶段 3：定向智能体实现。阶段 4：全新验证智能体审查。

**适用场景**：任何需要先理解系统再修改的任务，特别是横切变更。

**关键洞察**：验证智能体必须是**全新的** -- 没有实现阶段的上下文。否则会受确认偏差影响。

---

## 模式 4：两阶段权限验证

**问题**：执行工具前需要检查"输入有效吗？"和"用户允许吗？"很多系统将它们混为一谈。

**方案**：阶段 1：`validateInput()` 检查逻辑。阶段 2：`checkPermissions()` 检查授权。前者返回错误消息；后者返回 `allow`、`deny` 或 `ask`。

**适用场景**：任何工具需要有效性检查和授权的系统。

**关键洞察**：有效性是客观的。授权是策略。分离它们产生更好的错误消息："文件未找到" vs "权限被拒绝"。

---

## 模式 5：AsyncLocalStorage 实现智能体隔离

**问题**：同一 Node.js 进程中多个智能体并发运行，每个需要隔离状态。通过参数传递状态繁琐且易错。

**方案**：使用 `AsyncLocalStorage` 创建自动流经异步调用链的每智能体上下文。

**适用场景**：进程内多智能体系统，或需要请求作用域状态的 Node.js 应用。

**关键洞察**：消除通过每个函数签名传递上下文的需要。与 Express 用于请求作用域数据的机制相同，应用于智能体隔离。

---

## 模式 6：渐进式结果交付

**问题**：长时间工具在执行期间 UI 什么都不显示，用户怀疑系统冻结。

**方案**：类型化进度回调系统 `ToolCallProgress<P>`，工具报告增量更新，UI 渐进渲染。

**适用场景**：任何耗时超过几秒的工具。

**关键洞察**：进度更新有双重作用。对 UI 表明工作在进行。对系统可携带部分结果，模型可在工具完成前推理 -- 实现早期路线修正。

---

## 模式 7：延迟工具加载

**问题**：30+ 工具中只有 10-15 个常用。在系统提示词中包含所有工具 Schema 浪费 Token。

**方案**：标记不常用工具为 `shouldDefer: true`，从初始提示词省略。模型通过 `ToolSearchTool` 按关键词发现，工具 Schema 加载到下一回合。

**适用场景**：20+ 工具、Prompt 空间宝贵的系统。

**关键洞察**：每个延迟工具提供 `searchHint` 用于匹配。模型为发现工具付出一轮延迟成本，但在不需要该工具的每一轮都节省 Token。

---

## 模式 8：基于 XML 的异步通知

**问题**：后台智能体在不可预测的时间完成工作。主对话需要在结果到达时被告知。

**方案**：在回合之间将 XML 标签通知注入消息流。模型在对话上下文中看到这些通知，自然地据此行动。

**适用场景**：后台工作异步产生结果的多智能体系统。

**关键洞察**：XML 同时可被代码解析（路由和状态跟踪）和被模型阅读（理解发生了什么），无需格式转换。

---

## 模式 9：共享工作队列（Swarm）

**问题**：多个智能体处理大量任务。单一协调者逐个分发造成瓶颈。

**方案**：共享任务列表，智能体自主领取和完成工作。领导者发布任务；工作者轮询、领取、执行、标记完成。

**适用场景**：任务独立的可并行化工作负载。

**关键洞察**：工作者是自主的。领导者只定义任务列表并等待完成。消除协调者瓶颈，随工作者数量线性扩展。

---

## 模式 10：多来源命令注册中心

**问题**：命令来自不同来源。需要统一发现、过滤和调用。

**方案**：中央注册表从所有来源聚合命令，按名称去重，应用上下文感知过滤器。

**适用场景**：任何有多个扩展点的可扩展 CLI、聊天机器人或智能体系统。

**关键洞察**：注册表不关心命令来自哪里，只关心它是否实现了标准接口。将核心与扩展解耦。

---

## 模式 11：无 GC 的打包缓冲区渲染

**问题**：终端 UI 每秒重渲染多次。每次创建数千个对象。GC 暂停造成卡顿。

**方案**：用类型化数组替代每单元格对象。每个单元格两个 `Int32` 值：驻留字符 ID 和打包位字段。字符和样式驻留在池中。每帧零对象分配。

**适用场景**：GC 暂停对用户可见的高频渲染。

**关键洞察**：经典游戏引擎技术应用于终端渲染。代价是代码复杂性（位操作 vs 属性访问），但热渲染路径的性能提升显著。

---

## 模式 12：协调者-工作者 + 草稿板

**问题**：工作者产生大型中间产物。通过对话上下文传递浪费 Token 并达到上下文限制。

**方案**：所有智能体可访问的共享文件系统目录（草稿板）。工作者将发现写入文件；协调者读取用于综合。草稿板路径注入到每个智能体上下文。

**适用场景**：中间产物太大不适合对话上下文的多智能体系统。

**关键洞察**：文件跨智能体生命周期持久。工作者可以崩溃、重启，通过读取草稿板从上次中断处继续。比对话传递更持久，完全绕过上下文窗口限制。

---

## 融会贯通

这 12 个模式不是孤立技术。在 Claude Code 这样的生产系统中，它们协同工作：

- **工具**使用模式 1（安全默认值）和模式 4（两阶段验证）
- **多智能体编排**组合模式 2（上下文 Fork）、模式 3（扇出/综合）、模式 5（AsyncLocalStorage）和模式 12（草稿板）
- **命令系统**实现模式 10（多来源注册表）与模式 7（延迟加载）
- **终端 UI** 使用模式 6（渐进式交付）和模式 11（无 GC 渲染）
- **后台任务**依赖模式 8（XML 通知）和模式 9（工作队列）

每个模式解决一个特定的反复出现的问题。当你在自己的系统中遇到那个问题时，不需要重新发明解决方案 -- 从这本手册中找到它。
