# 第五章：权限与安全 — 给 AI 套上缰绳

## 为什么 AI Agent 的安全和传统软件不一样

传统软件的安全模型建立在确定性之上：你知道代码会做什么，因为你写了它。AI Agent 打破了这个前提 — 模型的输出是概率性的，你无法提前知道它会调用什么工具、传什么参数。

这带来了一个独特的工程难题：你需要给一个"不完全可预测的决策者"授予系统级的操作权限（文件读写、Shell 执行、网络访问），同时确保它不会搞砸。

Claude Code 的方案不是试图"让模型永远正确"，而是承认模型会犯错，然后在架构层面构建多层防线。这个思路值得每个 Agent 开发者学习。

## 权限模式（PermissionMode）

Claude Code 定义了几种权限模式，每种对应不同的安全级别。定义在 `src/types/permissions.ts`：

**`default` 模式** — 标准交互模式。只读操作（读文件、搜索代码）自动放行，写操作和 Shell 执行需要用户逐个审批。这是大多数场景的默认选择。

**`plan` 模式** — 只读探索模式。工具层面直接屏蔽所有写操作，模型只能使用 Read、Grep、Glob 等只读工具。用于"先看看再说"的场景：用户想让模型研究代码库结构，但不想让它做任何修改。

**`auto` 模式** — 模型自主决策 + 安全分类器审核。模型可以自行决定是否执行操作，但每次决策会经过一个安全分类器（`toAutoClassifierInput`）的审查。分类器基于工具的结构化输入判断风险等级，高风险操作仍然会弹出审批。

**`bypassPermissions` 模式** — 跳过所有权限提示。所有操作直接执行，不询问用户。这是一个危险的模式，仅在用户显式 opt-in 后才可用（通过设置 `isBypassPermissionsModeAvailable`），且 UI 会持续显示警告。

这种分级设计的精妙之处在于：不同阶段可以使用不同模式。比如研究阶段用 `plan`，实现阶段切到 `default`，CI 环境用 `auto`。模式切换不需要重新配置工具，因为权限检查是工具执行管道的一部分，而不是工具自身的属性。

## 两阶段验证：validateInput 与 checkPermissions

这是 Claude Code 权限系统最核心的设计。每个工具调用在执行前经过两道独立的检查：

**第一阶段：`validateInput(input, context)`**

回答的问题是"这个调用在逻辑上合法吗？"，和用户权限无关。比如：
- 文件路径是否存在？（FileReadTool）
- 正则表达式是否合法？（GrepTool）
- 超时时间是否在合理范围内？（BashTool）
- 路径是否越界，试图访问系统目录？（FileWriteTool）

如果验证失败，返回 `{ result: false, message: "路径不存在", errorCode: 404 }`，错误信息直接反馈给模型，模型可以据此调整参数重试。

**第二阶段：`checkPermissions(input, context)`**

回答的问题是"用户允许这个操作吗？"。返回三种结果：
- `{ behavior: 'allow' }` — 自动放行（匹配了 alwaysAllow 规则，或操作是只读的）
- `{ behavior: 'deny' }` — 自动拒绝（匹配了 alwaysDeny 规则）
- `{ behavior: 'ask' }` — 需要询问用户（弹出权限提示框）

**为什么分两阶段而不是一个函数？** 因为它们的失败语义不同。`validateInput` 失败意味着"输入有误，请修正"——模型应该重试。`checkPermissions` 失败意味着"你没权限"——模型不应该重试相同的操作，而应该换一种方式。

把这两种语义混在一起，模型就分不清"参数错了"和"被拒绝了"，导致无意义的重试循环。

## BashTool 安全系统：重中之重

BashTool 是整个系统中安全风险最高的工具 — 它可以执行任意 Shell 命令。Claude Code 为它构建了一套专门的安全子系统，代码量比其他任何工具都大。核心文件在 `src/tools/BashTool/` 目录下。

### bashSecurity.ts — 命令过滤

这个文件是安全系统的核心，实现了多层命令过滤。翻开源码，你会看到它对 Shell 的理解深度远超普通的"黑名单匹配"。

**命令替换检测**：文件开头定义了 `COMMAND_SUBSTITUTION_PATTERNS`，一个正则数组。它不只检测常见的 `$()` 和反引号，还覆盖了 Zsh 特有的攻击向量：

```
$()      — 标准命令替换
<()  >() — 进程替换
=()      — Zsh 进程替换
${}      — 参数替换
~[]      — Zsh 参数扩展
(e:      — Zsh glob 限定符（可执行代码）
(+       — Zsh glob 限定符 + 命令执行
```

一个特别有趣的条目是 `=cmd` — Zsh 的 EQUALS 扩展。`=curl evil.com` 会被 Zsh 展开为 `/usr/bin/curl evil.com`，绕过 `Bash(curl:*)` 的拒绝规则。Claude Code 的安全团队显然遇到过这个攻击，因为注释里专门解释了这个绕过路径。

**Zsh 危险命令黑名单**：`ZSH_DANGEROUS_COMMANDS` 集合包含了 `zmodload`、`sysopen`、`syswrite`、`zpty`、`ztcp` 等命令。注释解释了为什么 `zmodload` 是"万恶之源"：它可以加载 `zsh/mapfile`（通过数组赋值实现隐蔽文件 I/O）、`zsh/system`（绕过二进制检查的文件访问）、`zsh/zpty`（伪终端命令执行）、`zsh/net/tcp`（网络数据外泄）。

这种深度的 Shell 安全知识是构建安全 Agent 的关键 — 你不能只看 Bash，还得考虑 Zsh、Fish 等 Shell 的特有行为。

### commandSemantics.ts — 命令语义分类

这个文件把命令分类为四种语义：
- **read** — 只读操作：`cat`、`head`、`tail`、文件读取
- **search** — 搜索操作：`grep`、`find`、`rg`
- **list** — 目录列表：`ls`、`tree`、`du`
- **write** — 写操作：其他所有命令

分类的目的是优化 UI 渲染：read/search/list 操作的输出可以折叠显示（用户通常不需要看全部输出），write 操作的输出需要完整展示。

### destructiveCommandWarning.ts — 破坏性操作警告

这个模块识别高危命令并生成警告信息：
- `rm -rf` — 递归强制删除
- `git reset --hard` — 丢弃所有本地修改
- `git push --force` — 覆盖远程历史
- `chmod 777` — 开放全部权限
- `kill -9` — 强制杀进程

警告不会阻止执行（那是 checkPermissions 的职责），而是在权限提示中加入额外的风险说明，帮用户做出更明智的决定。

### pathValidation.ts — 路径越界检测

确保工具操作的文件路径在工作目录范围内。防止模型通过路径遍历（`../../etc/passwd`）访问系统文件。

### sedValidation.ts — Sed 命令校验

Sed 是个特殊的工具 — 它既可以只读（打印匹配行），也可以原地修改文件（`-i` 标志）。sedValidation 专门解析 sed 命令的参数，判断它是读操作还是写操作，从而决定是否需要额外的权限检查。

## 沙箱系统

当 `shouldUseSandbox.ts` 判定某个命令需要沙箱时（比如运行不受信任的脚本），命令会在受限环境中执行。沙箱限制包括：
- 文件系统访问限制（只能访问工作目录）
- 网络访问限制
- 进程权限限制

沙箱是权限系统的最后一道物理防线 — 即使前面的所有检查都被绕过，沙箱也能限制损害范围。

## 权限规则系统

用户可以通过设置文件配置三类规则：

**`alwaysAllow`** — 自动放行。比如 `Bash(git status)` 表示 `git status` 命令永远自动执行。支持通配符：`Bash(git *)` 允许所有 git 命令。

**`alwaysDeny`** — 自动拒绝。比如 `Bash(curl *)` 禁止所有 curl 命令。

**`alwaysAsk`** — 强制询问。即使在 auto 模式下也要弹出确认框。

规则的匹配通过 `preparePermissionMatcher` 方法实现。每个工具可以自定义匹配逻辑 — BashTool 按命令名和参数匹配，FileWriteTool 按文件路径匹配。

规则有优先级层次：policy（组织策略）> user（用户设置）> session（会话临时规则）。组织管理员可以通过 MDM（💡 移动设备管理，[搜索了解更多](https://www.baidu.com/s?wd=MDM+设备管理)）下发策略，覆盖用户的个人设置。

## 多智能体权限冒泡

当子 Agent 需要执行敏感操作时，权限请求不会在子 Agent 的上下文中处理（子 Agent 可能没有 UI），而是"冒泡"到主线程：

```
子 Agent → checkPermissions 返回 'ask'
  → 请求通过 AppState 传递到主线程
  → 主线程的 UI 弹出权限确认框
  → 用户批准/拒绝
  → 结果路由回子 Agent
  → 子 Agent 继续/终止操作
```

这个设计确保了：即使在多层嵌套的 Agent 中，敏感操作的最终审批权始终在用户手中。

## 拒绝追踪（Denial Tracking）

`denialTracking.ts` 跟踪权限拒绝的次数。如果模型反复尝试被拒绝的操作，系统会升级响应：

普通拒绝 → 模型收到"权限被拒绝"的消息
重复拒绝 → 系统提示模型停止尝试此类操作
持续重复 → 可能触发对话中断

这防止了模型陷入"拒绝→重试→拒绝"的死循环，浪费用户的 Token 和时间。

## Hook 系统：可扩展的安全检查

`src/utils/hooks/` 提供了两个关键扩展点：

**PreToolUse** — 工具执行前触发。外部插件可以注册 PreToolUse Hook 来添加额外的安全检查。比如，一个企业安全插件可以在此检查命令是否符合公司的安全策略。

**PostToolUse** — 工具执行后触发。可以用来审计日志记录、检查工具输出中是否包含敏感信息（如 API Key 泄露）。

Hook 的执行结果可以覆盖工具的权限决策 — 一个 Hook 可以将 `allow` 改为 `deny`，反之亦然。

## 设计你的安全系统：关键原则

从 Claude Code 的安全设计中可以提取这些原则：

1. **假设模型会犯错** — 不要试图通过 Prompt 工程让模型"永远安全"，而是在架构层面限制错误的影响范围
2. **分层防御** — Schema 验证、逻辑验证、权限检查、命令过滤、沙箱、Hook，每层独立工作
3. **失败时关闭（Fail-Closed）** — 默认拒绝，明确允许。`buildTool()` 的默认值是 `isReadOnly: false`、`isConcurrencySafe: false` — 假设最坏情况
4. **区分验证和授权** — validateInput 和 checkPermissions 的分离是刻意的设计
5. **深度理解你的执行环境** — Claude Code 对 Zsh 的安全理解程度说明：如果你要给 Agent Shell 执行权限，你需要比模型更懂 Shell
