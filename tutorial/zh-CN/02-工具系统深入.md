# 第二章：工具系统 -- 工具如何工作

## 简介

工具是 Claude Code 与外部世界交互的主要方式。模型不能自己读文件、运行命令或编辑代码。它必须请求工具调用，由系统执行。本章解释工具接口，走过一个真实工具，并展示如何设计你自己的工具。

## 工具接口

Claude Code 中的每个工具都实现了 `src/Tool.ts` 中定义的 `Tool` 接口。以下是其基本形状（代码使用 TypeScript（💡 微软开发的JavaScript超集，加了类型系统，让大型项目更安全）编写）：

```typescript
interface Tool {
  name: string;                    // 唯一标识符（如 "Read"）
  description: string;             // 在系统提示词中展示给模型
  inputSchema: ZodSchema;          // 用于输入验证的 Zod（💡 TypeScript校验库，检查数据格式是否正确，如 z.string() 验证某值必须是字符串）模式
  
  // 生命周期方法
  validateInput?(input): Promise<ValidationResult>;
  checkPermissions?(input, context): Promise<PermissionResult>;
  call(input, context): Promise<ToolResult>;
  
  // UI 方法
  renderToolUseMessage?(input): React.ReactNode;
  renderToolResultMessage?(result): React.ReactNode;
  
  // 元数据
  isReadOnly?: boolean;            // 此工具是否修改状态？
  isEnabled?(context): boolean;    // 此工具是否应该可用？
  prompt?: string;                 // 给模型的额外指令
  userFacingName?: string;         // UI 中的显示名称
}
```

这个接口是每个工具必须遵守的契约。注意关注点分离：验证与权限分离，权限与执行分离，执行与渲染分离。每个关注点都有自己的方法，使系统模块化且可测试。

## buildTool() 模式

工具不直接实现所有方法，而是使用 `buildTool()` 工厂函数。此函数接受配置对象并返回完整的 `Tool`：

```typescript
export const MyTool = buildTool({
  name: "MyTool",
  description: "做一些有用的事情",
  inputSchema: z.object({
    path: z.string().describe("要处理的文件路径"),
  }),
  isReadOnly: true,
  async call(input, context) {
    return { result: "完成" };
  },
});
```

`buildTool()` 为可选方法提供合理的默认值。关键是，默认值是**保守的**：新工具默认被假设为有写入能力且不支持并发，除非你显式声明。这是一种"关闭失败"（fail-closed）设计 -- 忘记设置标志会导致更多限制，而不是更少。

## 详解：FileReadTool

让我们以 `FileReadTool` 作为具体例子。此工具从磁盘读取文件并返回其内容。

**文件结构：**

```
src/tools/FileReadTool/
  |-- FileReadTool.ts   # 主实现（call、验证、权限）
  |-- UI.tsx            # 渲染函数
  |-- prompt.ts         # 工具描述和提示常量
  |-- limits.ts         # Token（💡 AI处理文本的基本单位，约4个英文字母或1.5个汉字算1个token）和大小限制
  |-- imageProcessor.ts # 图像处理逻辑
```

这是标准的工具目录布局：实现、UI、提示词和专用模块。

**输入模式：**

```typescript
const inputSchema = z.object({
  file_path: z.string().describe("文件的绝对路径"),
  offset: z.number().optional().describe("起始行号"),
  limit: z.number().optional().describe("读取行数"),
})
```

Zod 模式承担双重职责：定义模型在工具描述中看到的内容，以及在运行时验证模型的输出。

**validateInput()：** 检查文件路径是否为绝对路径、是否指向被阻止的设备如 `/dev/zero`、以及 offset 和 limit 是否非负。

**checkPermissions()：** 验证用户是否已授予对此文件路径的读取权限。如果路径在受保护目录内（如 `~/.ssh`），触发用户批准提示。

**call()：** 实际工作：解析路径、检测文件类型、读取内容、检查 token 限制、返回结果。

## 两阶段验证模式

Claude Code 使用两阶段验证方法：

**第一阶段：模式验证（自动）。** Zod 模式在任何方法调用之前自动验证。以低成本捕获结构性错误。

**第二阶段：语义验证（validateInput）。** 检查业务逻辑：文件是否存在？路径是否是绝对的？此阶段产生帮助模型自我修正的人类可读错误。

关键洞见：验证错误给模型返回清晰的错误消息（"路径必须是绝对路径"），而权限拒绝触发面向用户的权限提示（"允许读取 /etc/passwd 吗？"）。分离这些关注点使每一层保持简单。

## ToolUseContext

当工具的 `call()` 方法被调用时，它接收一个 `ToolUseContext` 对象。此上下文提供工具需要的一切，而不需要导入全局变量：

- **AppState** 提供对权限系统、对话历史、任务注册表和配置的访问。
- **AbortSignal** 允许工具在执行中途被取消。
- **readFileTimestamps** 被写入工具用来检测文件是否在上次读取后发生了变化。
- **tools** -- 完整的工具池，因为工具可以调用其他工具。

这种上下文注入方法使工具与全局状态解耦。它们不导入单例，而是接收所需的一切。

## 工具结果管理

工具的 `call()` 方法返回 `ToolResult`，可以是字符串、内容块数组或错误。

当结果超过 `maxResultSizeChars`（通常 40,000 字符）时，系统将完整输出持久化到磁盘，并向模型发送截断的预览。这防止了上下文窗口（💡 AI一次能"记住"的最大文本量，超过这个限制就会"遗忘"早期内容）的耗尽。

工具还可以使用 `setToolResult()` 产生**渐进式结果**，在最终结果准备好之前流式输出部分内容。

## 如何设计你自己的工具

1. **从模式开始。** 用 Zod 定义输入形状。使用 `.describe()` 要明确。
2. **保持工具聚焦。** 每个工具应该只做一件事。
3. **执行前验证。** 使用两阶段验证。
4. **返回结构化错误。** 出错时返回清晰的错误消息，而不是异常。
5. **标记只读工具。** 设置 `isReadOnly: true` 有助于权限系统并启用并行执行。
6. **注入上下文，不要导入全局变量。** 通过上下文对象传递工具所需的一切。
7. **管理结果大小。** 大型结果可能耗尽上下文窗口。
8. **尊重取消。** 在长时间运行的操作中检查 `AbortSignal`。（异步/async（💡 不等待某个操作完成就继续做其他事，就像发邮件后不用守着等回复）操作尤其需要此机制。）

工具系统是 Claude Code 最重要的抽象。每个能力 -- 从读取文件到生成子智能体 -- 都是工具。理解这个系统可以解锁一切。
