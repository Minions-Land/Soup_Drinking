# 第三章：多智能体系统

## 1. 为什么需要多智能体

单个 Agent 在处理结构化、边界清晰的任务时表现很好——修一个 bug、写一段函数、回答一个关于代码结构的问题。但真实的工程场景远比这复杂。

考虑一个具体例子：你要在一个 500+ 文件的 monorepo 里做一次安全审计（Security Audit）。这个任务涉及：

- **搜索阶段**：在不同模块中并行搜索敏感函数调用（SQL 拼接、`eval()`、未过滤的用户输入）
- **分析阶段**：每个发现都需要追踪数据流，判断是否真的可被利用
- **修复阶段**：修改代码、更新测试、确保没引入新的回归
- **验证阶段**：独立地检查修复是否完整，是否引入了新问题

如果让一个 Agent 线性地做这些事，它会在搜索阶段就消耗大量上下文窗口（Context Window）💡 *模型一次能"看到"的全部文本长度限制*，等到修复阶段，早期的搜索结果可能已经被压缩或丢失了。更糟的是，让同一个 Agent 既负责实现又负责验证，它会倾向于确认自己的工作是正确的——这是认知偏差（Confirmation Bias）在 LLM 中的表现。

Claude Code 的解决方案是：把"委托工作给另一个智能体"这件事本身变成一个工具调用。主 Agent 可以像调用 `Bash` 或 `Read` 一样调用 `AgentTool`，生成子 Agent 来处理特定子任务。每个子 Agent 拥有独立的对话上下文、专属的系统提示词（System Prompt）和受限的工具集。

这个设计有三个关键的工程权衡值得理解：

**上下文隔离 vs. 信息共享**。子 Agent 不应该被父 Agent 正在进行的讨论"污染"——一个专注于搜索 SQL 注入的子 Agent 不需要看到父 Agent 关于 CI/CD 流水线的思考。但它又需要继承足够的上下文来理解代码库结构。Claude Code 通过 Context Fork（上下文分叉）来解决这个问题，后面会详细分析。

**成本控制 vs. 能力完整性**。每个子 Agent 都是一次独立的 API 调用链，都在消耗 token。如果子 Agent 工具太多、提示词太长，成本会失控。但如果限制太严格，子 Agent 又无法完成工作。不同类型的内置 Agent 通过精确的工具限制来实现这个平衡。

**并行性 vs. 协调复杂度**。并行生成多个 Agent 可以大幅提速，但需要管理结果汇总、冲突检测和资源竞争。Claude Code 的后台任务系统（第四章详述）和 `<task-notification>` 机制为此提供了基础设施。

## 2. AgentTool 的完整生命周期

### 2.1 AgentDefinition：智能体的蓝图

在深入执行流程之前，先理解子 Agent 是怎么被定义的。每个 Agent 类型都对应一个 `AgentDefinition` 对象（定义在 `src/tools/AgentTool/loadAgentsDir.ts`）。

核心字段包括：

- **`agentType`**：唯一标识符，如 `'Explore'`、`'general-purpose'`、`'verification'`
- **`whenToUse`**：自然语言描述，告诉模型什么时候应该选择这个 Agent。这段文字会出现在 AgentTool 的工具定义（Tool Schema）中，模型根据它来决策
- **`tools`**：工具白名单。`['*']` 表示所有工具；也可以列出具体名称
- **`disallowedTools`**：工具黑名单。优先级高于 `tools`。例如 Explore Agent 禁用了 `FileEdit`、`FileWrite`、`NotebookEdit` 和 `Agent`（防止递归生成）
- **`getSystemPrompt`**：返回该 Agent 专属系统提示词的函数
- **`model`**：模型选择。`'inherit'` 表示使用父 Agent 的模型；也可以指定如 `'haiku'`（用于需要速度而非深度的场景）
- **`permissionMode`**：权限模式，如 `'bubble'`（权限提示冒泡到父级终端显示）
- **`source`**：来源标记——`'built-in'`（内置）、`'user'`（`.claude/agents/` 目录中的自定义）、`'plugin'`（插件提供）
- **`omitClaudeMd`**：是否省略 CLAUDE.md 上下文。只读搜索 Agent 不需要编码规范和提交规则，省略后可节省大量 token
- **`maxTurns`**：最大对话轮数限制，防止失控循环
- **`background`**：是否默认在后台运行
- **`hooks`**：Agent 级别的生命周期钩子（Hooks）

对于自定义 Agent，定义以 Markdown 或 JSON 文件的形式放在 `.claude/agents/` 目录。JSON 格式遵循 `AgentJsonSchema`（在 `loadAgentsDir.ts` 中定义），包含 `description`、`tools`、`prompt`、`model`、`permissionMode` 等字段。Markdown 格式则通过 frontmatter 元数据头部声明配置。

### 2.2 从模型决策到子 Agent 返回：完整流程

当模型在主循环中决定需要委托任务时，它调用 `AgentTool`。以下是完整的执行流程：

**第 1 步：模型生成 tool_use 块**

模型输出一个 `tool_use` 内容块，包含 `description`（3-5 字简述）、`prompt`（子 Agent 的具体任务指令）、`subagent_type`（可选，指定 Agent 类型）和 `run_in_background`（可选，是否后台运行）。

**第 2 步：AgentTool.call() 接收调用**

`AgentTool.tsx` 中的 `call()` 方法启动。它首先解析输入参数、确定目标 Agent 类型。如果 `subagent_type` 未指定且 Fork Subagent 实验开启，走 fork 路径（后面详述）；否则从内置 Agent 列表或 `.claude/agents/` 中查找匹配的定义。

**第 3 步：工具池组装**

调用 `assembleToolPool()` 为子 Agent 组装可用工具集。这一步很关键——子 Agent 的工具池是**独立计算的**，不是简单继承父级的。`resolveAgentTools()` 函数（`agentToolUtils.ts`）会根据 Agent 定义中的 `tools` 白名单和 `disallowedTools` 黑名单过滤。

**第 4 步：确定执行模式（前台/后台/fork）**

这是一个重要的分支点，下一节详细分析。

**第 5 步：调用 runAgent()**

核心执行函数。它是一个 `AsyncGenerator`——通过 `yield` 逐条返回子 Agent 生成的消息，而不是等全部完成后一次性返回。这使得前台模式可以流式显示进度。

**第 6 步：runAgent 内部的初始化**

在进入查询循环之前，`runAgent()` 做了大量准备工作（见 `runAgent.ts` 第 330-747 行）：

- 解析模型：调用 `getAgentModel()` 确定实际使用的模型
- 创建 Agent ID：`createAgentId()` 生成唯一标识
- 准备上下文消息：合并 fork 上下文消息和新的任务提示
- 克隆文件状态缓存：如果是 fork，复制父级的 `readFileState`；否则创建全新的
- 构建用户上下文和系统上下文：决定是否包含 CLAUDE.md（`omitClaudeMd` 控制）和 git 状态
- 配置权限模式：根据 Agent 定义覆盖权限设置，处理 `shouldAvoidPermissionPrompts` 标志
- 组装系统提示词：调用 `getAgentSystemPrompt()` 或使用 fork 提供的 `override.systemPrompt`
- 创建 AbortController：异步 Agent 获得独立的取消控制器，同步 Agent 与父级共享
- 执行 SubagentStart 钩子：收集额外上下文
- 注册 frontmatter 钩子：Agent 定义中的生命周期钩子在此注册
- 预加载 Skills：Agent 定义中声明的 skills 在此加载
- 初始化 Agent 级 MCP 服务器：Agent 可以声明自己的 MCP 服务器配置
- 创建子 Agent 上下文：调用 `createSubagentContext()` 组装完整的 `ToolUseContext`

**第 7 步：查询循环**

```
runAgent.ts 第 748-860 行：
for await (const message of query({...})) {
    // 处理每条消息：
    // - stream_event: 转发 API 指标到父级
    // - attachment: 处理 max_turns_reached 信号
    // - recordable messages: 记录到 sidechain transcript，yield 给调用者
}
```

这个循环本质上就是为子 Agent 运行一个独立的 `query()` 循环——和主 Agent 的查询循环结构完全一致。子 Agent 可以调用工具、思考、生成文本，直到它认为任务完成或达到 `maxTurns` 限制。

**第 8 步：清理**

`runAgent()` 的 `finally` 块执行全面清理（第 816-858 行）：
- 清理 Agent 级 MCP 服务器连接
- 清除 session hooks
- 释放 Prompt Cache 追踪状态
- 清空文件状态缓存和初始消息数组（释放内存）
- 注销 Perfetto trace 条目
- 清除 Agent 的 todo 列表条目（防止内存泄漏）
- **调用 `killShellTasksForAgent()`**——终止该 Agent 生成的所有后台 bash 任务

最后一点特别重要。如果 Agent 在执行过程中通过 `BashTool` 启动了后台进程（比如 `run_in_background` 的 `tail -f`），这些进程不应该在 Agent 结束后继续运行。`killShellTasksForAgent()` 遍历 `AppState.tasks`，找到所有 `agentId` 匹配且状态为 `running` 的 `LocalShellTask`，逐一终止。

**第 9 步：结果返回给父 Agent**

`AgentTool.tsx` 收集 `runAgent()` yield 出的所有消息，提取最后一条 assistant 消息作为子 Agent 的"答案"，打包为工具结果（tool_result）返回给父 Agent 的查询循环。

### 2.3 三种执行模式的详细对比

子 Agent 有三种执行模式，选择哪种对用户体验和系统行为有显著影响：

**前台模式（Foreground）**

子 Agent 阻塞父 Agent 的查询循环。父 Agent 发出 `AgentTool` 调用后，必须等子 Agent 完成才能继续。在 UI 上，子 Agent 的输出直接流式显示给用户。

适用场景：简单的搜索或分析任务，用户正在等待一个快速答案。比如 Explore Agent 查找几个文件的位置——几秒钟就能完成，没必要后台化。

**后台模式（Background）**

子 Agent 在独立的异步上下文中运行。父 Agent 发出调用后立即获得一个 task ID，可以继续处理其他工作。子 Agent 完成后，通过 `<task-notification>` XML 块通知父 Agent。

关键实现细节：
- 后台 Agent 获得**独立的 AbortController**（`runAgent.ts` 第 524-528 行）。父 Agent 被取消不影响后台 Agent
- 后台 Agent 的 `isNonInteractiveSession` 被设为 `true`（第 668-672 行），意味着它无法显示权限提示对话框。遇到需要权限的操作，要么自动拒绝，要么通过 `bubble` 模式冒泡到父级
- 通过 `registerAsyncAgent()` 在任务系统中注册，使用户可以通过 UI 查看进度、取消运行

适用场景：耗时较长的任务，用户不需要立即看到结果。比如运行完整测试套件、执行验证检查。

**Fork 模式**

这是最新的模式，通过 `FORK_SUBAGENT` 特性门控制（Feature Gate）💡 *一种控制新功能渐进式发布的开关机制，通常由远程配置服务控制*。Fork 模式的核心思想是：子 Agent 继承父 Agent 的**完整对话上下文**，而不仅仅是任务提示。

Fork 模式的触发条件是：`subagent_type` 参数被省略，且 `FORK_SUBAGENT` 实验处于开启状态，且当前不在 Coordinator 模式下（`forkSubagent.ts` 第 32-38 行）。满足条件时，使用 `FORK_AGENT` 定义（第 60-71 行）而非注册的 Agent 类型。

Fork Agent 的特殊之处：
- `tools: ['*']` + `useExactTools: true`：使用父 Agent **完全相同**的工具集，不做任何过滤。这保证了 API 请求中的 tools 定义字节一致，是 Prompt Cache 共享的关键
- `model: 'inherit'`：使用父 Agent 的模型，保证上下文窗口长度一致
- `permissionMode: 'bubble'`：权限提示冒泡到父级终端
- `getSystemPrompt: () => ''`：空的系统提示词函数——实际上不会被调用，因为 fork 路径直接传递 `override.systemPrompt`（父 Agent 已渲染的系统提示词字节）

Fork 模式默认在后台运行，所有 fork 子 Agent 共享统一的 `<task-notification>` 交互模型。

**决策树：如何选择执行模式**

```
模型调用 AgentTool
  │
  ├─ subagent_type 已指定？
  │    ├─ YES → 查找对应的 AgentDefinition
  │    │         ├─ definition.background == true？ → 后台模式
  │    │         ├─ run_in_background == true？ → 后台模式
  │    │         └─ 否则 → 前台模式
  │    │
  │    └─ NO → FORK_SUBAGENT 实验开启？
  │              ├─ YES 且不在 Coordinator 模式 → Fork 模式（后台）
  │              └─ NO → 使用 general-purpose Agent，走标准路径
  │
  └─ Coordinator 模式下？
       └─ 所有 Agent 强制后台运行
```

## 3. Context Fork 的深入分析

Context Fork 是整个多智能体系统中最精妙的设计，直接决定了系统的成本效率。

### 3.1 forkSubagent.ts 具体做了什么

`buildForkedMessages()` 函数（`forkSubagent.ts` 第 107-169 行）负责构建 fork 子 Agent 的初始消息。它的输入是一个 `directive`（子 Agent 的具体指令）和父 Agent 最后一条 `assistantMessage`。

构建过程如下：

1. **克隆父 Agent 的 assistant 消息**：保留所有内容块（thinking、text、以及每一个 tool_use 块）
2. **为每个 tool_use 构建占位 tool_result**：所有占位结果使用完全相同的文本 `'Fork started — processing in background'`——这是关键，保证字节一致
3. **拼接最终 user 消息**：`[...所有占位 tool_result, 每个子 Agent 特有的 directive text block]`

最终的消息序列是：

```
[...父级历史消息, 完整 assistant 消息(所有 tool_use), user 消息(占位 results + directive)]
```

只有最后一个 text block（directive）在不同 fork 子 Agent 之间有差异。这意味着整个 API 请求的前缀——系统提示词、工具定义、历史消息、assistant 消息、占位 tool_result——在所有 fork 子 Agent 之间都是**字节精确一致**的。

### 3.2 Prompt Cache 共享的原理

Anthropic API 的 Prompt Cache（提示词缓存）💡 *API 服务端缓存已处理的提示词前缀，后续请求如果前缀相同则跳过重新分词（Tokenization）和编码，直接复用缓存* 的工作机制是基于请求前缀匹配的。缓存键（Cache Key）由以下部分组成：系统提示词、工具定义、模型名称、消息前缀和 thinking 配置。

如果两个 API 请求的这些部分完全一致（字节级别），第二个请求可以命中第一个请求创建的缓存，避免重新分词——这不仅加速了响应，还显著降低了成本（缓存读取的 token 价格远低于新处理）。

Fork 模式的设计正是围绕最大化缓存命中来构建的：

- **系统提示词**：fork 路径使用 `override.systemPrompt` 传递父 Agent 已渲染的系统提示词**字节**，而非重新调用 `getSystemPrompt()` 生成。为什么？因为系统提示词的生成依赖运行时状态（比如 GrowthBook 特性门的状态可能从 cold 变为 warm），重新生成可能产生微小差异。只有传递已渲染的原始字节才能保证精确匹配
- **工具定义**：`useExactTools: true` 跳过 `resolveAgentTools()` 的过滤逻辑，直接使用父 Agent 的工具池。即使 fork 子 Agent 理论上不需要某些工具，为了缓存一致性也保留它们
- **thinking 配置**：fork 路径继承父 Agent 的 `thinkingConfig`（第 682-684 行），而普通子 Agent 会禁用 thinking 来控制输出 token 成本
- **占位 tool_result**：所有 fork 子 Agent 使用相同的占位文本 `FORK_PLACEHOLDER_RESULT`，保证 user 消息的前缀一致

### 3.3 为什么要复制 renderedSystemPrompt 而不是重新生成

这个设计决策在 `forkSubagent.ts` 的注释中有明确记录（第 53-58 行）：

> The getSystemPrompt here is unused: the fork path passes `override.systemPrompt` with the parent's already-rendered system prompt bytes, threaded via `toolUseContext.renderedSystemPrompt`. Reconstructing by re-calling getSystemPrompt() can diverge (GrowthBook cold→warm) and bust the prompt cache; threading the rendered bytes is byte-exact.

"GrowthBook cold→warm" 指的是：系统提示词生成过程中会查询特性门状态。GrowthBook SDK 在首次查询时可能返回默认值（cold），缓存建立后返回实验值（warm）。如果父 Agent 的系统提示词是在 cold 状态下生成的，而子 Agent 在 warm 状态下重新生成，两者就会不同——哪怕只差一个字符，缓存就彻底失效。

这个教训是实践中得来的：prompt cache 失效是一种**静默的成本膨胀**——功能上一切正常，但费用悄然翻倍。

### 3.4 成本影响的估算

假设一个父 Agent 的对话上下文有 50K token。在没有缓存共享的情况下，fork 出 3 个子 Agent：

- 每个子 Agent 独立处理 50K 输入 token → 总计 150K 输入 token（全价）

有缓存共享：
- 第 1 个子 Agent：50K 输入 token（全价，同时创建缓存）
- 第 2、3 个子 Agent：各 50K 输入 token（缓存读取价格，约为全价的 1/10）
- 总计：50K 全价 + 100K 缓存价 ≈ 60K 等效全价 token

对于大规模部署（比如 Anthropic 内部的使用数据显示每周 34M+ Explore Agent 生成），这个优化的累积效果是巨大的。`runAgent.ts` 中还有一条注释（第 387 行）提到，仅省略 Explore/Plan Agent 的 CLAUDE.md 上下文就可以节省"~5-15 Gtok/week"。

### 3.5 防止递归 fork

Fork 子 Agent 保留了 `AgentTool` 在工具池中（为了缓存一致性），所以理论上它可以再次调用 `AgentTool` 并触发新的 fork。`isInForkChild()` 函数（第 78-89 行）通过检查消息历史中是否存在 `<fork-boilerplate>` 标签来检测当前是否已在 fork 子 Agent 内部，如果是则拒绝新的 fork 请求。

## 4. 内置智能体类型详解

### 4.1 general-purpose Agent

定义在 `src/tools/AgentTool/built-in/generalPurposeAgent.ts`。

**系统提示词核心**：通用的"完成任务并简洁报告"指令，加上文件搜索和分析的指导原则。

**工具配置**：`tools: ['*']`——拥有所有工具的完整访问权限，包括文件编辑、bash 执行等。

**模型**：未指定 `model` 字段，使用 `getDefaultSubagentModel()` 返回的默认值。

**使用场景**：模型在不确定该用哪个专业 Agent 时的后备选择。`whenToUse` 描述中明确说"when you are searching for a keyword or file and are not confident that you will find the right match in the first few tries use this agent"。

这个 Agent 类型是最"宽松"的——它几乎什么都能做。工程权衡在于：通用性带来了灵活性，但也意味着更高的成本（工具定义本身占用 token）和更大的出错空间。

### 4.2 Explore Agent

定义在 `src/tools/AgentTool/built-in/exploreAgent.ts`。

**系统提示词核心**：明确声明"READ-ONLY MODE"，列出所有被禁止的操作（创建文件、修改文件、删除文件、使用重定向写文件、运行修改系统状态的命令）。强调速度——"you are meant to be a fast agent that returns output as quickly as possible"，鼓励并行的工具调用。

**工具配置**：通过 `disallowedTools` 禁用 `Agent`、`ExitPlanMode`、`FileEdit`、`FileWrite`、`NotebookEdit`。注意它没有在白名单里列工具，而是用黑名单排除，这意味着它可以使用 `Bash`（但系统提示词限制了 Bash 只能做只读操作）、`Glob`、`Grep`、`FileRead` 等。

**模型**：外部用户使用 `'haiku'`（更快更便宜），Anthropic 内部使用 `'inherit'`（继承父级模型，因为内部有 A/B 测试需求）。

**特殊优化**：
- `omitClaudeMd: true`——搜索 Agent 不需要编码规范
- 注释中提到 `EXPLORE_AGENT_MIN_QUERIES = 3`——暗示对搜索 Agent 的最小查询次数有要求

**使用场景**：快速搜索文件模式（如 `src/components/**/*.tsx`）、搜索代码中的关键字（如"API endpoints"）、回答关于代码库的问题（如"how do API endpoints work?"）。调用时需指定彻底程度：`quick`（基础搜索）、`medium`（适度探索）或 `very thorough`（全面分析）。

### 4.3 Plan Agent

定义在 `src/tools/AgentTool/built-in/planAgent.ts`。

**系统提示词核心**：定位为"software architect and planning specialist"。和 Explore 一样是只读模式，但目标不同——它要设计实现方案而非搜索信息。输出格式要求以"Critical Files for Implementation"列表结尾，列出 3-5 个最关键的文件。

**工具配置**：`disallowedTools` 与 Explore 相同，`tools` 直接引用 `EXPLORE_AGENT.tools`（代码级别的复用）。

**模型**：`'inherit'`——规划需要强模型的推理能力。

**特殊优化**：`omitClaudeMd: true`——Plan Agent 可以在需要时直接 `Read` CLAUDE.md 文件来了解项目规范，不需要在每次调用时都注入上下文。

**使用场景**：在开始实现之前，先让 Plan Agent 探索代码库并设计方案。它的输出是结构化的实现计划，包含步骤序列、依赖关系和潜在挑战。

### 4.4 Verification Agent

定义在 `src/tools/AgentTool/built-in/verificationAgent.ts`。这是最有趣的 Agent 类型。

**系统提示词核心**：长达 130 行的系统提示词，是所有内置 Agent 中最详细的。核心理念是："Your job is not to confirm the implementation works — it's to try to break it."

系统提示词明确识别了两种失败模式（Failure Patterns）：
1. **验证回避**（Verification Avoidance）：面对检查时找理由不运行——只读代码、叙述测试方案、写"PASS"就结束
2. **被前 80% 迷惑**（Being Seduced by the First 80%）：看到通过的测试套件就倾向通过，忽视后台崩溃、状态丢失等问题

提示词还包含了反自欺骗机制——列出常见的"自我合理化"（Rationalization）借口：
- "代码看起来是正确的"——读代码不是验证，运行它
- "实现者的测试已经通过了"——实现者也是 LLM，独立验证
- "我没有浏览器"——你检查过 MCP 工具列表吗？

**工具配置**：同样禁用 `Agent`、`FileEdit`、`FileWrite`、`NotebookEdit`。注意 `WebFetch` 没有被禁用——验证可能需要访问外部 API。

**特殊配置**：
- `background: true`——验证默认后台运行，不阻塞实现流程
- `color: 'red'`——在 UI 中用红色标识，提醒用户"这是验证者"
- `model: 'inherit'`——验证需要和实现者同等的推理能力
- `criticalSystemReminder_EXPERIMENTAL`——注入到每轮对话末尾的强制提醒文本

**为什么验证 Agent 只有只读工具？** 这不仅仅是安全考虑。如果验证 Agent 可以修改代码，它可能会"修复"发现的问题而不是报告它们——这会让验证结果失去客观性。验证者应该产出一份报告（PASS/FAIL/PARTIAL），由实现者或用户决定如何处理。

**输出格式**：必须以 `VERDICT: PASS`、`VERDICT: FAIL` 或 `VERDICT: PARTIAL` 结尾。每个检查点必须包含实际运行的命令和观察到的输出，没有命令运行记录的 PASS 被视为 skip。

### 4.5 其他内置 Agent

- **StatuslineSetup Agent**：配置终端状态栏的专用 Agent
- **ClaudeCodeGuide Agent**：帮助用户了解 Claude Code 本身功能的向导 Agent（仅在非 SDK 入口点可用）

### 4.6 自定义智能体：.claude/agents/ 目录

自定义 Agent 的加载由 `loadAgentsDir.ts` 中的 `loadMarkdownFilesForSubdir()` 和 JSON schema 验证处理。

JSON 格式的 Agent 定义支持以下字段（从 `AgentJsonSchema` 提取）：

```
.claude/agents/security-reviewer.json
{
  "description": "安全审查专家",        // 必填：简短描述
  "prompt": "你是安全审查专家...",       // 必填：系统提示词
  "tools": ["Bash", "Read", "Grep"],   // 可选：工具白名单
  "disallowedTools": ["FileEdit"],      // 可选：工具黑名单
  "model": "inherit",                   // 可选：模型选择
  "permissionMode": "bubble",           // 可选：权限模式
  "mcpServers": ["my-server"],          // 可选：MCP 服务器
  "hooks": {...},                       // 可选：生命周期钩子
  "maxTurns": 50,                       // 可选：最大轮数
  "skills": ["my-skill"],              // 可选：预加载的 skills
  "memory": "project",                  // 可选：记忆范围
  "background": true,                   // 可选：默认后台
  "isolation": "worktree"              // 可选：隔离模式
}
```

自定义 Agent 的 `source` 被标记为 `'user'`，这影响权限检查——在 `strictPluginOnlyCustomization` 模式下，用户定义的 Agent 的 MCP 服务器和 hooks 会被跳过（`runAgent.ts` 第 117-127 行和 563-566 行）。

## 5. 智能体间通信

### 5.1 SendMessageTool 的工作原理

`SendMessageTool`（定义在 `src/tools/SendMessageTool/`）是智能体间通信的唯一正式通道。它的设计基于一个关键认知：Agent 的纯文本输出对其他 Agent **不可见**——系统提示词中明确说"Your plain text output is NOT visible to other agents — to communicate, you MUST call this tool."

工具的输入 schema 包含三个字段：
- `to`：接收者。可以是队友名称（如 `"researcher"`）、广播 `"*"`（发送给所有队友）、UDS socket 路径（如 `"uds:/tmp/cc-socks/1234.sock"`，用于跨会话通信）、或 Bridge 会话 ID（用于远程控制对等通信）
- `summary`：5-10 字的预览摘要，显示在 UI 中
- `message`：消息内容。可以是纯文本字符串，或结构化消息（`shutdown_request`、`shutdown_response`、`plan_approval_response`）

### 5.2 邮箱模式（Mailbox Pattern）

`SendMessageTool` 的底层使用邮箱模式。每个 Agent/Teammate 有一个逻辑邮箱（通过 `teammateMailbox.ts` 中的 `writeToMailbox()` 实现）。消息发送后进入接收者的邮箱队列，接收者在下一次工具轮次（Tool Round）中从邮箱中取出并处理。

这个设计意味着消息传递是**异步非阻塞**的——发送者调用 `SendMessage` 后立即返回，不等待接收者处理。接收者在自己的查询循环中"自然地"接收消息。

### 5.3 任务通知：<task-notification> 机制

后台 Agent 完成时，通过 `<task-notification>` XML 块将结果注入到父 Agent 的消息流中。格式如下：

```xml
<task-notification>
  <task-id>{agentId}</task-id>
  <task-type>local_agent</task-type>
  <tool-use-id>{原始调用的 toolUseId}</tool-use-id>
  <status>completed|failed|killed</status>
  <summary>{人类可读的状态摘要}</summary>
  <output-file>{输出文件路径}</output-file>
</task-notification>
```

`enqueueAgentNotification()`（在 `LocalAgentTask.tsx` 中）负责构建这个 XML 并将其放入消息队列（`messageQueueManager.ts`）。父 Agent 的查询循环在两次工具调用之间检查队列，将 notification 作为 user 消息注入对话。

### 5.4 权限冒泡（Permission Bubbling）

当一个后台 Agent 遇到需要用户授权的操作（比如写文件到受保护的路径），它不能自己弹出对话框——它在后台运行，没有 UI 焦点。

`permissionMode: 'bubble'` 的行为是：
1. 子 Agent 的权限提示被传递到**父 Agent 的终端**
2. 如果父 Agent 也在后台，继续向上冒泡，直到到达有 UI 的层级
3. `runAgent.ts` 中的 `awaitAutomatedChecksBeforeDialog`（第 456-463 行）确保后台 Agent 在显示权限对话框之前先等待自动化检查（分类器、权限钩子）完成

工程考量：如果多个后台 Agent 同时请求权限，用户会被大量权限提示淹没。`shouldAvoidPermissionPrompts` 标志在大多数异步场景下默认启用，让 Agent 跳过需要权限的操作而不是等待。

## 6. 协调者模式（Coordinator Mode）

### 6.1 什么是协调者模式

协调者模式（`coordinatorMode.ts`）是多智能体系统的"管理层"抽象。通过环境变量 `CLAUDE_CODE_COORDINATOR_MODE=1` 启用，它将主 Agent 变成一个**纯协调者**——自己不直接执行任务，而是分配工作给 Worker Agent。

协调者的系统提示词（`getCoordinatorSystemPrompt()`，第 111-179 行）明确定义了角色边界：

> You are a **coordinator**. Your job is to: help the user achieve their goal, direct workers to research, implement and verify code changes, synthesize results and communicate with the user, answer questions directly when possible — don't delegate work that you can handle without tools.

协调者只有四个工具：`AgentTool`（生成新 Worker）、`SendMessageTool`（给现有 Worker 发消息）、`TaskStopTool`（停止 Worker）、以及 PR 订阅工具（如果可用）。

### 6.2 四阶段工作流

**阶段 1：扇出研究（Fan-out Research）**

协调者并行生成多个 Worker 来收集信息。每个 Worker 负责不同方面——比如一个搜索 API 端点、一个分析数据库 schema、一个检查现有测试。所有 Worker 都是后台运行，协调者告知用户"已启动调查，结果会陆续到达"后立即结束当前轮次。

Worker 结果通过 `<task-notification>` 到达协调者。提示词中有一条重要指令："Worker results arrive as user-role messages containing `<task-notification>` XML. They look like user messages but are not."

**阶段 2：综合规划（Synthesis & Planning）**

收到所有研究 Worker 的结果后，协调者综合信息、设计实现方案。如果 Scratchpad（草稿板）💡 *一个所有 Worker 都可以读写的共享目录，用于跨 Worker 的知识持久化* 可用（由 `tengu_scratch` 特性门控制），协调者将计划写入 scratchpad 目录。

`getCoordinatorUserContext()` 会在上下文中注入 scratchpad 路径（第 104-106 行）：

> Scratchpad directory: {path}. Workers can read and write here without permission prompts. Use this for durable cross-worker knowledge — structure files however fits the work.

Scratchpad 目录位于项目临时目录中，所有 Worker 都有读写权限（无需权限提示），是跨 Worker 信息共享的桥梁。

**阶段 3：实现（Implementation）**

协调者根据计划生成实现 Worker，每个 Worker 负责计划中的特定部分。可以通过 `SendMessageTool` 给已完成的 Worker 发送后续指令（`to` 为 agent ID），利用它们已加载的上下文继续工作。

**阶段 4：验证（Verification）**

这一步的关键是：验证者必须是**全新的** Agent。

为什么？因为如果让参与实现的 Agent 来验证，它已经"知道"自己做了什么、为什么这样做。这种上下文会引入确认偏差——Agent 倾向于解释为什么发现的问题"其实不是真正的问题"。

通过生成全新的 Verification Agent，它没有实现过程的任何记忆。它只知道：原始需求是什么、哪些文件被修改了、修改方式是什么。这让它可以作为一个独立的审查者来评估工作——就像代码审查（Code Review）中审查者不应该参与编码一样。

这是**认知偏差的工程化解决方案**：不是教 Agent "请客观"（这几乎不可能），而是通过架构设计确保验证者和实现者在物理上不共享上下文。

### 6.3 协调者模式与 Fork 模式的互斥性

`isForkSubagentEnabled()` 函数（`forkSubagent.ts` 第 32-38 行）中有一个关键检查：

```typescript
if (isCoordinatorMode()) return false
```

协调者模式和 Fork 模式是互斥的。原因在于二者的设计哲学冲突：

- **Fork 模式**强调上下文继承——子 Agent 继承父 Agent 的完整对话
- **协调者模式**强调上下文隔离——Worker 只接收任务指令，不需要协调者的完整思考过程

如果在协调者模式下启用 fork，Worker 会继承协调者的所有上下文（包括对其他 Worker 的管理思考），这会污染 Worker 的判断并浪费 token。

## 7. 设计要点与工程教训

**1. 委托是一个工具调用**

将 Agent 生成实现为 `buildTool()` 创建的普通工具，意味着模型用选择其他工具完全相同的机制来选择委托。不需要特殊的"多智能体调度器"——模型的工具选择能力本身就是调度器。

**2. 字节精确是 Prompt Cache 共享的不可谈判条件**

任何影响 API 请求前缀的差异——哪怕是 GrowthBook 状态的微妙变化、一个工具的有无、thinking 配置的不同——都会导致缓存失效。Fork 模式的所有设计决策（传递 renderedSystemPrompt 而非重新生成、使用完全相同的工具池、继承 thinkingConfig）都服务于这个目标。

**3. 只读 Agent 的工具限制应该通过 disallowedTools 而非 tools 实现**

Explore 和 Plan Agent 使用 `disallowedTools` 排除写入工具，而不是用 `tools` 白名单。这意味着它们自动获得新添加到系统中的只读工具（比如新的 MCP 工具）。如果用白名单，每次加新工具都需要更新 Agent 定义。

**4. 验证者的"全新"不是建议，是架构约束**

通过在协调者提示词中将验证描述为"spawn a new agent"，并在 Verification Agent 定义中使用独立的系统提示词（没有任何实现上下文），系统在结构层面保证了验证的独立性。

**5. 僵尸防护不能遗漏**

`runAgent()` 的 `finally` 块中调用 `killShellTasksForAgent()` 不是可选的清理——没有它，一个运行 `tail -f` 的后台 bash 任务会在 Agent 结束后以 PPID=1 的僵尸进程永远存活。在大规模部署中（Whale Session 可以在 2 分钟内生成 292 个 Agent），这种泄漏会耗尽系统资源。

**6. 递归生成的防御必须多层**

防止 Agent 无限递归生成有三层防线：(a) `disallowedTools` 中排除 `AgentTool`（大多数 Agent 类型）；(b) Fork 子 Agent 中通过 `isInForkChild()` 检测 `<fork-boilerplate>` 标签拒绝再次 fork；(c) `maxTurns` 限制兜底。依赖单一防线是危险的——autocompact（自动压缩上下文）可能重写消息历史导致消息扫描失效，所以 fork 路径还在 `context.options.querySource` 中存储了标记（第 688-694 行）来抵抗 autocompact。
