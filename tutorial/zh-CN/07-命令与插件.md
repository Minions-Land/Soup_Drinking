# 第七章：命令与插件 -- 可扩展智能体的骨骼

> 上一章：[第六章：终端 UI](06-终端UI框架.md) | 下一章：[第八章：远程连接](08-远程连接.md)

Claude Code 在 `src/commands.ts` 中维护着一个命令注册中心（Command Registry），这个文件大约 600 行，但它承载的职责远比行数暗示的要重——它是所有用户可触达功能的唯一入口点。本章从这个文件出发，逐层拆解命令系统、技能系统（Skill System）和插件架构（Plugin Architecture）的完整设计。

## 1. 命令注册中心的完整架构

### 1.1 四种命令来源

打开 `src/commands.ts`，前 150 行全是 import 语句。这不是坏代码风格，而是有意为之的架构决策——这个文件是所有命令的汇聚点。命令来自四个不同来源：

**来源一：内置命令（Built-in Commands）**

直接从 `src/commands/` 目录下的各个模块 import 进来，硬编码在 `COMMANDS()` 数组中：

```typescript
// src/commands.ts:258-346
const COMMANDS = memoize((): Command[] => [
  addDir, advisor, agents, branch, btw, chrome, clear, color,
  compact, config, copy, desktop, context, cost, diff, doctor,
  effort, exit, fast, files, help, ide, init, keybindings,
  // ... 约 60 个内置命令
])
```

`memoize()` 💡（来自 [lodash](https://lodash.com/docs/4.17.15#memoize)，对函数结果做缓存，同参数只执行一次）包裹了这个数组构造。原因写在注释里：底层函数会读取 config，而 config 在模块初始化阶段还不可用。

**来源二：技能（Skills）**

通过 `getSkills()` 函数动态加载，从磁盘上的 `.claude/skills/` 目录读取 SKILL.md 文件：

```typescript
// src/commands.ts:353-398
async function getSkills(cwd: string): Promise<{
  skillDirCommands: Command[]   // 用户在 .claude/skills/ 创建的
  pluginSkills: Command[]       // 插件提供的技能
  bundledSkills: Command[]      // CLI 自带的预打包技能
  builtinPluginSkills: Command[] // 内置插件的技能
}>
```

四个子来源并行加载（`Promise.all`），任何一个失败都不会阻塞其余的——每个 Promise 单独 catch 并返回空数组。这是生产级系统里常见的**优雅降级**（Graceful Degradation）模式。

**来源三：插件命令（Plugin Commands）**

通过 `getPluginCommands()` 加载，来自已安装的第三方插件：

```typescript
// src/commands.ts:449-469
const loadAllCommands = memoize(async (cwd: string): Promise<Command[]> => {
  const [
    { skillDirCommands, pluginSkills, bundledSkills, builtinPluginSkills },
    pluginCommands,
    workflowCommands,
  ] = await Promise.all([
    getSkills(cwd),
    getPluginCommands(),
    getWorkflowCommands ? getWorkflowCommands(cwd) : Promise.resolve([]),
  ])
  // 合并顺序：bundled → builtinPlugin → skillDir → workflow → plugin → COMMANDS
  return [
    ...bundledSkills, ...builtinPluginSkills, ...skillDirCommands,
    ...workflowCommands, ...pluginCommands, ...pluginSkills, ...COMMANDS(),
  ]
})
```

合并顺序有讲究：用户自定义的技能和插件排在内置命令前面，这意味着用户可以通过同名技能**覆盖**内置命令的行为。

**来源四：工作流命令（Workflow Commands）**

受 `WORKFLOW_SCRIPTS` feature flag 控制，通过条件 require 加载：

```typescript
// src/commands.ts:401-406
const getWorkflowCommands = feature('WORKFLOW_SCRIPTS')
  ? require('./tools/WorkflowTool/createWorkflowCommand.js').getWorkflowCommands
  : null
```

`feature()` 💡 是 Bun 的编译期死代码消除（Dead Code Elimination）API。当 feature flag 为 false 时，整个 require 和关联代码在构建阶段就被移除，不会进入最终 bundle。

### 1.2 命令的三种类型

`src/types/command.ts` 定义了命令的类型系统。每个命令必须属于三种类型之一：

```typescript
// src/types/command.ts:205-206
export type Command = CommandBase & (PromptCommand | LocalCommand | LocalJSXCommand)
```

**类型一：prompt（提示词命令）**

这类命令不执行本地逻辑，而是生成一段提示词注入到对话中，由 AI 模型处理。

```typescript
// src/types/command.ts:25-57
export type PromptCommand = {
  type: 'prompt'
  progressMessage: string       // 执行中显示的加载提示
  contentLength: number          // 内容字符长度，用于 token 估算
  source: SettingSource | 'builtin' | 'mcp' | 'plugin' | 'bundled'
  context?: 'inline' | 'fork'   // inline=展开到当前对话 fork=子智能体执行
  agent?: string                 // fork 模式下的智能体类型
  model?: string                 // 可指定特定模型
  effort?: EffortValue           // 推理力度控制
  paths?: string[]               // 路径匹配模式，条件激活
  getPromptForCommand(args: string, context: ToolUseContext): Promise<ContentBlockParam[]>
}
```

`/commit`、`/review-pr` 这类技能都是 prompt 类型。关键方法是 `getPromptForCommand`，它接收用户参数和上下文，返回注入到对话中的内容块。

`context` 字段控制执行策略：`inline` 把提示词展开到当前对话（共享 token 预算），`fork` 则启动一个独立的子智能体（Sub-agent），有自己的 token 预算和工具集合。

**类型二：local（本地命令）**

执行纯 JavaScript 逻辑，不涉及 AI 模型：

```typescript
// src/types/command.ts:74-78
type LocalCommand = {
  type: 'local'
  supportsNonInteractive: boolean  // 是否支持非交互模式
  load: () => Promise<LocalCommandModule>  // 懒加载模块
}
```

`load()` 是懒加载函数——模块只在命令被实际调用时才 import，避免启动时加载所有命令的重量级依赖。

**类型三：local-jsx（交互式 JSX 命令）**

返回 React 组件，在终端中渲染交互式 UI：

```typescript
// src/types/command.ts:144-152
type LocalJSXCommand = {
  type: 'local-jsx'
  load: () => Promise<LocalJSXCommandModule>
}
```

`/config` 就是典型的 local-jsx 命令——它渲染一个交互式设置界面，用户可以用方向键导航、回车选择。命令完成后通过 `onDone` 回调通知系统：

```typescript
// src/types/command.ts:117-126
export type LocalJSXCommandOnDone = (
  result?: string,
  options?: {
    display?: CommandResultDisplay     // 'skip' | 'system' | 'user'
    shouldQuery?: boolean              // 完成后是否发送给模型
    metaMessages?: string[]            // 插入的隐藏消息
    nextInput?: string                 // 预填下一条输入
  },
) => void
```

### 1.3 命令过滤——三道关卡

命令从注册到用户可见，要通过三道过滤：

**第一道：可用性过滤（Availability）**

```typescript
// src/commands.ts:417-443
export function meetsAvailabilityRequirement(cmd: Command): boolean {
  if (!cmd.availability) return true  // 无限制 = 所有人可用
  for (const a of cmd.availability) {
    switch (a) {
      case 'claude-ai':
        if (isClaudeAISubscriber()) return true  // claude.ai 订阅用户
        break
      case 'console':
        // 直连 api.anthropic.com 的 Console 用户
        if (!isClaudeAISubscriber() && !isUsing3PServices() && isFirstPartyAnthropicBaseUrl())
          return true
        break
    }
  }
  return false
}
```

这道过滤根据用户的认证方式（OAuth 订阅 vs API Key vs 第三方提供商）决定命令是否可见。没设 `availability` 的命令对所有人可见。

**第二道：启用状态过滤（isEnabled）**

```typescript
// src/types/command.ts:214-216
export function isCommandEnabled(cmd: CommandBase): boolean {
  return cmd.isEnabled?.() ?? true
}
```

每个命令可以定义自己的启用条件——通常是 feature flag 检查、平台检测或环境变量判断。

**第三道：动态技能去重**

```typescript
// src/commands.ts:476-517
export async function getCommands(cwd: string): Promise<Command[]> {
  const allCommands = await loadAllCommands(cwd)
  const dynamicSkills = getDynamicSkills()  // 运行时发现的条件技能
  const baseCommands = allCommands.filter(
    _ => meetsAvailabilityRequirement(_) && isCommandEnabled(_),
  )
  // 去重：已存在同名命令的动态技能被跳过
  const baseCommandNames = new Set(baseCommands.map(c => c.name))
  const uniqueDynamicSkills = dynamicSkills.filter(
    s => !baseCommandNames.has(s.name) && meetsAvailabilityRequirement(s) && isCommandEnabled(s),
  )
  // 动态技能插在插件命令和内置命令之间
  // ...
}
```

`getDynamicSkills()` 返回的是运行过程中动态发现的条件技能（Conditional Skills）——在 SKILL.md frontmatter 中声明了 `paths` 字段的技能。它们只在模型操作了匹配路径的文件后才被激活。

## 2. 技能系统（Skill System）

### 2.1 技能文件格式

技能文件是带 YAML frontmatter 的 Markdown 文件，放在 `.claude/skills/` 目录下。文件名即为技能名（去掉 `.md` 后缀）。

以下是 SKILL.md 支持的所有 frontmatter 字段，从 `src/skills/loadSkillsDir.ts:185-264` 的 `parseSkillFrontmatterFields()` 函数提取：

```yaml
---
name: my-skill                    # 显示名，可选（默认用文件名）
description: "做某件事"            # 简短描述
when_to_use: "当用户要求做X时"      # 模型判断何时触发的详细说明
arguments: ["file", "mode"]       # 参数名列表
argument-hint: "<file> [--verbose]" # 参数提示文本
allowed-tools: ["Bash", "Edit"]   # 限制可用工具集
model: opus                       # 指定模型（inherit=继承当前模型）
effort: high                      # 推理力度 low/medium/high
disable-model-invocation: true    # 禁止模型主动调用此技能
user-invocable: false             # 是否允许用户手动 /技能名 调用
context: fork                     # fork=子智能体执行
agent: Bash                       # fork 模式下的智能体类型
paths: ["src/**/*.ts"]            # 路径匹配模式，条件激活
hooks:                            # 技能级别的 Hook 配置
  pre_tool_use:
    - matcher: { tool_name: "Bash" }
      hooks: [{ command: "echo 'before'" }]
shell:                            # Shell 命令相关配置
  type: bash
---

这里是技能的正文内容，会作为提示词注入到对话中...
```

### 2.2 whenToUse 字段——模型决策的关键

`whenToUse` 字段对技能系统的运作至关重要。它出现在 SkillTool 的 system prompt 中，帮助模型判断何时触发技能。

`src/tools/SkillTool/prompt.ts` 中的 `formatCommandDescription()` 函数控制这个字段如何呈现给模型：

```typescript
// src/tools/SkillTool/prompt.ts:43-50
function getCommandDescription(cmd: Command): string {
  const desc = cmd.whenToUse
    ? `${cmd.description} - ${cmd.whenToUse}`
    : cmd.description
  return desc.length > MAX_LISTING_DESC_CHARS  // 250 字符上限
    ? desc.slice(0, MAX_LISTING_DESC_CHARS - 1) + '…'
    : desc
}
```

所有技能的描述会被拼接成一个列表注入到 system prompt 中，但有预算限制——默认占上下文窗口的 1%（`SKILL_BUDGET_CONTEXT_PERCENT = 0.01`）。当技能数量多到超出预算时，描述会被截断甚至退化为只显示名称：

```typescript
// src/tools/SkillTool/prompt.ts:70-171
export function formatCommandsWithinBudget(commands: Command[], contextWindowTokens?: number): string {
  // 1. 先尝试完整描述
  // 2. 超预算时：bundled 技能保留完整描述，其余按比例截断
  // 3. 极端情况：非 bundled 技能只显示名称
}
```

这种分级截断策略（Tiered Truncation）是个好设计：保证核心内置技能的描述完整，而用户自定义技能在数量过多时优雅退化。

### 2.3 SkillTool 的调度逻辑

`src/tools/SkillTool/SkillTool.ts` 是技能系统的运行时核心。当模型决定调用一个技能时，执行路径是：

**验证阶段**（`validateInput`，354-430 行）：
1. 检查技能名是否为空
2. 去掉前导 `/`（兼容 `/commit` 和 `commit` 两种写法）
3. 远程技能特殊处理（实验性功能，带 `_canonical_` 前缀）
4. 在命令列表中查找匹配命令
5. 检查 `disableModelInvocation` 标志
6. 确认类型是 `prompt`（local 和 local-jsx 不能通过 SkillTool 调用）

**权限检查阶段**（`checkPermissions`，432-578 行）：
1. 检查 deny 规则（支持精确匹配和前缀通配 `review:*`）
2. 检查 allow 规则
3. 仅含安全属性（Safe Properties）的技能自动放行
4. 其他技能弹出权限确认对话框

"安全属性"的白名单在 `SAFE_SKILL_PROPERTIES`（875-908 行）中定义，包含 `type`、`name`、`description`、`model`、`effort` 等。如果一个技能只用了这些字段，不需要用户授权就能执行。有 `hooks`、`allowedTools` 等字段的技能则需要明确授权。

**执行阶段**（`call`，580-841 行）：

执行分两条路径：

```
call()
  ├── fork 模式 → executeForkedSkill() → runAgent()
  │     创建独立子智能体，有自己的 token 预算
  │     通过 onProgress 回调报告工具使用进度
  │     完成后返回 agentMessages 的摘要文本
  │
  └── inline 模式 → processPromptSlashCommand()
        把技能内容展开为 UserMessage 注入到当前对话
        返回 contextModifier 修改后续对话的工具集和模型
```

inline 模式的 `contextModifier` 是个精妙设计——它返回一个函数，在后续对话轮次中动态修改上下文：

```typescript
contextModifier(ctx) {
  // 1. 如果技能指定了 allowedTools，把它们加入 alwaysAllowRules
  // 2. 如果技能指定了 model，覆盖 mainLoopModel（保留 [1m] 后缀）
  // 3. 如果技能指定了 effort，覆盖 effortValue
  return modifiedContext
}
```

### 2.4 自定义技能的开发流程

创建一个技能的最小步骤：

```bash
# 1. 在项目 .claude/skills/ 目录下创建文件
mkdir -p .claude/skills
cat > .claude/skills/deploy.md << 'EOF'
---
description: "部署到生产环境"
when_to_use: "当用户说 deploy 或发布时"
allowed-tools: ["Bash"]
---

按以下步骤执行部署：
1. 运行 npm run build
2. 运行 npm run test
3. 如果测试通过，执行 ./scripts/deploy.sh
EOF
```

技能立即可用（热加载），输入 `/deploy` 或在对话中提到部署即可触发。

如果需要条件技能（只在特定文件被操作时才出现）：

```yaml
---
description: "数据库迁移助手"
when_to_use: "当修改了数据库 schema 文件"
paths: ["db/migrations/**", "prisma/schema.prisma"]
---
```

`paths` 字段使用 [ignore](https://www.npmjs.com/package/ignore) 库的 glob 语法。技能会在模型首次读取或编辑匹配路径的文件时被动态激活。

## 3. 插件架构

### 3.1 插件能提供什么

一个插件是一个 Git 仓库，包含 `plugin.json` 清单文件。插件可以提供三类扩展：

| 扩展类型 | 说明 |
|---------|------|
| 技能（Skills） | `.claude/skills/` 下的 SKILL.md 文件 |
| Hook | 工具调用前后的拦截逻辑 |
| MCP 服务器 | 提供自定义工具的 [Model Context Protocol](https://modelcontextprotocol.io/) 服务器 |

### 3.2 内置插件 vs 市场插件

内置插件（Builtin Plugins）通过 `src/plugins/builtinPlugins.ts` 注册，它们的技能通过 `getBuiltinPluginSkillCommands()` 加载。

市场插件（Marketplace Plugins）通过 `pluginIdentifier.ts` 解析来源。官方市场插件享有信任优势——`isOfficialMarketplaceName()` 用于判断插件是否来自官方市场，官方插件在遥测日志中不会被脱敏处理：

```typescript
// src/tools/SkillTool/SkillTool.ts:935-942
function isOfficialMarketplaceSkill(command: PromptCommand): boolean {
  if (command.source !== 'plugin' || !command.pluginInfo?.repository) return false
  return isOfficialMarketplaceName(
    parsePluginIdentifier(command.pluginInfo.repository).marketplace,
  )
}
```

### 3.3 插件的加载和激活流程

插件加载涉及 `src/utils/plugins/loadPluginCommands.ts` 中的两个入口：

- `getPluginCommands()` — 加载插件提供的 local/local-jsx 命令
- `getPluginSkills()` — 加载插件提供的 prompt 类型技能

两者都有缓存（`clearPluginCommandCache` / `clearPluginSkillsCache` 用于清除），在 `clearCommandsCache()` 时统一清理。

插件的技能在 SkillTool 中通过 `pluginInfo` 字段被识别，携带插件清单和仓库地址信息：

```typescript
pluginInfo?: {
  pluginManifest: PluginManifest  // 插件清单
  repository: string              // Git 仓库地址
}
```

## 4. 具体命令案例分析

### 4.1 /config——local-jsx 类型的典型实现

`/config`（`src/commands/config/index.ts`）是 local-jsx 命令。它：

1. 通过 `load()` 懒加载 UI 组件
2. `call(onDone, context, args)` 返回 React 组件
3. 组件渲染交互式设置界面
4. 用户完成设置后调用 `onDone()` 关闭组件

```typescript
const config: Command = {
  type: 'local-jsx',
  name: 'config',
  description: 'View and modify configuration',
  load: () => import('./ConfigUI.js'),
}
```

`onDone` 的 `display` 选项控制结果如何展示：
- `'skip'` — 不显示任何内容
- `'system'` — 显示为系统消息（灰色）
- `'user'` — 显示为用户消息

### 4.2 /commit——prompt 类型的工作流技能

`/commit`（`src/commands/commit.ts`）是 prompt 类型命令。它生成一段包含 Git 提交最佳实践的提示词，模型根据这段提示词分析变更、生成提交消息、执行 `git commit`：

```typescript
const commit: Command = {
  type: 'prompt',
  name: 'commit',
  description: 'Commit changes to git',
  progressMessage: 'preparing commit',
  contentLength: 0,  // 动态计算
  source: 'builtin',
  async getPromptForCommand(args, context) {
    // 生成包含 git 状态信息和提交规范的提示词
  },
}
```

prompt 类型命令的优势在于它不需要硬编码任何 Git 操作——所有具体步骤都由模型根据提示词和可用工具自行决定。

### 4.3 /compact——对话历史压缩

`/compact`（`src/commands/compact/index.ts`）返回一个特殊的 `LocalCommandResult`：

```typescript
{ type: 'compact', compactionResult: CompactionResult, displayText?: string }
```

`type: 'compact'` 告诉消息处理管线这不是普通文本输出，而是要执行对话压缩——用模型将完整对话历史总结为一个摘要，然后用摘要替换原始消息。这大幅减少 token 使用，同时保留关键上下文。

## 5. Vim 模式

`src/commands/vim/index.ts` 注册了 `/vim` 命令，切换 Vim 键绑定模式。

Vim 模式的核心是一个状态机（State Machine），管理 Normal/Insert/Visual 三种模式之间的转换。每种模式有自己的键映射表：

- **Normal 模式**：`h/j/k/l` 移动光标、`dd` 删行、`w/b` 按词移动、`/` 搜索
- **Insert 模式**：所有按键直接输入文本，`Escape` 回到 Normal
- **Visual 模式**：扩展选择区域，`y` 复制、`d` 删除

状态转换通过按键事件触发，每个按键先查询当前模式的映射表，未匹配则 fallthrough 到默认行为。这种实现简洁但有效——它不需要完整的 Vim 引擎，只需要覆盖最常用的操作。

## 6. 快捷键系统

`src/commands/keybindings/index.ts` 管理快捷键注册和分发。

系统支持多键组合（Multi-key Sequences），比如 `Ctrl+K Ctrl+C` 要求用户按顺序按两个组合键。实现方式是维护一个待匹配前缀缓冲区——第一个键按下后启动超时计时器，在超时前按下第二个键则匹配成功，否则清空缓冲区。

快捷键是**上下文感知**的（Context-aware）：同一个按键在不同状态下可能触发不同行为。比如 `Escape` 在正常模式下清空输入、在 Vim Insert 模式下切换到 Normal 模式、在自动补全弹窗中关闭弹窗。上下文信息通过 `getAppState()` 获取当前状态来决定执行哪个处理函数。

## 7. 从 Claude Code 的命令系统学到的可扩展设计模式

### 模式一：多来源聚合 + 延迟过滤

不要在加载时过滤命令，而是加载所有来源然后在查询时过滤。这允许认证状态变化（比如用户登录后）立即生效，无需重新加载：

```typescript
// 加载是 memoized 的（只做一次）
const allCommands = await loadAllCommands(cwd)
// 过滤是每次查询都重新执行的
return allCommands.filter(
  _ => meetsAvailabilityRequirement(_) && isCommandEnabled(_),
)
```

### 模式二：Feature Flag 驱动的条件加载

用编译期 feature flag 控制可选模块的加载，不合格的代码在编译阶段就被消除：

```typescript
const bridge = feature('BRIDGE_MODE')
  ? require('./commands/bridge/index.js').default
  : null
```

这比运行时 `if` 检查更好——不仅是条件执行，而是条件链接（Conditional Linking）。未启用的功能连代码都不会出现在最终产物中。

### 模式三：类型区分执行策略

`prompt | local | local-jsx` 三种类型用 TypeScript 可辨识联合类型（Discriminated Union）实现，处理代码通过 `type` 字段 switch 到不同执行路径。这比继承更灵活——新增类型只需要修改联合类型定义和 switch 语句，不需要改变已有类型的代码。

### 模式四：懒加载 + 预算控制

技能的完整内容只在调用时加载（`getPromptForCommand` 是 async 的），列表展示时只用 frontmatter 的元信息。同时通过字符预算（Character Budget）控制注入到 system prompt 中的总量，防止技能数量膨胀导致首轮 token 消耗失控。

### 模式五：权限白名单 + 渐进信任

`SAFE_SKILL_PROPERTIES` 白名单让简单技能自动放行，复杂技能（带 hooks、allowedTools 等）需要用户授权。新增属性默认需要授权——这是安全的默认值（Secure by Default）设计。

### 模式六：上下文修改器（Context Modifier）

技能执行后返回 `contextModifier` 函数，在后续对话轮次中动态修改工具可用性、模型选择、推理力度等。这实现了"技能改变对话环境"的语义，比简单地修改全局状态更安全——修改是作用域内的（scoped），技能结束后自动恢复。
