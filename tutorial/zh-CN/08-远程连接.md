# 第八章：远程连接 -- Bridge 架构深度解析

> 上一章：[第七章：命令与插件](07-命令与插件.md) | 下一章：[第九章：设计模式总结](09-设计模式总结.md)

Claude Code 在本地终端运行。但 `claude.ai/code` 上的 Web 界面也能操控同一个终端会话——你在手机上打字，笔记本上的代码被修改。这个跨设备控制能力靠的是 Bridge 子系统，核心代码在 `src/bridge/` 目录，约 15 个文件、6000+ 行。

本章拆解 Bridge 的完整架构：两代协议的演进、会话生命周期、传输层实现、认证体系、上游代理以及弹性恢复机制。

## 1. 设计动机——为什么需要远程控制

三个核心使用场景驱动了 Bridge 的诞生：

**场景一：手机端操控**。开发者通勤时想让 Claude Code 在桌面机器上跑一个长任务。Bridge 把终端变成了可远程访问的服务，`claude.ai/code` 是它的 Web 前端。

**场景二：容器内执行**。CI/CD 或安全沙箱（Sandbox）环境里，Claude Code 运行在没有直接终端接入的容器中。Bridge 提供了控制通道。

**场景三：多客户端接入**。同一个会话可以从多个界面（桌面客户端、Web、移动端）接入，Bridge 负责同步状态。

核心设计原则：**本地执行，远程控制**。代码文件留在本地机器上，所有工具（Bash、文件编辑）在本地执行，Bridge 只传递指令和结果。这规避了把源代码上传到云端的安全和隐私问题。

## 2. 两代架构的演进

### 2.1 V1——基于环境的轮询（Environment-based Polling）

V1 架构的入口是 `src/bridge/bridgeMain.ts`，它依赖 Anthropic 的 Environments API（环境 API）作为工作调度层。完整生命周期：

```
1. 注册 → POST /environments/{env_id}/workers/register
   CLI 告诉服务器："我是 worker，准备接收任务"

2. 轮询 → GET /environments/{env_id}/work
   CLI 每隔几秒向服务器请求："有新消息吗？"
   空响应 = 无事发生，继续轮询

3. 确认 → POST /environments/{env_id}/work/{work_id}/ack
   CLI 收到任务后确认："收到消息 X，正在处理"

4. 心跳 → POST /environments/{env_id}/workers/heartbeat
   CLI 定期报告："我还活着"
   服务器通过心跳超时判断 worker 是否掉线

5. 注销 → POST /environments/{env_id}/workers/deregister
   CLI 正常退出时告诉服务器："我下线了"
```

轮询模式的最大问题是**延迟**和**资源浪费**。每次轮询都是一次 HTTP 请求，即使没有新消息也要发送。轮询间隔越短延迟越低但资源消耗越高，间隔越长资源消耗越低但用户体验越差。

`src/bridge/bridgeApi.ts` 中的 `createBridgeApiClient()` 是 V1 的 HTTP 客户端。它的 `getHeaders()` 方法展示了认证头的构成：

```typescript
// src/bridge/bridgeApi.ts:76-89
function getHeaders(accessToken: string): Record<string, string> {
  const headers: Record<string, string> = {
    Authorization: `Bearer ${accessToken}`,
    'Content-Type': 'application/json',
    'anthropic-version': '2023-06-01',
    'anthropic-beta': BETA_HEADER,       // 'environments-2025-11-01'
    'x-environment-runner-version': deps.runnerVersion,
  }
  const deviceToken = deps.getTrustedDeviceToken?.()
  if (deviceToken) {
    headers['X-Trusted-Device-Token'] = deviceToken  // 可信设备令牌
  }
  return headers
}
```

`anthropic-beta: environments-2025-11-01` 表明 Environments API 仍在 beta 阶段，版本号绑定了 API 行为。

### 2.2 V2——直连端点（Env-less Direct Connection）

V2 架构的入口是 `src/bridge/remoteBridgeCore.ts`。文件顶部的注释精确描述了它与 V1 的关系：

```typescript
// src/bridge/remoteBridgeCore.ts:2-29
/**
 * Env-less Remote Control bridge core.
 *
 * "Env-less" = no Environments API layer. Distinct from "CCR v2" (the
 * /worker/* transport protocol) — the env-based path (replBridge.ts) can also
 * use CCR v2 transport via CLAUDE_CODE_USE_CCR_V2. This file is about removing
 * the poll/dispatch layer, not about which transport protocol is underneath.
 *
 * 1. POST /v1/code/sessions              (OAuth, no env_id) → session.id
 * 2. POST /v1/code/sessions/{id}/bridge  (OAuth)           → {worker_jwt, expires_in, api_base_url, worker_epoch}
 * 3. createV2ReplTransport(worker_jwt, worker_epoch)       → SSE + CCRClient
 * 4. createTokenRefreshScheduler                           → proactive /bridge re-call
 * 5. 401 on SSE → rebuild transport with fresh /bridge credentials
 *
 * No register/poll/ack/stop/heartbeat/deregister environment lifecycle.
 */
```

两个关键改进：

第一，**去掉了轮询**。V2 使用 SSE（Server-Sent Events）💡（服务器主动推送事件的 HTTP 协议，参见 [MDN SSE 文档](https://developer.mozilla.org/en-US/docs/Web/API/Server-sent_events)）作为读取通道，服务器有新消息时主动推送，零延迟。

第二，**去掉了 Environment API 层**。V1 需要通过 Environment API 间接获取 worker JWT（一种限定权限的短期令牌），V2 的 `/bridge` 端点直接用 OAuth 令牌换取 worker JWT。

### 2.3 bridgeMain.ts vs remoteBridgeCore.ts 的职责划分

| 维度 | bridgeMain.ts (V1) | remoteBridgeCore.ts (V2) |
|------|-------------------|-------------------------|
| 入口函数 | `initBridgeCore()` (~2400 行) | `initEnvLessBridgeCore()` |
| 调度层 | Environments API (env_id) | 直连 /v1/code/sessions |
| JWT 获取 | 工作调度层铸造 | POST /bridge 直接获取 |
| 读取通道 | WebSocket (V1) 或 SSE (可选 V2) | SSE 固定 |
| 写入通道 | HTTP POST | CCRClient |
| Feature Flag | 默认路径 | `tengu_bridge_repl_v2` |

V2 受 GrowthBook 💡（[开源的 feature flag 管理平台](https://www.growthbook.io/)）中的 `tengu_bridge_repl_v2` 标志控制。在 `initReplBridge.ts` 中判断走哪条路径。

## 3. 会话管理

### 3.1 会话的创建

V2 路径的会话创建在 `remoteBridgeCore.ts:166-186`：

```typescript
// 1. 创建会话
const createdSessionId = await withRetry(
  () => createCodeSession(baseUrl, accessToken, title, cfg.http_timeout_ms, tags),
  'createCodeSession',
  cfg,
)
```

`createCodeSession()` 发送 `POST /v1/code/sessions`，请求体包含：
- `title` — 会话标题（自动生成或用户指定）
- `tags` — 会话标签（如 `['ccr-mirror']` 标识镜像会话）

返回值是 `cse_` 前缀的会话 ID，用于后续所有 API 调用。

### 3.2 会话的元数据

会话创建后，CLI 通过 `reportMetadata()` 上报元数据：

```typescript
// 通过 ReplBridgeTransport.reportMetadata()
transport.reportMetadata({
  project_dir: '/Users/dev/my-project',
  git_branch: 'feature/new-ui',
  git_remote: 'https://github.com/org/repo',
})
```

这些元数据在 `claude.ai/code` 界面上展示，帮助用户识别哪个会话对应哪个项目。

### 3.3 会话归档和清理

当 CLI 正常退出或遇到不可恢复的错误时，调用 `archiveSession()` 归档会话：

```typescript
// src/bridge/remoteBridgeCore.ts:203-208
void archiveSession(
  sessionId, baseUrl, accessToken, orgUUID, cfg.http_timeout_ms,
)
```

`void` 前缀表示这是 fire-and-forget——归档请求的成功与否不影响退出流程。

### 3.4 bridgePointer.ts 的崩溃恢复

`src/bridge/bridgePointer.ts`（由 `bridgeMain.ts` 和 `replBridge.ts` 引用）在本地文件系统上存储当前会话/环境的 ID。如果 CLI 进程崩溃后重启，可以从这个文件恢复之前的会话，而不是创建新会话。

这是一种**检查点**（Checkpoint）模式：在关键状态变更后写入持久存储，崩溃后从持久存储恢复到最近的一致状态。

## 4. 传输层——四种传输方式

`src/bridge/replBridgeTransport.ts` 定义了 `ReplBridgeTransport` 接口，它抽象了所有传输方式的共同表面：

```typescript
// src/bridge/replBridgeTransport.ts:23-70
export type ReplBridgeTransport = {
  write(message: StdoutMessage): Promise<void>
  writeBatch(messages: StdoutMessage[]): Promise<void>
  close(): void
  isConnectedStatus(): boolean
  getStateLabel(): string
  setOnData(callback: (data: string) => void): void
  setOnClose(callback: (closeCode?: number) => void): void
  setOnConnect(callback: () => void): void
  connect(): void
  getLastSequenceNum(): number        // SSE 序列号高水位
  readonly droppedBatchCount: number  // 被静默丢弃的批次计数
  reportState(state: SessionState): void       // 报告状态（V2）
  reportMetadata(metadata: Record<string, unknown>): void  // 报告元数据（V2）
  reportDelivery(eventId: string, status: 'processing' | 'processed'): void  // 确认处理（V2）
  flush(): Promise<void>             // 排空写入队列（V2）
}
```

### 4.1 SSE（Server-Sent Events）——V2 读取通道

`SSETransport`（`src/cli/transports/SSETransport.ts`）是 V2 的读取通道。服务器通过 SSE 流主动推送事件到 CLI。

SSE 的关键设计点：

- **序列号（Sequence Number）**：每个 SSE 事件带有 `id` 字段，SSETransport 维护 `lastSequenceNum` 高水位标记。断线重连时发送 `Last-Event-ID` 头，服务器从该序列号之后重新推送。
- **重连**：SSE 标准规定客户端在连接断开后应自动重连，SSETransport 实现了这个行为。

```typescript
// src/bridge/replBridgeTransport.ts:39-42
// v1 returns 0 — Session-Ingress WS doesn't use SSE sequence numbers;
// replay-on-reconnect is handled by the server-side message cursor.
getLastSequenceNum(): number
```

V1 适配器硬编码返回 0，因为 V1 的 WebSocket 不使用序列号机制。

### 4.2 WebSocket——V1 双向通道

V1 使用 WebSocket 作为双向通道（通过 `HybridTransport` 封装）。WebSocket 的好处是全双工——读写共享一个连接。但 V1 路径已经在向 V2 迁移。

V1 适配器 `createV1ReplTransport()` 是一个薄包装层，把 `HybridTransport` 适配到 `ReplBridgeTransport` 接口。大部分 V2 特有的方法（`reportState`、`reportMetadata`、`reportDelivery`、`flush`）是空操作（no-op）：

```typescript
// src/bridge/replBridgeTransport.ts:78-103
export function createV1ReplTransport(hybrid: HybridTransport): ReplBridgeTransport {
  return {
    write: msg => hybrid.write(msg),
    writeBatch: msgs => hybrid.writeBatch(msgs),
    // ... 直接代理到 HybridTransport
    getLastSequenceNum: () => 0,  // V1 不用序列号
    reportState: () => {},         // V1 不支持
    reportMetadata: () => {},      // V1 不支持
    reportDelivery: () => {},      // V1 不支持
    flush: () => Promise.resolve(), // V1 不需要
  }
}
```

### 4.3 HTTP POST——V2 写入通道

V2 的写入通道是 `CCRClient`（`src/cli/transports/ccrClient.ts`），它通过 HTTP POST 发送事件到 `/worker/events` 端点。

CCRClient 的内部实现用了 `SerialBatchEventUploader`——把多个事件批量上传，减少 HTTP 请求数。它还管理心跳（Heartbeat）：定期 PUT `/worker` 端点告知服务器 worker 还活着。

```typescript
// src/bridge/replBridgeTransport.ts:119-156
export async function createV2ReplTransport(opts: {
  sessionUrl: string
  ingressToken: string
  sessionId: string
  initialSequenceNum?: number   // 从上一个 transport 继承的序列号
  epoch?: number                // 来自 /bridge 响应的 epoch
  heartbeatIntervalMs?: number  // 心跳间隔，默认 20s
  heartbeatJitterFraction?: number  // 抖动因子，防止惊群
  outboundOnly?: boolean        // 仅写入模式（镜像会话用）
  getAuthToken?: () => string | undefined  // 每实例认证（多会话安全）
}): Promise<ReplBridgeTransport>
```

`outboundOnly` 参数值得注意——当设为 true 时跳过 SSE 读取流，只激活写入路径。这用于镜像模式（Mirror Mode），一个辅助会话只转发事件但不接收指令。

### 4.4 HybridTransport——为什么要混合

`HybridTransport`（`src/cli/transports/HybridTransport.ts`）是 V1 路径的传输层，名字里的 "Hybrid" 指的是**读写分离**——读取走 WebSocket，写入走 HTTP POST。这不是技术上的必须，而是 GKE L7 负载均衡器的限制：WebSocket 的写入可靠性不如 HTTP POST（负载均衡器可能在长连接上做不可预测的事情）。

### 4.5 序列号管理和流恢复

当传输层需要重建（令牌刷新、401 恢复）时，新 transport 必须从旧 transport 停止的位置继续接收。这通过序列号（Sequence Number）实现：

```typescript
// 重建 transport 时，保存旧 transport 的序列号高水位
const lastSeq = transport.getLastSequenceNum()
// 创建新 transport 时传入，让 SSE 从这个位置之后开始
transport = await createV2ReplTransport({
  // ...
  initialSequenceNum: lastSeq,
})
```

服务器收到 `from_sequence_num` 后，只推送该序列号之后的事件。没有这个机制，每次重连都会重放整个会话历史。

## 5. 认证体系——五层安全

### 5.1 OAuth 令牌

Bridge 的基础认证是 OAuth 2.0 令牌，通过 claude.ai 登录获取。`src/bridge/remoteBridgeCore.ts` 中所有对 `/v1/code/sessions` 和 `/bridge` 的请求都带 `Authorization: Bearer ${accessToken}`。

`resolveAuth()` 函数在每次请求前调用 `getAccessToken()` 获取当前令牌。令牌由外部注入（依赖倒置模式），Bridge 代码不知道令牌从哪来，只知道如何使用。

### 5.2 JWT Worker Token

OAuth 令牌用于管理面（创建会话、获取凭证），但数据面（SSE 读取、事件写入）使用 JWT Worker Token——一个权限更窄、有效期更短的令牌。

V2 路径通过 `POST /v1/code/sessions/{id}/bridge` 用 OAuth 令牌换取 Worker JWT：

```typescript
// src/bridge/remoteBridgeCore.ts:188-214
const credentials = await withRetry(
  () => fetchRemoteCredentials(sessionId, baseUrl, accessToken, cfg.http_timeout_ms),
  'fetchRemoteCredentials', cfg,
)
// credentials = { worker_jwt, expires_in, api_base_url, worker_epoch }
```

`worker_jwt` 有 `sk-ant-si-` 前缀（Session Ingress），`expires_in` 是秒数（通常 30-60 分钟）。

`worker_epoch` 是单调递增的数字。每次调用 `/bridge` 都会 bump epoch，服务器用 epoch 来判断哪个 transport 是当前有效的——旧 epoch 的请求会收到 409（Conflict）。这解决了 transport 重建时的竞态问题（Race Condition）。

### 5.3 TokenRefreshScheduler——提前刷新策略

`src/bridge/jwtUtils.ts` 中的 `createTokenRefreshScheduler()` 是令牌刷新的核心。它的策略是**提前刷新**——在令牌过期前 5 分钟就开始刷新：

```typescript
// src/bridge/jwtUtils.ts:52-55
const TOKEN_REFRESH_BUFFER_MS = 5 * 60 * 1000       // 5 分钟缓冲
const FALLBACK_REFRESH_INTERVAL_MS = 30 * 60 * 1000  // 30 分钟回退
const MAX_REFRESH_FAILURES = 3                        // 最大连续失败次数
```

调度器的工作流程：

1. `schedule(sessionId, token)` — 解码 JWT 的 `exp` 字段，计算 `exp - now - 5min` 作为刷新延迟
2. 到期后调用 `doRefresh(sessionId, gen)` — 获取新 OAuth 令牌，通过 `onRefresh` 回调通知调用方
3. 成功后安排下一次刷新（30 分钟后的回退计时器）
4. 失败后重试（60 秒后），最多连续失败 3 次

`gen` 参数（Generation Counter）防止**过时刷新**——如果 `schedule()` 被调用两次，第一次的 `doRefresh` 在异步等待期间被第二次 `schedule()` 的新 generation 淘汰：

```typescript
// src/bridge/jwtUtils.ts:165-183
async function doRefresh(sessionId: string, gen: number): Promise<void> {
  let oauthToken = await getAccessToken()
  // 如果 generation 已变，说明有新的 schedule() 调用，当前刷新作废
  if (generations.get(sessionId) !== gen) {
    logForDebugging(`...stale (gen ${gen} vs ${generations.get(sessionId)}), skipping`)
    return
  }
  // ... 执行实际刷新
}
```

这是一个精巧的并发控制模式——不用锁（Lock），用单调递增的版本号（Monotonic Version Number）实现互斥。

JWT 解码不验证签名（`decodeJwtPayload`，21-32 行），因为客户端不需要验证——服务器会验证。客户端只需要读取 `exp` 来安排刷新时机。

### 5.4 Work Secrets

`src/bridge/workSecret.ts` 中的 `registerWorker()` 和 `buildCCRv2SdkUrl()` 处理 Work Secrets——一种与 JWT 配合使用的会话级凭证，用于构造 CCR v2 端点的 URL。

### 5.5 可信设备验证

`src/bridge/trustedDevice.ts` 提供 `getTrustedDeviceToken()`，返回可信设备令牌（Trusted Device Token）。在 `bridgeApi.ts:84-86` 中，这个令牌作为 `X-Trusted-Device-Token` 头发送。

服务器端的 `ConnectBridgeWorker` 在 enforcement flag 开启时要求可信设备验证。这是防止 OAuth 令牌被盗用的额外保护层——即使攻击者拿到了 OAuth 令牌，没有可信设备令牌也无法建立 Bridge 连接。

```typescript
// src/bridge/bridgeApi.ts:28-35
/**
 * Returns the trusted device token to send as X-Trusted-Device-Token on
 * bridge API calls. Bridge sessions have SecurityTier=ELEVATED on the
 * server (CCR v2); when the server's enforcement flag is on,
 * ConnectBridgeWorker requires a trusted device at JWT-issuance.
 */
getTrustedDeviceToken?: () => string | undefined
```

## 6. 上游代理和 Relay

### 6.1 为什么安全容器需要代理

当 Claude Code 运行在 CCR（Claude Code Runner）容器中时，容器的出口流量被网关（Egress Gateway）拦截。容器不能直接访问外部服务（如 GitHub API、Datadog），需要通过上游代理（Upstream Proxy）中继请求。

### 6.2 relay.ts 的 TCP-to-WebSocket 中继原理

`src/upstreamproxy/relay.ts` 实现了一个本地 TCP 服务器，将 HTTP CONNECT 请求转换为 WebSocket 隧道：

```
curl/gh/kubectl → TCP CONNECT → relay (本地) → WebSocket → CCR 代理服务器 → 目标服务
```

为什么不直接用 HTTP CONNECT？文件顶部注释解释了原因：

```typescript
// src/upstreamproxy/relay.ts:9-11
// WHY WebSocket and not raw CONNECT: CCR ingress is GKE L7 with path-prefix
// routing; there's no connect_matcher in cdk-constructs. The session-ingress
// tunnel already uses this pattern.
```

GKE L7 负载均衡器只支持路径前缀路由，不支持 CONNECT 方法的匹配。WebSocket 是 HTTP Upgrade 的标准方式，负载均衡器可以正确路由。

### 6.3 Protobuf 编码

字节数据通过手写的 Protobuf 编码传输（不依赖 protobufjs 库）：

```typescript
// src/upstreamproxy/relay.ts:57-80
/**
 * Encode an UpstreamProxyChunk protobuf message by hand.
 * message UpstreamProxyChunk { bytes data = 1; }
 * tag = (1 << 3) | 2 = 0x0a, followed by varint length, followed by bytes.
 */
export function encodeChunk(data: Uint8Array): Uint8Array {
  const len = data.length
  const varint: number[] = []
  let n = len
  while (n > 0x7f) {
    varint.push((n & 0x7f) | 0x80)
    n >>>= 7
  }
  varint.push(n)
  const out = new Uint8Array(1 + varint.length + len)
  out[0] = 0x0a           // field 1, wire type 2 (length-delimited)
  out.set(varint, 1)      // varint-encoded length
  out.set(data, 1 + varint.length)  // payload
  return out
}
```

手写编码的理由：单字段 bytes 消息的编码只需 10 行，引入 protobufjs 运行时依赖不划算。`MAX_CHUNK_BYTES = 512 * 1024` 限制了单块最大 512KB，匹配 Envoy 的请求缓冲区上限。

### 6.4 relay 的启动和运行时适配

`startUpstreamProxyRelay()` 根据运行时环境选择不同的 TCP 服务器实现：

```typescript
// src/upstreamproxy/relay.ts:155-174
export async function startUpstreamProxyRelay(opts: {
  wsUrl: string
  sessionId: string
  token: string
}): Promise<UpstreamProxyRelay> {
  const relay = typeof Bun !== 'undefined'
    ? startBunRelay(opts.wsUrl, authHeader, wsAuthHeader)
    : await startNodeRelay(opts.wsUrl, authHeader, wsAuthHeader)
  return relay
}
```

Bun 和 Node 的 socket API 差异显著：Bun 的 `sock.write()` 返回实际写入字节数（可能部分写入），需要手动管理写入缓冲区；Node 的 `net.Socket` 自动缓冲，从不丢弃数据。relay 代码用 `ClientSocket` 抽象层屏蔽了这些差异。

### 6.5 TLS MITM 和 NO_PROXY

CCR 代理服务器充当 TLS 中间人——终止客户端 TLS，注入组织凭证（如 `DD-API-KEY`），再与目标建立新 TLS 连接。`getWebSocketTLSOptions()` 从 `src/utils/mtls.ts` 获取配置。环境变量 `NO_PROXY` 控制哪些地址绕过代理直连。

## 7. 结构化 IO 协议

### 7.1 NDJSON 消息格式

`src/cli/structuredIO.ts` 实现了 Bridge 的消息序列化协议。所有消息使用 NDJSON 💡（Newline-Delimited JSON，每行一个 JSON 对象，参见 [NDJSON 规范](https://github.com/ndjson/ndjson-spec)）格式。

消息分为两大类：

```typescript
// 输出消息（CLI → 远程）
type StdoutMessage = SDKMessage | SDKControlRequest | StreamEvent

// 输入消息（远程 → CLI）
type StdinMessage = SDKUserMessage | SDKControlResponse
```

### 7.2 control_request 和 control_response 的交互

权限请求是 Bridge 协议中最复杂的交互模式。当模型要执行一个需要用户授权的工具时：

```
1. CLI 发送 control_request:
   {
     "type": "control_request",
     "request": {
       "subtype": "can_use_tool",
       "tool_name": "Bash",
       "tool_use_id": "uuid-123",
       "request_id": "req-456",
       "input": { "command": "rm -rf /tmp/cache" }
     }
   }

2. 远程 UI 显示权限弹窗

3. 用户点击"允许"

4. 远程发送 control_response:
   {
     "type": "control_response",
     "response": {
       "request_id": "req-456",
       "result": { "behavior": "allow" }
     }
   }
```

`StructuredIO` 类管理 pending requests：

```typescript
// src/cli/structuredIO.ts:135-157
export class StructuredIO {
  readonly structuredInput: AsyncGenerator<StdinMessage | SDKMessage>
  private readonly pendingRequests = new Map<string, PendingRequest<unknown>>()
  private readonly resolvedToolUseIds = new Set<string>()  // 去重保护
  readonly outbound = new Stream<StdoutMessage>()
  // ...
}
```

`resolvedToolUseIds` 防止重复处理——如果同一个 `tool_use_id` 的 `control_response` 到达两次（网络重传或 SSE 重连后的历史重放），第二次会被忽略。集合上限 `MAX_RESOLVED_TOOL_USE_IDS = 1000`，用 FIFO 淘汰防止内存泄漏。

`buildRequiresActionDetails()` 构造权限请求的描述信息，先尝试工具的 `getActivityDescription()`、`getToolUseSummary()`，最后 fallback 到 `userFacingName()`——三级降级确保即使工具实现有 bug 也能生成可展示的描述。

## 8. 弹性设计——五种互补的恢复机制

Bridge 运行在真实网络环境中——WiFi 切换、VPN 中断、服务器升级、令牌过期随时会发生。系统设计了五种互补的恢复机制：

### 机制一：序列号恢复（Sequence-based Recovery）

SSE 序列号实现断线续传。重连时传 `Last-Event-ID`，服务器只推送丢失的事件。没有这个机制，每次重连都要重放全量历史。

### 机制二：令牌提前刷新（Proactive Token Refresh）

`TokenRefreshScheduler` 在令牌到期前 5 分钟开始刷新。如果刷新成功，整个过程对用户透明——连接无中断。如果失败，有 3 次重试机会，每次间隔 60 秒。

### 机制三：401 恢复（Auth Recovery）

当 SSE 连接因 401（Unauthorized）断开时，`remoteBridgeCore.ts` 中的 `rebuildTransport` 被触发：

```typescript
// remoteBridgeCore.ts 中的恢复路径（概念）
// 1. 调用 onAuth401 尝试刷新 OAuth 令牌
// 2. 用新 OAuth 令牌重新调用 /bridge 获取新 worker_jwt
// 3. 用新 worker_jwt + 保存的 lastSequenceNum 创建新 transport
// 4. 新 transport 从断点继续接收
```

`connectCause` 字段记录重连原因（`'initial' | 'proactive_refresh' | 'auth_401_recovery'`），用于遥测分析。

### 机制四：Epoch 冲突检测

当 CCRClient 收到 409（Conflict）时，说明有新的 transport 注册了更高的 epoch。旧 transport 的 `onEpochMismatch` 处理函数清理资源并触发重连：

```typescript
// src/bridge/replBridgeTransport.ts:209-228
onEpochMismatch: () => {
  logForDebugging('[bridge:repl] CCR v2: epoch superseded (409) — closing for poll-loop recovery')
  try {
    ccr.close()
    sse.close()
    onCloseCb?.(4090)  // 自定义 close code 标识 epoch 冲突
  } catch (closeErr) { /* ... */ }
}
```

### 机制五：消息去重（UUID-based Deduplication）

三层去重机制防止消息重复处理：

```typescript
// src/bridge/remoteBridgeCore.ts:261-276
// 1. recentPostedUUIDs — 2000 容量的环形缓冲区，识别自己发出的消息回声
const recentPostedUUIDs = new BoundedUUIDSet(cfg.uuid_dedup_buffer_size)

// 2. initialMessageUUIDs — 初始消息的 UUID 集合，防止历史重放
const initialMessageUUIDs = new Set<string>()

// 3. recentInboundUUIDs — 入站消息去重，处理 SSE 重连后的重放
const recentInboundUUIDs = new BoundedUUIDSet(cfg.uuid_dedup_buffer_size)
```

`BoundedUUIDSet` 是有上限的集合——超过 `uuid_dedup_buffer_size`（默认 2000）后最旧的条目被淘汰。这是内存安全和去重能力之间的工程权衡（Trade-off）。

**FlushGate**（`src/bridge/flushGate.ts`）保证初始历史刷新（History Flush）和后续实时写入的顺序性——历史刷新 POST 在途期间，实时写入被队列缓存，直到历史刷新完成后才释放。

## 9. 架构决策的工程思考

### 为什么读写分离

V2 用 SSE 读、HTTP POST 写而不是 WebSocket 双向。原因有三：

1. SSE 在 L7 负载均衡器后面更可靠——它就是普通的 HTTP 长连接，不需要特殊的 WebSocket 升级支持
2. HTTP POST 写入有天然的请求-响应语义，便于确认送达和重试
3. SSE 的自动重连是标准行为，不需要额外实现

### 为什么 JWT 而不直接用 OAuth

数据面使用窄权限的 Worker JWT 而不是全权限的 OAuth 令牌：

1. 最小权限原则——Worker JWT 只能操作特定 session，OAuth 令牌能操作整个账户
2. 短有效期——JWT 30-60 分钟过期，OAuth 令牌可能数小时有效
3. 令牌隔离——Worker JWT 不会泄露到 MCP 服务器（通过闭包注入而非环境变量），OAuth 令牌存在 env var 中可能被不信任的子进程读取

### 为什么手写 Protobuf / 为什么 Generation Counter

relay.ts 手写 `encodeChunk`/`decodeChunk`：只有一个字段（`bytes data = 1`），10 行搞定，减少运行时依赖。`TokenRefreshScheduler` 用 generation number 而不是 mutex：JavaScript 单线程不需要真正互斥，竞态来自异步而非并行，generation number 是纯数据更容易推理正确性，过时的异步回调自行退出不需要取消机制。

---

Bridge 是 Claude Code 中复杂度最高的子系统之一，但它的每个设计决策都有清晰的工程理由。核心教训：分布式系统不怕复杂，怕的是每一层的复杂度没有对应的价值。
