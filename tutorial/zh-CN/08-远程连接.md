# 第八章：远程连接 -- 连接本地与云端

> 上一章：[第七章：命令与插件](07-命令与插件.md) | 下一章：[第九章：设计模式总结](09-设计模式总结.md)

你一直在笔记本电脑上使用 Claude Code。它快速、私密且强大。现在想象你在手机上，或者在另一台电脑上，你想要从完全相同的地方继续 -- 同一个项目、同一个对话、同样的工具。一个本地 CLI 工具如何变得可以从任何地方访问？

这就是 **Bridge**（桥接）要解决的问题。它将你的本地 Claude Code 实例连接到 `claude.ai/code`，将你的本地终端变成一个可远程控制的智能体。理解这个系统教会我们构建弹性分布式架构的广泛适用模式。

## 核心理念：本地执行，远程控制

Bridge 不会把你的代码搬到云端。你的文件留在你的机器上，你的工具在本地运行。Bridge 做的是创建一个安全的通信通道，这样远程界面（比如浏览器中的 claude.ai）就可以向你本地的 Claude Code 发送命令并接收结果。

把它想象成远程桌面，但只针对你的 AI 编码助手。键盘和屏幕在云端；实际工作在你的机器上进行。

核心源文件位于 `src/bridge/`：

| 文件 | 用途 |
|------|------|
| `bridgeApi.ts` | 环境和会话 API 的 HTTP 客户端 |
| `bridgeMain.ts` | V1 基于环境的连接流程 |
| `remoteBridgeCore.ts` | V2 直连流程 |
| `bridgeMessaging.ts` | 消息序列化、去重、控制请求 |
| `bridgePointer.ts` | 存储会话/环境 ID 的本地文件，用于崩溃恢复 |

## 两代架构

Bridge 经历了两代演进，每一代以不同方式解决问题。

### V1：基于环境（旧版）

第一代使用了一个叫做"环境"的概念。你的本地 CLI 会在 Anthropic 的服务器上将自己注册为可用环境，然后持续轮询传入的工作：

```
V1 流程：
  本地 CLI -> 注册环境 -> 轮询工作 -> 确认
       ^                                 |
       +------------- 心跳 -------------+
```

当工作到达时（来自 claude.ai 的用户消息），CLI 会处理它、发回结果、继续轮询。这是经典的**长轮询**模式 -- 简单可靠，但本质上是浪费的，因为即使没有任何事情发生 CLI 也在轮询。

V1 使用 **Work Secrets** -- 充当环境凭证的不透明连接字符串。

### V2：无环境（直连）

当前这一代完全消除了环境抽象：

```
V2 流程：
  本地 CLI -> POST /bridge -> 接收 JWT 令牌
           -> 打开 SSE 流（服务器推送事件给客户端）
           -> POST 消息（客户端发送结果给服务器）
```

V2 更简单也更高效。CLI 直接连接到 `/bridge` 端点，接收短期 JWT，打开 SSE 流接收消息，用 HTTP POST 发送消息。没有轮询、没有环境注册、没有浪费的请求。

## 会话生命周期

无论哪个 Bridge 版本处于活跃状态，核心概念都是**会话**。会话代表远程用户与其本地 Claude Code 之间的一次持续交互：

```
会话生命周期：
  创建（POST /v1/sessions）-> 活跃（消息流动中）-> 归档（清理）
```

会话携带丰富的元数据：项目目录、git 仓库 URL、当前分支和人类可读的标题。这些元数据让远程 UI 可以显示"正在 `feature/auth` 分支上处理 `my-app`"，而不仅仅是一个会话编号。

### 崩溃恢复：Bridge 指针

最重要的设计决策是**崩溃恢复**。`bridgePointer.ts` 模块将会话 ID 和环境 ID 持久化到磁盘上的本地文件。如果 Claude Code 崩溃或笔记本重启：

1. 重启时，Claude Code 读取指针文件
2. 找到之前的会话 ID
3. 重新连接到同一个会话，而不是创建新的
4. 对话就像什么都没发生一样继续

这种"原地重连"策略至关重要。claude.ai 上的用户看到的是短暂断连，而不是丢失的对话。

## 传输机制：字节如何流动

Bridge 使用多种传输协议，每种针对不同场景优化：

| 传输 | 协议 | 方向 | 使用场景 |
|------|------|------|---------|
| **SSE** | Server-Sent Events | 服务器到客户端 | V2 读取 |
| **WebSocket** | WS | 双向 | V1、直连 |
| **HTTP POST** | HTTP | 客户端到服务器 | V2 写入 |
| **Hybrid** | WS + POST | 混合 | V1 组合 |

**为什么不全部用 WebSocket？** 因为读和写有不同的可靠性要求。服务器到客户端的消息受益于 SSE 内置的自动重连机制。客户端到服务器的消息作为可独立重试的单个 HTTP POST 效果更好。

`HybridTransport` 结合了两者：从 WebSocket 读取（低延迟服务器推送），通过 HTTP POST 写入（带重试的可靠传递）。这承认了分布式通信中的根本性不对称。

### 通过序列号实现流恢复

每个传输都跟踪一个**序列号**（`lastTransportSequenceNum`）。如果连接断开，客户端告诉服务器："我最后收到的是消息 #47 -- 从 #48 开始发送。"这实现了无缝的流恢复，无需重放整个对话。

## 认证：多层安全

远程控制开发者的机器是一项高风险能力。Bridge 实现了五个重叠的认证层：

**OAuth（系统钥匙串）**：基础层。Claude Code 将 OAuth 令牌存储在系统钥匙串中。证明你是合法的 Claude Code 用户。

**JWT Worker 令牌**：V2 会话中，`/bridge` 端点发行限定于特定会话的短期 JWT 令牌。很快过期（通常几分钟），限制令牌泄露的影响范围。

**Work Secrets**：V1 的更简单方法 -- 既充当标识符又充当环境凭证的不透明连接字符串。

**可信设备令牌**：`X-Trusted-Device-Token` HTTP 头验证物理设备，在用户凭证之外增加安全层。

**进程级安全**：在 Linux 上，Bridge 调用 `prctl(PR_SET_DUMPABLE, 0)` 防止其他进程读取其内存，保护令牌免受恶意软件窃取。

## 令牌刷新：主动而非被动

`TokenRefreshScheduler` 类展示了一个重要的分布式系统原则。天真的方法等到请求因"401 未授权"失败，然后刷新。这会造成可见的卡顿。

Claude Code 的方法是主动的：计算 JWT 何时过期，减去 5 分钟，在需要之前刷新。

```
令牌在 T=0 发行，T=60分钟过期
  -> 调度器在 T=55分钟设置刷新定时器
  -> T=55分钟时：后台获取新令牌
  -> 旧令牌仍有效直到 T=60分钟
  -> 无缝过渡，零停机
```

这个模式 -- 主动凭证更新 -- 适用于任何使用过期令牌的系统。

## 上游代理：安全容器

当 Claude Code 在安全容器内运行时，它无法直接访问互联网。`src/upstreamproxy/` 中的上游代理解决这个问题：

```
工具（curl/npm）-> HTTP CONNECT -> 本地 Relay（relay.ts）
  -> WebSocket -> Anthropic 服务器 -> TLS MITM -> 目标
```

`relay.ts` 模块在 localhost 上创建 TCP 到 WebSocket 的中继。工具认为它们在与普通 HTTP 代理对话。中继通过 WebSocket 将流量隧道到 Anthropic 的服务器。受信任的域名（GitHub、npm 注册表）通过 `NO_PROXY` 绕过代理。

## 结构化 IO：消息协议

所有通信通过 `src/cli/structuredIO.ts` 使用 **NDJSON**（换行符分隔的 JSON）。两种消息类型至关重要：

- **`control_request`**：本地 CLI 需要决定。"此工具想写入 `/etc/hosts`。允许吗？"
- **`control_response`**：远程用户的回答返回。"是的，允许。"

NDJSON 非常适合：人类可读（便于调试）、对流式传输友好（每行到达时即可解析）、除换行符外无需额外帧格式。

## 会话恢复：五个机制协同工作

Bridge 使用五个互补机制实现弹性：

1. **指针文件** -- 写入磁盘的会话 ID 能存活进程崩溃和机器重启。
2. **序列号** -- 每条消息都有编号。重连时只重放遗漏的消息。
3. **消息去重** -- `bridgeMessaging.ts` 中的 `BoundedUUIDSet` 跟踪最近消息 ID，丢弃网络重试的重复。
4. **心跳管理** -- 检测陈旧连接，在用户注意到之前触发重连。
5. **Epoch 追踪** -- 确保只有最新连接实例活跃，防止幽灵会话。

没有哪个单一机制能提供完整弹性。它们共同在本质上不稳定的网络上创造了稳定连接的假象。

## 核心要点：弹性分布式架构

> 构建远程连接时，假设一切都会失败。使用指针文件进行崩溃恢复，序列号用于流恢复，有界 UUID 集用于去重，主动令牌刷新保证认证连续性，多种传输协议适应网络变化。分布式系统中的可靠性不是单一特性 -- 它是一系列小型互补机制的集合，共同创造稳定的假象。

Bridge 教会我们，弹性不在于防止故障，而在于恢复得如此迅速和透明，以至于用户永远不会注意到故障的发生。
