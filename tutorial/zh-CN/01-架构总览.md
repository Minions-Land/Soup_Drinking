# 第一章：Claude Code 架构全景——从按键到结果的完整旅程

## Claude Code 到底在做什么？

先说一个你可能已经感受到但没细想过的问题：Claude Code 不是一个"聊天机器人套壳"。

假设你在终端里输入 `请帮我把 src/utils/format.ts 中的 formatDate 函数重构成支持时区`，从你按下回车的那一刻起，系统做了以下事情：

1. 解析你的输入，判断这不是斜杠命令（Slash Command）
2. 组装系统提示词（System Prompt），把你的工作目录、项目上下文、所有可用工具的描述一起打包
3. 调用 Claude API，流式接收模型的响应
4. 模型决定先调用 `Read` 工具读取文件——系统验证输入、检查权限、执行读取、把结果送回模型
5. 模型看到文件内容后，决定调用 `Edit` 工具修改代码——再走一遍验证-权限-执行的管道
6. 模型可能还会调用 `Bash` 工具跑一下测试确认没破坏什么
7. 所有工具调用完毕后，模型生成最终的文本回复，告诉你改了什么

这整个过程涉及十几个文件、数十个函数调用，但核心架构其实只有三个支柱。搞懂它们，你就掌握了 Claude Code 的骨架。

再举一个更具体的场景：你在一个 monorepo 里工作，输入 `找出所有使用了已废弃 API 的文件并修复它们`。Claude Code 会自主地先用 `Grep` 搜索关键字，用 `Glob` 找文件，用 `Read` 读取内容，用 `Edit` 逐个修改，最后用 `Bash` 跑测试。这整个流程没有任何硬编码的工作流——模型自己决定要做什么、用什么工具、按什么顺序。

这就是智能体（Agent）架构的核心思想：**你提供能力，模型决定编排**。

## 三大支柱：QueryEngine、Tools、Tasks

### QueryEngine（查询引擎）——大脑

💡 **QueryEngine** 是管理整个对话生命周期的核心类。一个 `QueryEngine` 实例对应一个对话会话，从创建到结束，所有状态都在这个实例上。

打开 `src/QueryEngine.ts`，你会看到这个类的构造函数接收一个 `QueryEngineConfig` 对象（大约在第 200 行）。这个配置对象包含了引擎运转所需的一切：

```
QueryEngineConfig {
  cwd: string                    // 工作目录
  tools: Tools                   // 可用工具列表
  commands: Command[]            // 斜杠命令列表
  mcpClients: MCPServerConnection[] // MCP 服务器连接
  agents: AgentDefinition[]      // 子智能体定义
  canUseTool: CanUseToolFn       // 权限检查回调
  getAppState / setAppState      // 应用状态读写
  readFileCache: FileStateCache  // 文件状态缓存
  customSystemPrompt?: string    // 自定义系统提示词
  thinkingConfig?: ThinkingConfig // 思考模式配置
  maxTurns?: number              // 最大轮次限制
  maxBudgetUsd?: number          // 预算上限
  ...
}
```

这个设计选择很聪明：通过把所有依赖注入到构造函数，`QueryEngine` 本身不依赖任何全局状态。你完全可以在同一个进程里同时运行多个 `QueryEngine` 实例——事实上子智能体（Subagent）就是这样工作的。

`QueryEngine` 最重要的方法是 `submitMessage()`（第 209 行），它是一个 AsyncGenerator（异步生成器），每当有新的事件（流式文本、工具调用、状态变更）时就 yield 出来。SDK 消费者可以逐个处理这些事件，而不必等到整个响应生成完毕。

### Tools（工具）——双手

工具系统是 Claude Code 与真实世界交互的唯一通道。模型本身无法读文件、跑命令、写代码——它只能通过请求工具调用来完成这些操作。

💡 **Tool 接口**定义在 `src/Tool.ts` 中，是所有工具必须遵守的契约。核心方法包括 `call()`（执行）、`validateInput()`（验证输入）、`checkPermissions()`（检查权限）等。

工具的实现分散在 `src/tools/` 目录下，每个工具一个子目录。目前大约有 30+ 个工具，涵盖文件读写（`FileReadTool`、`FileEditTool`、`FileWriteTool`）、搜索（`GrepTool`、`GlobTool`）、命令执行（`BashTool`）、子智能体（`AgentTool`）等。

工具系统的详细剖析在第二章，这里只需要知道一点：**每个工具都是一个数据结构，而不是一段过程式代码**。你可以在不执行工具的情况下查看它的名称、描述、输入模式、权限需求。这使得工具可以被自动发现、过滤和组合。

### Tasks（任务）——后台工人

💡 **Tasks** 是处理与主对话并行运行的长时间操作的机制。与工具不同，任务不是由模型直接调用的——它们在后台运行，完成后把结果注入到对话中。

典型的任务场景包括：后台代码分析、异步搜索、长时间运行的测试。任务有自己的状态管理和生命周期，定义在 `src/Task.ts` 中。

三大支柱的关系可以用一句话概括：**QueryEngine 决定"做什么"，Tools 决定"怎么做"，Tasks 处理"在后台做"的工作**。

## 一次请求的完整旅程

这是最重要的一节。我会逐步追踪一个用户请求在系统中经过的每一个关键节点，引用真实的文件和函数。

### 第一步：输入解析——processUserInput()

用户按下回车后，输入首先到达 `src/utils/processUserInput/processUserInput.ts` 中的 `processUserInput()` 函数。

这个函数是一个分支路由器。它做了以下判断：

1. **是斜杠命令吗？** 如果输入以 `/` 开头（如 `/compact`、`/help`、`/clear`），调用 `parseSlashCommand()` 解析并在本地执行，不会发送到模型
2. **有特殊关键字吗？** 比如 `ultraplan` 关键字会触发特殊处理流程（`hasUltraplanKeyword()` / `replaceUltraplanKeyword()`，第 58-60 行的引入）
3. **包含图片粘贴吗？** 如果检测到图片数据（`isValidImagePaste()`），会进行压缩和格式化
4. **有附件吗？** 文件附件、代理提及（Agent Mention）等需要预处理

如果都不是，就走普通文本路径 `processTextPrompt()`，生成一个标准的用户消息。

返回值的类型是 `ProcessUserInputBaseResult`，包含几个关键字段：

```typescript
{
  messages: (UserMessage | AssistantMessage | AttachmentMessage | SystemMessage)[]
  shouldQuery: boolean      // 是否需要调用模型
  allowedTools?: string[]   // 限制本次可用的工具
  model?: string            // 覆盖本次使用的模型
  effort?: EffortValue      // 控制推理深度
}
```

`shouldQuery` 这个字段是关键——斜杠命令会返回 `shouldQuery: false`，因为它们在本地就处理完了，不需要调用 API。

### 第二步：QueryEngine.submitMessage()

如果 `shouldQuery` 为 true，控制权交给 `QueryEngine.submitMessage()`（`src/QueryEngine.ts:209`）。

这个方法的内部流程大致如下：

**2a. 准备阶段：**
- 调用 `setCwd()` 确保工作目录正确
- 调用 `fetchSystemPromptParts()` 组装系统提示词的各个部分——默认系统提示、用户上下文（工作目录、项目信息）、系统上下文（工具描述、MCP 资源）
- 解析模型配置——用户指定的模型还是默认模型？思考模式是否启用？
- 包装 `canUseTool` 回调以追踪权限拒绝（第 244-271 行）

**2b. 权限拒绝追踪——这是一个值得注意的设计：**

`submitMessage()` 会包装传入的 `canUseTool` 回调，在原始逻辑之外加入拒绝追踪：

```
// src/QueryEngine.ts:244-271
const wrappedCanUseTool: CanUseToolFn = async (...) => {
  const result = await canUseTool(...)
  if (result.behavior !== 'allow') {
    this.permissionDenials.push({
      tool_name: sdkCompatToolName(tool.name),
      tool_use_id: toolUseID,
      tool_input: input,
    })
  }
  return result
}
```

这样 SDK 消费者就能知道哪些工具调用被拒绝了，但不需要修改权限系统本身。装饰器模式（Decorator Pattern）的典型应用。

**2c. 进入查询循环：**

准备工作完成后，`submitMessage()` 调用 `query()` 函数进入核心循环。

### 第三步：query() 循环——智能体循环的心脏

`src/query.ts` 中的 `query()` 函数是整个 Claude Code 最核心的代码。它实现了一个经典的 [ReAct 循环](https://arxiv.org/abs/2210.03629)（💡 ReAct = Reasoning + Acting，一种让模型交替进行推理和行动的范式）。

`query()` 本身是一个薄包装器（第 219-239 行），真正的循环在 `queryLoop()` 中（第 241 行开始）。

`queryLoop()` 首先初始化一个 `State` 对象来管理跨迭代的可变状态：

```typescript
let state: State = {
  messages: params.messages,          // 对话历史
  toolUseContext: params.toolUseContext, // 工具执行上下文
  autoCompactTracking: undefined,     // 自动压缩追踪
  maxOutputTokensRecoveryCount: 0,    // 输出token溢出恢复计数
  hasAttemptedReactiveCompact: false,  // 是否尝试过响应式压缩
  turnCount: 1,                       // 当前轮次
  pendingToolUseSummary: undefined,    // 待处理的工具使用摘要
  transition: undefined,              // 上一次迭代的继续原因
}
```

然后是一个 `while (true)` 循环（第 307 行），每次迭代：

1. **yield `stream_request_start`** 事件通知消费者新的 API 请求即将开始
2. **启动技能发现预取**（Skill Discovery Prefetch）——在模型流式输出的同时，后台异步查找相关的技能
3. **调用 Claude API**——发送消息历史 + 系统提示词 + 工具描述，流式接收响应
4. **处理响应中的内容块（Content Blocks）**——文本块直接 yield 给消费者，`tool_use` 块收集起来等待执行
5. **检查停止原因（stop_reason）**：
   - `end_turn`：模型认为任务完成，退出循环
   - `tool_use`：模型请求调用工具，进入工具执行阶段
   - `max_tokens`：输出太长被截断，尝试恢复（最多 3 次）
6. **执行工具调用**——这是下一步的内容

### 第四步：工具执行——并发与串行的智能调度

当模型返回 `tool_use` 块时，控制权转给 `src/services/tools/toolOrchestration.ts` 中的 `runTools()` 函数。

这个函数做了一件很聪明的事：**把工具调用分成"并发安全"和"串行执行"两组**。

分组逻辑在 `partitionToolCalls()` 中。它检查每个工具的 `isConcurrencySafe()` 方法：

- 只读工具（如 `Read`、`Grep`、`Glob`）返回 `true`，可以并发执行
- 写入工具（如 `Edit`、`Write`、`Bash`）返回 `false`，必须串行执行

如果模型在一次响应中同时请求读取三个文件和编辑一个文件，系统会：
1. 先并发执行三个 `Read` 调用
2. 然后串行执行 `Edit` 调用

并发上限由环境变量 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY` 控制，默认是 10（`toolOrchestration.ts:8-12`）。

这个设计选择反映了一个工程权衡：并发读取能显著加速信息收集阶段，但写入操作必须串行以避免竞态条件（Race Condition）。如果是我来做，可能也会考虑更细粒度的锁（比如按文件路径加锁），但那样会大幅增加复杂性，收益有限。

### 第五步：工具验证管道——四个阶段

每个工具调用在执行前必须通过四个验证阶段。这是安全模型（Security Model）的核心。

**阶段一：Schema 验证（自动）**

工具的 `inputSchema`（用 [Zod](https://zod.dev/) 定义）自动验证输入结构。比如 `FileReadTool` 要求 `file_path` 必须是字符串，`offset` 如果提供必须是非负整数。

这一步在 `runToolUse()` 内部自动完成，工具作者不需要写任何代码。如果模型生成了格式错误的输入（比如 `offset` 是一个字符串），Zod 会立即捕获并返回结构化的错误信息。

**阶段二：语义验证（validateInput）**

通过 Schema 验证后，调用工具的 `validateInput()` 方法进行业务逻辑检查。以 `FileReadTool` 为例（`src/tools/FileReadTool/FileReadTool.ts:418-495`）：

- 如果提供了 `pages` 参数，检查格式是否合法（`parsePDFPageRange()`）
- 检查文件路径是否命中拒绝规则（`matchingRuleForInput()` 查询 deny 列表）
- 检查是否是 UNC 路径（防止 NTLM 凭据泄露——这是一个安全细节）
- 检查是否是二进制文件（`.exe`、`.dll` 等无法作为文本读取）
- 检查是否是危险的设备文件（如 `/dev/zero` 会产生无限输出）

注意这些检查**都不进行 I/O 操作**——它们只基于路径字符串做判断。这是故意的：I/O 操作（如 `stat()`）应该在权限检查通过之后才执行，否则恶意的 UNC 路径可能在权限检查前就泄露了 NTLM 凭据。

**阶段三：权限检查（checkPermissions）**

验证通过后，检查用户是否授权了这次操作。`checkPermissions()` 返回一个 `PermissionResult`，可能是：

- `allow`：直接执行
- `ask`：弹出权限提示让用户确认
- `deny`：拒绝执行并告诉模型原因

`FileReadTool` 的权限检查委托给 `checkReadPermissionForTool()`（`src/utils/permissions/filesystem.ts`），它会查询多层权限规则：全局配置、项目配置、用户的运行时选择。

**阶段四：执行（call）**

所有检查通过后，调用 `call()` 方法执行实际操作。工具的结果包装在 `ToolResult` 中返回，格式化为 `tool_result` 消息追加到对话历史。

然后控制权回到 `queryLoop()`，带着更新后的消息历史进入下一次迭代——模型看到工具结果，决定下一步做什么。

### 第六步：循环终止

循环终止的条件有几个：

1. 模型发出 `stop_reason: "end_turn"`——正常完成
2. 达到 `maxTurns` 限制——防止无限循环
3. 预算耗尽——`maxBudgetUsd` 控制的费用上限
4. 用户中断——通过 `AbortController` 取消
5. Token 预算用完——`tokenBudget` 相关的检查（`checkTokenBudget()`）

终止后，`query()` 返回一个 `Terminal` 对象，`submitMessage()` 完成最后的清理工作（刷新会话存储、记录 transcript 等），yield 最后的状态事件。

## 目录结构——每个目录存在的理由

下面不是简单的目录列表，而是解释每个目录为什么存在、它和其他目录的关系。

### 核心骨架

```
src/QueryEngine.ts    # 对话生命周期管理。一个实例 = 一个对话。
src/query.ts          # ReAct 循环实现。被 QueryEngine 调用，但也可以独立使用。
src/Tool.ts           # 工具接口定义 + buildTool() 工厂。所有工具的"宪法"。
src/Task.ts           # 任务抽象。后台操作的统一接口。
src/commands.ts       # 斜杠命令注册表。本地命令和技能命令的入口。
src/tools.ts          # 工具池组装。把所有工具实例收集到一个数组里。
```

`QueryEngine.ts` 和 `query.ts` 的分离是刻意的。`QueryEngine` 管理会话级的状态（消息历史、文件缓存、用量追踪），`query.ts` 只关心单次查询循环的逻辑。这意味着你可以在不理解 `QueryEngine` 的情况下理解查询循环的机制，反之亦然。

### 工具目录

```
src/tools/
  BashTool/           # Shell 命令执行。包含 bashSecurity.ts——安全过滤的核心
  FileReadTool/       # 文件读取。支持文本、图片、PDF、Jupyter Notebook
  FileEditTool/       # 精确的文件编辑。基于字符串替换，不是行号
  FileWriteTool/      # 完整文件写入。新建或覆写
  GrepTool/           # 正则搜索。底层用 ripgrep
  GlobTool/           # 文件模式匹配。用于"找出所有 *.ts 文件"
  AgentTool/          # 子智能体生成。每个子智能体是一个独立的 QueryEngine
  SendMessageTool/    # 智能体间通信。coordinator 模式下使用
  TodoWriteTool/      # 任务列表管理。模型可以创建和更新 TODO
  NotebookEditTool/   # Jupyter Notebook 编辑
  WebSearchTool/      # 网络搜索
  WebFetchTool/       # 网页抓取
  SyntheticOutputTool/ # 合成输出。用于结构化输出场景
  SleepTool/          # 等待。在协调模式下用于轮询
  ...
```

每个工具子目录通常包含：
- `XxxTool.ts` 或 `XxxTool.tsx`——主实现（使用 `buildTool()` 构建）
- `UI.tsx`——终端渲染组件
- `prompt.ts`——工具描述和提示词常量
- 可选的辅助模块（如 `BashTool/bashSecurity.ts`、`FileReadTool/limits.ts`）

为什么每个工具一个目录而不是每个工具一个文件？因为工具的关注点天然是分离的：执行逻辑、渲染逻辑、提示词、安全规则——把它们塞在一个文件里会导致 1000+ 行的巨型文件。分目录后，你修改 `BashTool` 的安全规则时完全不需要看它的 UI 代码。

### 服务层

```
src/services/
  api/                # Claude API 客户端，包含重试、错误处理、日志
  analytics/          # 遥测和分析事件
  compact/            # 对话压缩——当上下文窗口快满时自动摘要
  mcp/                # Model Context Protocol 客户端
  tools/              # 工具编排——并发/串行调度
  tokenEstimation.ts  # Token 计数估算
  skillSearch/        # 技能发现和搜索
```

`services/compact/` 值得特别关注。当对话历史太长快要超出模型的上下文窗口（Context Window）时，系统不会简单地截断，而是调用模型生成一个摘要来"压缩"历史。这个过程由 `autoCompact` 模块管理，是 Claude Code 能处理长会话的关键。

### 基础设施

```
src/utils/
  permissions/        # 权限系统——文件系统权限、Shell 命令权限
  hooks/              # 生命周期钩子——PreToolUse、PostToolUse 等
  processUserInput/   # 输入预处理管道
  model/              # 模型选择和配置
  bash/               # Bash 命令解析（shell-quote、heredoc 提取等）
  messages/           # 消息格式化和规范化
  config.js           # 配置管理
  cwd.ts              # 工作目录管理
  fileHistory.ts      # 文件历史快照（用于撤销）
  fileStateCache.ts   # 文件状态 LRU 缓存
  ...

src/state/
  AppState.ts         # 全局应用状态类型定义

src/context/
  notifications.ts    # 通知系统
  ...

src/bridge/           # IDE 集成桥接（VS Code 等）
src/ink/              # 终端 UI 组件（基于 React/Ink）
src/entrypoints/      # 各种入口点（CLI、SDK、headless）
```

`src/utils/permissions/` 和 `src/utils/hooks/` 的关系是：权限系统做最终的"允许/拒绝"决策，钩子系统让你可以在工具执行前后注入自定义逻辑（比如在每次文件写入后自动跑 lint）。

## 架构图

```
+------------------------------------------------------------------------+
|                          用户输入                                        |
+------------------------------------------------------------------------+
                              |
                    processUserInput()
                     (斜杠命令分流)
                              |
                     shouldQuery == true?
                       /          \
                     否              是
                      |              |
              本地执行完毕    QueryEngine.submitMessage()
                              |
                     fetchSystemPromptParts()
                      组装系统提示词 + 上下文
                              |
                        query() / queryLoop()
                              |
                     +--------+---------+
                     |  while(true) 循环 |
                     +--------+---------+
                              |
                     调用 Claude API (流式)
                              |
                     解析响应内容块
                              |
               stop_reason == "end_turn"?
                    /                 \
                  是                    否 (tool_use)
                   |                    |
            yield 最终响应        runTools()
                              并发/串行工具调度
                                    |
                           partitionToolCalls()
                             /            \
                       并发安全组         串行组
                       (Read,Grep...)   (Edit,Write,Bash...)
                             |                |
                   runToolsConcurrently()  runToolsSerially()
                             |                |
                     每个工具经过 4 阶段验证管道:
                     Schema → validateInput → checkPermissions → call
                                    |
                            tool_result 消息
                            追加到对话历史
                                    |
                            回到 while(true) 循环顶部
                            下一次 API 调用
```

## 设计哲学——带有工程判断的分析

### "工具是数据，不是代码"

这是最根本的设计决策。看 `src/Tool.ts` 中 `Tool` 类型的定义（第 362 行开始）——它不是一个抽象类让你继承，而是一个接口让你实现。更进一步，`buildTool()` 工厂函数（第 783 行）接受一个普通对象并补全默认值。

这意味着你可以在运行时动态地：
- 列出所有可用工具并展示给模型
- 根据上下文过滤工具（`isEnabled()` 方法）
- 把工具描述注入到系统提示词中
- 对工具进行分组（并发安全 vs 非并发安全）

如果工具是通过类继承实现的，这些操作要么不可能，要么需要大量反射（Reflection）代码。

### "模型是编排器"

Claude Code 没有硬编码的工作流。不存在"先搜索再读取再编辑再测试"的流程定义。模型拿到一组工具和上下文，自己决定执行顺序。

这带来了惊人的灵活性——同一个系统既能处理"重命名一个变量"这样的简单任务，也能处理"重构整个模块"这样的复杂任务，不需要任何代码改动。

代价是什么？你失去了确定性。同样的输入，模型可能走不同的路径。对于需要可重复性的场景（如 CI/CD），这可以通过 `maxTurns`、`allowedTools` 等约束来缓解。

### "安全是分层的"

Claude Code 的安全不依赖单一防线，而是多层叠加：

1. **输入验证**（Zod Schema + validateInput）——过滤格式错误和不合理的输入
2. **权限系统**（checkPermissions）——确保用户授权了操作
3. **命令安全过滤**（`bashSecurity.ts`）——阻止危险的 Shell 命令模式
4. **钩子系统**（PreToolUse / PostToolUse）——用户自定义的安全策略
5. **结果大小限制**（maxResultSizeChars）——防止上下文窗口被撑爆

即使其中一层被绕过，其他层仍然提供保护。这是经典的纵深防御（Defense in Depth）策略。

### "UI 是解耦的"

终端界面使用 [Ink](https://github.com/vadimdemedes/ink)（💡 一个基于 React 的终端 UI 框架，用 JSX 写终端界面）渲染，完全独立于核心逻辑。`QueryEngine` 和工具系统不知道也不关心输出最终被渲染成什么样。

工具的 `renderToolUseMessage()` 和 `renderToolResultMessage()` 方法返回 React 节点，但它们是可选的——SDK 模式下根本不需要终端渲染。这种分离让同一套核心逻辑可以同时服务 CLI、VS Code 扩展和 headless SDK。

## 如果你只读五个文件

如果你想通过阅读源码来理解 Claude Code 的架构，我建议按以下顺序：

### 1. `src/Tool.ts`——从"宪法"开始

从这里开始，因为它定义了整个系统的核心抽象。重点看：
- `Tool` 类型定义（第 362 行）——理解每个方法的签名和用途
- `ToolUseContext` 类型（第 158 行）——理解工具执行时能访问什么
- `buildTool()` 工厂函数（第 783 行）——理解默认值策略
- `TOOL_DEFAULTS` 常量（第 757 行）——理解"关闭失败"（fail-closed）设计

### 2. `src/tools/FileReadTool/FileReadTool.ts`——一个典范工具

这个文件展示了一个完整工具的所有方面：
- 如何用 `lazySchema()` 定义延迟加载的输入 Schema（第 227 行）
- 如何实现 `validateInput()` 做安全检查（第 418 行）——注意所有检查都不做 I/O
- 如何实现 `checkPermissions()` 委托给权限系统（第 398 行）
- 如何在 `call()` 中处理多种文件类型（文本、图片、PDF、Notebook）
- 文件去重优化——如果文件没变就返回 stub 而不是重新发送内容（第 536-573 行）

### 3. `src/query.ts`——理解循环

跳过前面的 import，直接看 `queryLoop()` 函数（第 241 行）。重点关注：
- `State` 类型（第 204 行）——循环携带的可变状态
- `while (true)` 循环的主体（第 307 行开始）
- 工具调用的分发逻辑——`runTools()` 的调用位置

### 4. `src/QueryEngine.ts`——理解会话管理

看 `QueryEngineConfig` 类型（第 130 行）理解引擎需要什么，然后看 `submitMessage()` 方法（第 209 行）理解一个完整的用户消息是如何被处理的。

如果你对多智能体架构感兴趣，这个文件展示了子智能体是如何被创建的——本质上是启动一个新的 `QueryEngine` 实例，带有独立的上下文和受限的工具集。
